# StoryLock Local Agent Gateway Design

## Current Positioning

The local agent gateway corresponds to Layer 3 `storylock-remote-gateway-skill`. It is not an ordinary business Skill, but a packaging, validation, forwarding, and redaction layer before remote requests enter local sensitive capabilities.

Current external main interfaces are only:

1. `requestSignature`
2. `requestPasswordFill`

Old interfaces `requestStoryRead`, `requestStoryWrite`, `requestChallengeSign`, `requestCapabilityStatus`, `requestLocalStoryAssist`, `queryStoryMetadata` are no longer part of the current design baseline.

## Gateway Responsibilities

Layer 3 is responsible for:

1. Validating request fields.
2. Validating basic formats of `nonce` and `expiry`.
3. Generating `StoryLockSignatureRequest` EIP-712 request structures.
4. Handing off to the local Host via `transport` or optional local executors.
5. Performing unified redaction on returned results.
6. Ensuring only whitelisted capabilities are exposed.

Layer 3 is not responsible for:

1. Saving private keys or passwords.
2. Verifying grid challenge answers.
3. Issuing sessions.
4. Directly reading secret objects.
5. Exposing Layer 2 internal APIs.

## Execution Modes

Current code supports two execution modes.

### Mode A: Transport Packaging

`StoryLockRemoteGateway({ transport })` hands the standardized request object to the host environment. The host environment is responsible for connecting the request into Layer 2 or other local execution chains.

### Mode B: Optional Local Executors

`StoryLockRemoteGateway` can be injected with:

1. `signatureExecutor`
2. `passwordFillExecutor`

In this case, the gateway still first constructs a standard request, then calls the local executor, and redacts the execution results.

## Signature Request

`requestSignature` required fields:

1. `requestId`
2. `nonce`
3. `expiry`
4. `identityId`
5. `keyId`
6. `algorithm`
7. `payload`

Supported algorithms:

1. `ed25519`
2. `secp256k1`

The EIP-712 type generated by the gateway is `StoryLockSignatureRequest`, with fields including:

1. `action`
2. `resource`
3. `scope`
4. `expiry`
5. `nonce`
6. `requestedBy`
7. `delegationContext`

## Password Fill Request

`requestPasswordFill` required fields:

1. `requestId`
2. `nonce`
3. `expiry`
4. `identityId`
5. `credentialRef`
6. `targetOrigin`

Results returned to the remote side must not contain plaintext passwords.

## Redaction Rules

The gateway recursively replaces the following fields:

1. `answers`
2. `signingKey`
3. `signingKeyBytes`
4. `secretBytes`
5. `secretValue`
6. `password`
7. `privateKey`
8. `mnemonic`
9. `seed`
10. `rawSecret`
11. `keyMaterial`

Safe reference fields may be retained, such as `secretReference`, `secretObjectId`.

## Error Codes

Layer 3 should pass through or wrap local `SLG` error codes. The current error code table is based on `THREE_PACKAGE_CONTRACT.md`, where replay conflicts use `SLG-013`.

## Performance Targets

Current stage should only retain testable targets:

| Operation | Target |
| --- | --- |
| Request structure validation | < 50ms |
| EIP-712 request construction | < 50ms |
| Gateway redaction | < 50ms |
| Local signature execution | Determined by local signer or hardware device |
