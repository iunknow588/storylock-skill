# StoryLock 运行层级与 Skill 分层设计

## 目的

本文档用于回答以下问题：

1. StoryLock 是否应该分成多层 Skill
2. 每一层 Skill 分别负责什么
3. 哪一层必须纯本地执行
4. 在当前阶段，如何先让本地 Agent 跑通
5. 远程 Skill 与代理授权 Skill 应该放在哪一层

## 先给结论

你的三层思路是对的，但当前阶段建议优先采用“本地优先”的最小落地版本：

1. **第一层：纯本地内容处理 Skill**
2. **第二层：故事访问 / 受保护对象访问 Skill**
3. **第三层：远程 Skill / 代理授权 / 编排 Skill**

当前阶段先不讨论第二层上云，默认按本地实现。

## 三层 Skill 结构

## 第一层：纯本地故事处理 Skill

这是最底层，也是最敏感的一层。

负责：

1. 故事编辑
2. 故事修改
3. 故事完善
4. 故事发布前整理
5. 私密故事输入处理

这一层处理的是：

1. 原始用户记忆
2. 私密故事片段
3. 用户未脱敏草稿
4. 需要强隐私保护的生成过程

推荐特征：

1. 纯本地执行
2. 不依赖远程秘密托管
3. 不把原始故事材料直接暴露给第三方 Skill

关于第一层模型选择，当前阶段建议：

1. 默认按“本地规则处理 / 本地轻量模型 / 本地人工辅助”理解
2. 若必须调用远程 LLM，前提是输入已脱敏
3. 一旦调用远程 LLM，则该次处理不再属于“纯本地秘密处理链路”

当前阶段建议优先实现：

1. 规则引擎
2. 模板化草稿整理
3. 本地人工辅助编辑

而不是把第一阶段强依赖在本地大模型上。

因此第一层的“纯本地”含义应明确为：

1. 原始私密输入不出本地
2. 未脱敏故事正文不出本地
3. 远程模型只允许处理脱敏后的派生内容

这意味着当前阶段第一层的最小可运行版本可以：

1. 完全不依赖本地 LLM
2. 先用规则引擎和模板能力跑通接口
3. 再把本地 LLM 作为后续增强能力接入

建议对应能力：

1. `StoryDraftAssistSkill` 的本地模式
2. `StoryRefineAssistSkill` 的本地模式
3. 后续可扩展 `StoryPersistSkill`

## 第二层：故事访问 Skill

这一层负责“能不能访问受保护故事对象”，而不是负责“怎么改写文本”。

负责：

1. 访问受保护故事对象
2. 读取故事素材
3. 提供受控写回入口
4. 根据 challenge / session / policy 决定对象是否可用

这一层可以细分成两种模式：

### 模式 A：本地受控访问

负责：

1. 本地 challenge 验证
2. 本地读取 secret object
3. 本地提供故事对象给第一层处理

这是默认安全模式，也是推荐模式。

### 当前阶段建议

当前阶段不引入云端托管访问。

建议先把第二层定义为：

1. 本地 challenge 验证
2. 本地受保护对象读取
3. 本地受保护对象写回
4. 本地策略判断

## 第三层：远程 Skill / 代理授权 / 编排 Skill

这一层负责：

1. 接收第三方 Skill 请求
2. 对请求做结构化包装
3. 路由到本地能力或受控访问层
4. 返回结构化结果
5. 承担代理授权、委托审计、流程编排

这一层适合承载：

1. 远程 Agent
2. 第三方 Skill 接入层
3. 代理签名 Skill
4. 本地能力网关的远程入口

它不应直接负责：

1. 明文私密故事处理
2. 长期持有私钥
3. 直接保存 challenge answers

## 三层职责总表

| 层级 | 名称 | 核心职责 | 是否可远程 | 默认推荐位置 | 核心风险 |
| --- | --- | --- | --- | --- | --- |
| 第一层 | 本地故事处理 Skill | 编辑、修改、完善、发布准备 | 不推荐 | 本地 | 原始隐私数据外泄 |
| 第二层 | 故事访问 Skill | 授权读取 / 写回受保护故事对象 | 条件成立 | 本地优先，云端受限可选 | 明文访问、对象滥用、会话滥用 |
| 第三层 | 远程 Skill / 代理授权 Skill | 编排、包装、委托、审计、第三方接入 | 可以 | 远程 | 过度授权、结果转授、能力外泄 |

## 当前阶段推荐架构

当前阶段建议采用最小可实现架构：

1. 第一层：本地
2. 第二层：本地
3. 第三层：先概念定义，后续再接入

也就是：

1. 先把本地故事处理跑通
2. 再把本地故事访问跑通
3. 最后再接第三层远程 Skill / 代理授权层

## 最小落地目录结构建议

当前阶段建议最小目录结构如下：

```text
src/
  storylock-local-story-processing-skill/
    SKILL.md
    index.js
    references/
  storylock-local-story-access-skill/
    SKILL.md
    index.js
    storage/
    policies/
  storylock-remote-gateway-skill/
    SKILL.md
    index.js
    references/
  shared/
    secret-store/
    crypto/
    sqlite/
    schemas/
```

建议开发顺序：

1. 先完成 `shared/crypto` 与 `shared/sqlite`
2. 再完成第二层 `challenge/session/access`
3. 再把第一层和第三层挂接到第二层上

## 第二层实现前置条件

第二层在实现前，至少还需要以下两份定义：

1. `Challenge状态机.md`
2. `EIP-712最小请求定义.md`

前者解决 challenge / session 生命周期问题，后者解决第三层委托请求的最小结构化表示问题。

## 结论

你的三层划分是成立的，当前阶段建议固定成下面这个版本：

1. **第一层：纯本地故事处理 Skill**
   - 负责故事编辑、修改、完善、发布准备
2. **第二层：故事访问 Skill**
   - 负责授权访问和受保护对象读写
   - 当前阶段按本地实现
3. **第三层：远程 Skill / 代理授权 Skill**
   - 负责第三方接入、委托、包装、编排与审计
   - 当前阶段先只做接口边界定义，不急着部署
