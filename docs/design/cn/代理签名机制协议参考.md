# StoryLock 代理签名机制参考设计

## 目的

本文档用于回答以下问题：

1. StoryLock 是否应该提供“代理签名”能力
2. 第三方 Skill 是否可以把签名动作委托给本地 Skill
3. 现有协议中哪些适合直接借鉴
4. 哪些协议适合做执行层参考，哪些只适合作为审计或治理层参考

## 核心判断

结论是：

1. StoryLock 应该提供“本地签名委托能力”
2. 第三方 Skill 不应直接持有私钥
3. 第三方 Skill 应提交结构化签名请求
4. 本地 Skill / 本地 Host 在通过授权后完成签名
5. 返回给第三方 Skill 的应是结构化签名结果，而不是私钥或长期签名能力

因此，远程 Agent 的角色不是“私钥持有者”，而是：

1. 请求路由器
2. 能力包装器
3. 结果整理器

真正的签名执行者应始终在本地。

## 建议的代理签名机制分层

建议把代理签名机制拆成四层：

### 1. 请求表示层

负责：

1. 描述“要签什么”
2. 描述“允许在什么范围内签”
3. 描述“签名用途是什么”

这一层最值得参考：

1. `EIP-712`

### 2. 本地授权层

负责：

1. challenge 创建
2. answer 提交
3. scope 校验
4. 本地对象访问策略判断

这一层主要由 StoryLock 自己负责，不直接照搬外部协议。

### 3. 签名验证 / 账户能力层

负责：

1. 链上如何承认签名是有效的
2. 合约账户如何替代 EOA
3. 是否允许智能账户、抽象账户参与

这一层最值得参考：

1. `EIP-1271`
2. `ERC-4337`
3. `EIP-7702`

### 4. 委托审计与责任链层

负责：

1. 记录是谁授权了什么
2. 记录远程 Agent 是否只是转发
3. 记录本地执行结果的责任来源

这一层最值得参考：

1. `HDP Protocol`

## 协议参考总表

| 名称 | 类型 | 对 StoryLock 代理签名的参考价值 | 建议使用方式 |
| --- | --- | --- | --- |
| `EIP-712` | 链下签名标准 | 很高 | 作为第三方 Skill 提交签名请求的结构化消息格式 |
| `EIP-1271` | 链上标准 | 很高 | 作为“本地签名者不是 EOA，而是合约账户”时的验证接口 |
| `EIP-2612` | 链上标准 | 中 | 作为 permit 型签名子场景参考，而非总代理签名框架 |
| `ERC-4337` | 链上标准 | 高 | 作为智能账户执行层参考，适用于抽象账户钱包或策略钱包 |
| `EIP-7702` | 链上标准 | 中到高 | 作为 EOA 临时委托执行逻辑的能力参考 |
| `Metaplex Agent Registry` | 生态框架 | 中 | 参考“链上身份 + 链下执行”的模式，不直接解决本地签名委托 |
| `Keyless Collective SDK` | 项目框架 | 高 | 参考“Agent 不持私钥、由外部协调层执行”的总体架构 |
| `Turnkey Agentic Wallet` | 项目框架 | 高 | 参考“密钥在受控基础设施中、Agent 只调用受限能力”的模型 |
| `HDP Protocol` | 标准化协议 / IETF 草案 | 很高 | 作为“谁授权了签名请求、经过哪些 Agent 转手”的审计链模型 |

## 最值得直接借鉴的协议

### 一、EIP-712

最值得借鉴的点：

1. 结构化签名数据
2. 域隔离
3. 可读性强
4. 适合离线签名请求表达

对 StoryLock 的建议：

1. 远程 Agent 发给本地 Agent 的签名请求，可以采用 EIP-712 风格的结构化对象
2. 请求中应至少包含：
   - `action`
   - `resource`
   - `scope`
   - `expiry`
   - `nonce`
   - `requestedBy`
   - `delegationContext`

这能避免“盲签名式”请求进入本地签名网关。

### 二、EIP-1271

最值得借鉴的点：

1. 签名有效性不再绑定到单一 EOA
2. 合约账户可以定义自己的签名验证逻辑

对 StoryLock 的建议：

1. 如果后续本地签名者升级为智能账户、本地安全模块控制的合约钱包、或策略钱包，那么应优先兼容 `EIP-1271`
2. 这使 StoryLock 的本地签名委托不被“EOA 私钥直签”绑死

### 三、HDP Protocol

最值得借鉴的点：

1. 记录人类授权来源
2. 记录多跳 Agent 委托链
3. 记录 scope 和 provenance

对 StoryLock 的建议：

1. 在本地 Agent 网关里为每次代理签名附加 delegation provenance
2. 远程 Agent 只做转发时，也要被记录在委托链上
3. 这样可证明：
   - 谁发起了请求
   - 哪个 Agent 转交了请求
   - 本地 Agent 在什么 scope 下执行了签名

## 值得在执行层参考的协议

### 四、ERC-4337

最值得借鉴的点：

1. `UserOperation` 作为统一操作对象
2. 账户抽象
3. 支持策略钱包、Gas 代付、批量执行

对 StoryLock 的建议：

1. 如果 StoryLock 的本地签名委托最终面向智能账户执行交易，可以借鉴 `UserOperation` 的请求建模
2. 但它不是本地授权层本身的替代品
3. 它更适合放在“签名后的执行层”

### 五、EIP-7702

最值得借鉴的点：

1. EOA 可临时委托执行逻辑
2. 更接近“按单次交易授予能力”

对 StoryLock 的建议：

1. 可以作为“本地签名后，EOA 临时借用合约逻辑执行”的思路参考
2. 适合做“有限范围、短时授权”的执行层补充
3. 不适合作为 StoryLock 本地授权层的唯一方案

## 值得作为特定业务子场景参考的协议

### 六、EIP-2612

最值得借鉴的点：

1. permit 模式
2. 离线授权 + 第三方代提交流程

对 StoryLock 的建议：

1. 适合参考“离线授权后，由第三方提交”的用户体验
2. 很适合 token allowance 这类窄场景
3. 但它不够广，不能直接承担“通用代理签名机制”

## 值得作为产品架构参考的框架

### 七、Turnkey Agentic Wallet

最值得借鉴的点：

1. 密钥不暴露给 Agent
2. 由受控基础设施执行业务签名
3. 用策略限制 Agent 行为

对 StoryLock 的建议：

1. 这是非常接近 StoryLock 目标架构的产品参考
2. StoryLock 可以借鉴它的思路，但把“受控基础设施”替换为“本地 Agent / 本地 Host”

### 八、Keyless Collective SDK

最值得借鉴的点：

1. Agent 不自己管理私钥
2. 由外部协调层完成支付或执行
3. 通过策略限制权限范围

对 StoryLock 的建议：

1. 很适合参考其“能力代理”思路
2. StoryLock 可将本地网关视作自己的协调层
3. 但具体密码学与网络拓扑未必需要照搬

### 九、Metaplex Agent Registry

最值得借鉴的点：

1. 链上身份
2. 可验证代理身份
3. 链下执行 + 链上登记的组合模式

对 StoryLock 的建议：

1. 适合参考身份登记与发现机制
2. 不直接解决“私钥不能交给第三方 Skill”这个问题
3. 更适合作为代理身份层，而不是签名委托层

## 对 StoryLock 最合适的组合方案

建议采用以下组合：

### 必选参考

1. `EIP-712`
   - 用于定义签名请求格式
2. `HDP Protocol`
   - 用于定义委托链、审计链、责任来源
3. `EIP-1271`
   - 用于兼容未来智能账户或合约验证签名

### 条件参考

1. `ERC-4337`
   - 当你们准备接入账户抽象钱包时引入
2. `EIP-7702`
   - 当你们需要 EOA 单次临时委托逻辑时引入
3. `EIP-2612`
   - 当你们要做 permit / token allowance 类特化功能时引入

### 架构参考

1. `Turnkey Agentic Wallet`
2. `Keyless Collective SDK`
3. `Metaplex Agent Registry`

## 本地密钥存储建议

当前阶段建议把密钥存储策略单独固定下来。

推荐优先级：

1. 操作系统密钥链
2. 平台安全存储
3. 可选硬件安全模块或硬件密钥

平台建议：

1. Windows：优先对接系统保护存储能力
2. macOS：优先使用 Keychain
3. Linux：优先使用 Secret Service 或等价安全存储

当前阶段不建议：

1. 直接把长期私钥明文写入普通配置文件
2. 直接把长期私钥放入可被第三层读取的 SQLite 表

## 多身份密钥管理建议

若后续支持多身份，建议采用派生式管理而不是散落式管理。

推荐方向：

1. 保留一个受控根密钥
2. 按 `identityId` 派生工作密钥或签名上下文
3. 派生逻辑由本地安全层掌握，第三层不可见

这样可以：

1. 减少多身份独立密钥文件数量
2. 降低密钥目录管理复杂度
3. 让审计更容易收敛到身份级别

## 签名审计建议

每次签名后建议至少记录：

1. `requestId`
2. `scope`
3. `resource`
4. `signatureHash`
5. `timestamp`

审计中不应记录：

1. 私钥原文
2. seed 原文
3. 可导致重放的原始 challenge material

## 签名算法选择建议

当前阶段不建议把所有算法一股脑同时开放，而应按场景做收敛。

建议决策顺序：

1. 若目标生态为以太坊 / EVM：
   - 优先 `secp256k1`
2. 若目标为通用离线签名、跨链代理标识或非 EVM 系统：
   - 可选 `ed25519`
3. 若存在明确合规或国密要求：
   - 预留 `SM2`，但当前阶段不默认实现

当前阶段推荐：

1. 以太坊相关请求默认 `secp256k1`
2. StoryLock 内部非链上演示或通用消息签名可使用 `ed25519`
3. 不在同一 identity 上混用多种算法，除非 Host 有明确策略

## 算法选择最小决策树

```text
目标是否为 EVM 链上签名？
|- 是 -> secp256k1
|- 否 -> 是否为通用消息/代理标识签名？
   |- 是 -> ed25519
   |- 否 -> 是否存在合规要求？
      |- 是 -> SM2（后续扩展）
      |- 否 -> 由 Host 白名单决定
```

## EIP-712 适配代码示例

当前阶段建议把第三层包装请求和第二层本地签名入口对齐成统一结构。

示例：

```ts
export function buildStoryLockEip712Request({
  action,
  resource,
  scope,
  expiry,
  nonce,
  requestedBy,
  delegationContext,
  chainId,
  verifyingContract,
}: {
  action: string;
  resource: string;
  scope: string;
  expiry: bigint;
  nonce: bigint;
  requestedBy: string;
  delegationContext: string;
  chainId: number;
  verifyingContract: `0x${string}`;
}) {
  const domain = {
    name: "StoryLock",
    version: "1-placeholder",
    chainId,
    verifyingContract,
  };

  const types = {
    StoryLockSignatureRequest: [
      { name: "action", type: "string" },
      { name: "resource", type: "string" },
      { name: "scope", type: "string" },
      { name: "expiry", type: "uint256" },
      { name: "nonce", type: "uint256" },
      { name: "requestedBy", type: "string" },
      { name: "delegationContext", type: "string" },
    ],
  };

  const value = {
    action,
    resource,
    scope,
    expiry,
    nonce,
    requestedBy,
    delegationContext,
  };

  return { domain, types, value };
}
```

第二层本地网关收到请求后，至少应校验：

1. `action`
2. `scope`
3. `expiry`
4. `nonce`
5. `requestedBy`
6. `delegationContext`

只有通过 challenge / session / scope / policy 检查后，才允许进入实际签名步骤。

## 建议的 StoryLock 代理签名定位

StoryLock 不应把“代理签名”定义成单纯的签名 API。

更合理的定位是：

1. 第三方 Skill 提交结构化签名意图
2. 远程 Agent 对请求做包装
3. 本地 Agent 执行 challenge、scope 检查、对象访问控制、签名
4. 返回结构化签名结果与最小审计元信息

所以它更像：

1. `Local Signing Delegate`
2. `Local Capability Gateway`
3. `Policy-controlled Signing Broker`

而不是：

1. “把私钥给远程 Skill”
2. “让远程 Skill 直接签”

## 结论

对于 StoryLock 来说，最正确的定位不是“第三方 Skill 学会签名”，而是：

1. 第三方 Skill 学会发起签名委托
2. 远程 Agent 学会包装与转交请求
3. 本地 Skill 学会在授权后完成签名

在协议参考上，最值得优先借鉴的是：

1. `EIP-712` 用于请求表示
2. `EIP-1271` 用于签名验证兼容性
3. `HDP Protocol` 用于委托与审计链
4. `ERC-4337` / `EIP-7702` 用于后续执行层扩展
