# StoryLock 对象访问策略

## 目的

本文档定义 StoryLock 应如何：

1. 对对象进行分类
2. 确定访问强度
3. 确定访问是一次性的还是可复用的
4. 判定远程 Agent 是否可以保留某种能力或结果

它与以下文档配套使用：

1. `storylock_skill_positioning_and_boundary_cn.md`
2. `src/storylock-skill-engine/SKILL.md`

## 策略目标

本地运行时应能够明确回答以下问题：

1. 当前访问的是哪类对象
2. 需要多高的 challenge 强度
3. 访问是单次还是会话型
4. 远程 Agent 是否可以在访问后保留任何能力

## 对象分类

| 对象类别 | 示例 | 敏感度 | 默认运行位置 |
| --- | --- | --- | --- |
| `story_public` | 通用故事模板、非私密草稿 | 低 | 远程安全 |
| `story_private` | 用户记忆、个人叙事、私密草稿 | 中到高 | 本地优先 |
| `auth_login` | 用户名 / 密码字段包 | 高 | 仅本地 |
| `auth_signing` | 签名密钥、签名授权材料 | 很高 | 仅本地 |
| `review_data` | 24 题题集与就绪度评估数据 | 中 | 视内容而定，可本地或远程 |

## 访问强度级别

系统可以暴露多个访问门槛。

| 级别 | 示例门槛 | 典型用途 |
| --- | --- | --- |
| `L1` | 弱挑战或无挑战 | 公共模板、低风险元数据 |
| `L2` | 6-of-24 | 低风险读取或有限评估数据 |
| `L3` | 12-of-24 | 中等敏感秘密访问或短时字段读取 |
| `L4` | 22-of-24 | 签名类或高信任操作 |
| `L5` | Host 决定的更高门槛 | 根级对象或特别敏感操作 |

当前 StoryLock 常量已经给出一个基础映射：

1. basic access threshold：`6`
2. batch access threshold：`12`
3. modify threshold：`22`

## 渐进式挑战策略

24 题题集是统一托底题集，不是三套不同题集。

当前阶段建议采用：

1. `L2 = 6-of-24`
2. `L3 = 12-of-24`
3. `L4 = 22-of-24`

它体现的是：

1. 同一题集
2. 不同通过阈值
3. 对应不同访问强度

这样做的好处是：

1. 用户只维护一套题集
2. 阈值差异直接体现权限强度差异
3. 不会把系统复杂度浪费在多套题集管理上

## 会话模型

访问必须显式声明持续时间与复用能力。

| 会话类型 | 生命周期 | 读取次数 | 是否可复用 | 远程保留规则 |
| --- | --- | --- | --- | --- |
| `one_shot` | 单次调用 | 1 | 否 | 否 |
| `short_session` | 短 TTL | 有限 | 有限 | 仅在允许时保留不透明 handle |
| `batch_session` | 中等 TTL | 多次 | 有限 | 仅在策略允许时可保留 |
| `privileged_session` | 严格 TTL | 最少 | 否 | 永不保留 |

当前实现中的常量可以作为策略输入：

1. challenge TTL
2. session TTL
3. basic session access counts
4. batch session access counts

## 远程能力保留规则

远程 Agent 不应长期保留敏感访问能力。

允许保留：

1. 结构化结果
2. 脱敏摘要
3. 在显式批准下的短时不透明 handle

不允许保留：

1. 原始密码
2. 私钥
3. challenge answers
4. 长时可复用的 secret access token
5. 第二层内部 session material

## 本地执行强制规则

若对象访问涉及以下任一项，则必须在本地执行：

1. challenge answers
2. session material
3. secret-object access
4. signing-key handling
5. 根级对象重建
6. 题集答案摘要校验

这意味着：

1. 第三层只能发起请求，不能替代本地执行
2. Pharos 或其他远程 Skill 只能承接包装与能力索引
3. 第一层若处理原始私密故事内容，也应优先本地执行

## 访问决策表

| 对象类别 | 推荐强度 | 会话类型 | 远程可否缓存结果 | 远程可否保留后续访问能力 |
| --- | --- | --- | --- | --- |
| `story_public` | `L1` | `one_shot` | 可以 | 可以 |
| `story_private` | `L2-L3` | `short_session` | 仅脱敏后可缓存 | 不可以 |
| `auth_login` | `L3` | `short_session` | 不可以 | 不可以 |
| `auth_signing` | `L4-L5` | `privileged_session` | 不可以 | 不可以 |
| `review_data` | `L2-L3` | `short_session` 或 `batch_session` | 脱敏后可以 | 不可以 |

补充限制：

1. `story_private` 默认不返回完整原文给远程侧，除非显式策略允许且结果已降敏
2. `auth_signing` 默认只返回签名结果，不返回签名能力句柄
3. `review_data` 若包含题集结构、答案摘要或高可推测信息，应提升为本地优先对象

## 挑战配置示例

### L2：低风险读取

```json
{
  "accessLevel": "L2",
  "threshold": 6,
  "questionPoolSize": 24,
  "sessionType": "short_session",
  "sessionTtlSeconds": 600,
  "maxReads": 3,
  "redactionRequired": true
}
```

### L3：中敏读取或有限写回

```json
{
  "accessLevel": "L3",
  "threshold": 12,
  "questionPoolSize": 24,
  "sessionType": "short_session",
  "sessionTtlSeconds": 600,
  "maxReads": 1,
  "maxWrites": 1,
  "redactionRequired": true
}
```

### L4：高敏签名或高信任写操作

```json
{
  "accessLevel": "L4",
  "threshold": 22,
  "questionPoolSize": 24,
  "sessionType": "privileged_session",
  "sessionTtlSeconds": 180,
  "maxReads": 0,
  "maxWrites": 1,
  "redactionRequired": true,
  "requiresLocalExecution": true
}
```

## 运行规则

如果访问涉及以下任一项：

1. challenge answers
2. session material
3. secret-object reads
4. signing-key handling

则必须在本地运行时完成。

如果访问属于以下场景，默认也不应由第三层直接持久化结果：

1. 高敏签名授权结果之外的中间材料
2. 题集评估过程中的原始题集输入
3. 根级重建过程中的任何中间状态

## Session 与兼容窗口规则

单故事终身制下，访问策略不仅决定“能不能访问”，还要决定“旧能力还能保留多久”。

建议明确如下：

1. 常规题集优化时，已签发的 `short_session` 可运行到 TTL 结束
2. 常规对象封装迁移时，旧对象密文可保留短时兼容窗口
3. 根级重建时，旧 session 必须立即失效
4. `privileged_session` 无论何种优化，均不允许跨窗口复用

## 第三层可见信息边界

第三层远程 Agent 默认只应看到：

1. capability name
2. 结构化结果
3. redaction metadata
4. retentionGranted
5. 低敏元信息

第三层默认不应看到：

1. challenge 创建细节
2. 答案验证细节
3. 题集答案摘要
4. secret object 原始密文
5. rootKey / workKey / objectKey 相关信息

## 推荐实现形态

本地运行时应暴露一个小型策略网关，用于返回：

1. 所需访问门槛
2. 批准的会话类型
3. 读取预算
4. 是否可缓存
5. 是否要求脱敏

远程 Agent 应只消费：

1. capability name
2. result payload
3. redacted metadata
4. 显式批准下的短时不透明 handle

## 结论

正确的模型不是“任何能力都能在任何地方复用”。

正确的模型应是：

1. 访问按对象分类
2. 会话被严格限制
3. 远程复用受到约束
4. 敏感能力供给不应长期保存在远程侧
