# StoryLock 快速决策指南

本文用于快速判断一个能力应该放在哪一层，以及是否必须本地执行。

## 一页纸判断

### 第一步：是否接触敏感材料

如果能力会接触以下任一内容，就不能直接放在远程层：

1. 九宫格答案。
2. session 或短时授权材料。
3. 密码、私钥、签名器或等价凭据。
4. 原始私密故事内容。
5. 未脱敏的本地审计上下文。

### 第二步：它是在处理内容、控制授权，还是包装请求

如果它负责草稿、润色、题集强度评估，属于第一层。

如果它负责对象强度判断、九宫格验证、session、防重放和短时授权，属于第二层。

如果它负责接收远程请求、包装请求、调用本地执行器并返回脱敏结果，属于第三层。

## 当前三层

| 层级 | 当前定位 | 代表能力 |
| --- | --- | --- |
| 第一层 | 本地故事处理与强度分析 | `StoryDraftSkill`、`StoryRefineSkill`、`StrengthReviewSkill` |
| 第二层 | 对象强度、九宫格验证与本地授权 | `ObjectStrengthPolicySkill`、`GridChallengeSkill`、`LocalAuthorizationSkill` |
| 第三层 | 远程请求包装与代理授权入口 | `requestSignature`、`requestPasswordFill` |

## 常见场景对照

| 场景 | 建议归属 | 说明 |
| --- | --- | --- |
| 根据私密记忆生成故事草稿 | 第一层 | 原始私密输入不应交给远程侧 |
| 对已脱敏草稿进行润色 | 第一层或远程文本能力 | 取决于输入是否已经脱敏 |
| 判断签名对象需要什么验证强度 | 第二层 | 属于对象访问策略 |
| 生成九宫格并校验答案 | 第二层 | 答案只在本地处理 |
| 第三方请求签名 | 第三层入口，本地执行 | 使用 `requestSignature` 包装请求 |
| 第三方请求登录表单密码填充 | 第三层入口，本地执行 | 使用 `requestPasswordFill` 包装请求 |

## 反模式

不要这样做：

1. 让远程网关读取故事内容。
2. 让远程网关持有私钥或密码。
3. 让故事处理层绕过第二层授权。
4. 把第二层写成业务内容读写层。
5. 不应把签名接口命名为只适合 challenge 场景的已废弃旧接口 `requestChallengeSign`。

## 当前阶段推荐

当前阶段优先保证：

1. 第一层文本处理能力可运行。
2. 第二层对象强度、九宫格验证和授权链路清楚。
3. 第三层只暴露 `requestSignature` 与 `requestPasswordFill`。
4. 所有远程返回都遵守脱敏和最小化原则。
