# StoryLock 快速决策指南

## 目的

本文档用于快速判断：

1. 一个能力应该属于哪个 Skill
2. 一个能力应该放在哪一层
3. 一个能力是否必须本地执行

## 一页纸判断

### 第一步：它是否接触敏感对象

如果一个能力直接接触以下任一项：

1. challenge answers
2. session material
3. secret-object reads
4. signing-key access
5. 原始私密故事内容

那么它不能直接放在远程层。

### 第二步：它是在“处理内容”还是“控制访问”

如果它负责：

1. 编辑故事
2. 修改故事
3. 完善故事
4. 发布前整理

那么它属于：

1. 第一层：本地故事处理 Skill

如果它负责：

1. challenge 验证
2. session 建立
3. 故事对象读取
4. 故事对象写回
5. 对象访问策略判断

那么它属于：

1. 第二层：故事访问 Skill

如果它负责：

1. 接收第三方请求
2. 包装请求
3. 编排调用
4. 委托本地能力
5. 返回结构化结果

那么它属于：

1. 第三层：远程 Skill / 代理授权 Skill

## 常见场景对照表

| 场景 | 建议归属 | 说明 |
| --- | --- | --- |
| 根据私密记忆生成故事草稿 | 第一层 | 本地处理私密内容 |
| 对私密草稿进行润色 | 第一层 | 本地处理私密内容 |
| 读取受保护故事对象 | 第二层 | 本地访问控制 |
| 将编辑后的故事写回受保护对象 | 第二层 | 本地访问控制 |
| 第三方 Skill 请求签名 | 第三层 | 远程接入，本地执行 |
| 第三方 Skill 请求登录字段填充 | 第三层 -> 第二层 | 远程入口，本地授权读取 |

## 反模式清单

不要这样做：

1. 让远程 Skill 直接读取本地故事对象
2. 让远程 Skill 持有私钥
3. 让故事处理层自己绕过访问层读取对象
4. 让访问层直接承担故事生成逻辑

## 当前阶段推荐

当前阶段建议优先落地：

1. 第一层本地故事处理 Skill
2. 第二层本地故事访问 Skill
3. 第三层先定义接口边界，不急着完整实现
