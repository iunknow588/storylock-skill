# StoryLock 三包接口契约

## 目的

本文档定义以下三包之间的最小接口契约：

1. `storylock-local-story-processing-skill`
2. `storylock-local-story-access-skill`
3. `storylock-remote-gateway-skill`

目标是让三包之间的调用在实现前就有统一约束。

## 总体调用链

调用顺序固定为：

1. 第三包调用第二包
2. 第二包调用第一包
3. 第一包不直接对第三包开放

补充约束：

1. 第三包只能暴露白名单包装能力，不能暴露第二包内部接口
2. 第二包必须把第三包视为“不可信但可校验的调用方”
3. 第一包只处理第二包已批准交付的内容

## 第一包与第二包接口契约

### 契约 A：读取后交给本地故事处理

用途：

1. 第二包读取受保护故事对象
2. 再把读取结果交给第一包处理

请求结构：

```json
{
  "storyObjectId": "story-001",
  "storyObject": {
    "title": "private story",
    "content": "..."
  },
  "processingMode": "draft_refine",
  "processingPayload": {
    "goals": ["improve clarity"],
    "hintStyle": "short mnemonic cues"
  }
}
```

约束：

1. 第一包不负责 challenge
2. 第一包不负责 session
3. 第一包只处理第二包已经批准交付的内容

### 契约 B：处理结果写回请求

用途：

1. 第一包完成故事处理
2. 第二包决定是否允许写回受保护对象

写回请求结构：

```json
{
  "storyObjectId": "story-001",
  "content": {
    "title": "updated title",
    "content": "updated content"
  },
  "writePolicy": {
    "requiresChallenge": true,
    "scope": "story_write_basic"
  }
}
```

## 第二包与第三包接口契约

### 当前阶段允许对外暴露的第三包能力

1. `requestStoryRead`
2. `requestStoryWrite`
3. `requestChallengeSign`
4. `queryStoryMetadata`

### 当前阶段禁止第三包直接暴露的第二包内部接口

1. `createChallenge`
2. `submitChallengeAnswers`
3. `activateSession`
4. `readSecretObject`
5. `deriveKey`
6. `consumeReadBudget`
7. `expireSession`

### 契约 C：远程请求读取故事对象

能力名：

1. `requestStoryRead`

请求结构：

```json
{
  "requestId": "req-story-read-001",
  "capability": "requestStoryRead",
  "scope": "story_read_basic",
  "payload": {
    "identityId": "identity-001",
    "storyObjectId": "story-001",
    "answers": []
  },
  "policyHints": {
    "redactionPreferred": true
  },
  "requestedRetention": "result_only",
  "nonce": "n-001",
  "expiry": 1760000300
}
```

其中 `answers` 当前阶段建议采用如下结构：

```json
[
  {
    "questionId": "q-001",
    "answer": "normalized user answer",
    "answerFormat": "text"
  }
]
```

字段说明：

1. `questionId`
   - challenge 中对应的问题标识
2. `answer`
   - 本次提交的原始答案，只允许进入第二层本地校验链路
3. `answerFormat`
   - 当前阶段建议固定为 `text`
4. `nonce`
   - 用于第三层到第二层的最小防重放校验
5. `expiry`
   - 用于约束远程请求的有效时间窗

成功响应结构：

```json
{
  "requestId": "req-story-read-001",
  "status": "success",
  "capability": "requestStoryRead",
  "executionLocation": "local",
  "result": {
    "mode": "story_read_access",
    "storyObjectId": "story-001",
    "storyObject": {}
  },
  "redactionLevel": "none",
  "retentionGranted": "result_only",
  "auditMeta": {},
  "error": null
}
```

### 契约 D：远程请求写回故事对象

能力名：

1. `requestStoryWrite`

请求结构：

```json
{
  "requestId": "req-story-write-001",
  "capability": "requestStoryWrite",
  "scope": "story_write_basic",
  "payload": {
    "identityId": "identity-001",
    "storyObjectId": "story-001",
    "content": {},
    "answers": []
  },
  "policyHints": {
    "writeReason": "publish_update"
  },
  "requestedRetention": "result_only",
  "nonce": "n-002",
  "expiry": 1760000300
}
```

`requestStoryWrite` 中的 `answers` 字段沿用与 `requestStoryRead` 相同结构。

### 契约 E：远程请求本地签名

能力名：

1. `requestChallengeSign`

请求结构：

```json
{
  "requestId": "req-sign-001",
  "capability": "requestChallengeSign",
  "scope": "challenge_sign",
  "payload": {
    "identityId": "identity-001",
    "keyId": "wallet-key",
    "algorithm": "ed25519",
    "payload": "sign this payload",
    "resourceId": "eth-main",
    "primaryRole": "private_key"
  },
  "policyHints": {
    "minAccessLevel": "L4"
  },
  "requestedRetention": "result_only",
  "nonce": "n-003",
  "expiry": 1760000300
}
```

当前阶段签名请求中的 `algorithm` 建议采用分层校验：

1. 第三包检查结构是否合法、能力是否在白名单内
2. 第二包检查算法是否合法、`keyId` 是否支持该算法
3. 第二包不得因为第三包已放行就跳过本地语义校验

## 统一字段约定

当前阶段统一采用 `camelCase` 字段命名。

原因是：

1. 现有实现主要面向 JavaScript / TypeScript
2. 可减少文档与代码字段映射成本
3. 避免 `execution_location` / `executionLocation` 这类双写混乱

所有跨包请求建议统一包含：

1. `requestId`
2. `capability`
3. `scope`
4. `payload`
5. `policyHints`
6. `requestedRetention`
7. `nonce`
8. `expiry`

所有跨包响应建议统一包含：

1. `requestId`
2. `status`
3. `capability`
4. `executionLocation`
5. `result`
6. `redactionLevel`
7. `retentionGranted`
8. `auditMeta`
9. `error`

## 错误响应契约

错误响应也必须返回完整外层结构，不能只返回一个裸错误对象。

示例：

```json
{
  "requestId": "req-sign-001",
  "status": "error",
  "capability": "requestChallengeSign",
  "executionLocation": "local",
  "result": null,
  "redactionLevel": "full",
  "retentionGranted": "none",
  "auditMeta": {
    "errorCode": "SLG-003",
    "errorType": "CHALLENGE_FAILED",
    "timestamp": 1760000100
  },
  "error": {
    "code": "SLG-003",
    "type": "CHALLENGE_FAILED",
    "message": "挑战答案未通过验证",
    "suggestedAction": "允许重新提交答案",
    "retryable": true
  }
}
```

约束：

1. `status` 必须为 `error`
2. `result` 必须为 `null`
3. `auditMeta` 只保留最小故障审计信息
4. `error.message` 可面向调用方暴露，但不得泄露 challenge 细节或秘密材料

## 禁止越权规则

1. 第三包不得直接调用第一包
2. 第一包不得绕过第二包直接访问受保护对象
3. 第二包不得把私钥、challenge answers、长期 session 能力返回给第三包
4. 第三包不得缓存长期可复用的本地访问能力
5. 第三包不得把第二包内部接口重新包装成外部可调用能力

## 当前阶段最小实现建议

先实现以下最小链路：

1. `requestStoryRead`
2. `requestStoryWrite`
3. `requestChallengeSign`

在这三条链路打通后，再扩展更复杂的 story processing 组装流程。
