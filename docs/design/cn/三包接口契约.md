# StoryLock 三包接口契约

## 当前基线

本文档以当前 `skill/src` 代码为准，定义三个运行包之间的最小接口契约：

1. `storylock-local-story-processing-skill`
2. `storylock-local-story-access-skill`
3. `storylock-remote-gateway-skill`

旧评审中提到的 story read/write、`requestChallengeSign`、能力状态查询等接口不再作为当前设计基线。

## 三包职责

| 包 | 层级 | 当前职责 |
| --- | --- | --- |
| `storylock-local-story-processing-skill` | 第一层 | 本地文本处理、故事草稿/润色、24 题题集强度评估 |
| `storylock-local-story-access-skill` | 第二层 | 对象强度判断、九宫格验证、本地答案校验、短时授权结果与审计 |
| `storylock-remote-gateway-skill` | 第三层 | 远程请求包装、字段校验、EIP-712 请求表达、可选本地执行器调用、结果脱敏 |

## 当前主能力

### 第一层

1. `StoryDraftSkill`
2. `StoryRefineSkill`
3. `StrengthReviewSkill`

第一层不创建 challenge，不签发 session，不读取 secret，不直接面向第三方暴露敏感输入。

### 第二层

1. `ObjectStrengthPolicySkill`
2. `GridChallengeSkill`
3. `LocalAuthorizationSkill`

第二层不再承担故事对象读取或写回职责。当前 SQLite host 只保存 challenge、session、request、nonce、failure window、answer digest 与 audit log。

### 第三层

1. `requestSignature`
2. `requestPasswordFill`

第三层不暴露第二层内部能力，不返回九宫格答案、私钥、密码、signing key bytes 或长期 session material。

## 调用链

推荐链路：

1. 第三层接收远程结构化请求。
2. 第三层校验 `requestId`、`nonce`、`expiry`、`capability` 和字段形状。
3. 第三层可通过 `transport` 转交本地 Host，也可通过可选 `signatureExecutor` / `passwordFillExecutor` 执行本地能力。
4. 第二层按对象与动作判断强度，生成九宫格验证并完成本地授权。
5. 本地 Host 在授权通过后执行签名或密码填充。
6. 第三层统一脱敏返回结果。

## 第一层强度评估契约

请求：

```json
{
  "questions": [
    {
      "nodeId": "q-001",
      "validAnswers": ["answer"],
      "distractors": ["d1", "d2", "d3", "d4", "d5", "d6", "d7", "d8"]
    }
  ]
}
```

约束：

1. `questions` 必须包含 24 个节点。
2. 每题必须能形成 9 个唯一候选项。
3. 输出只包含强度评估、问题数量、建议动作与边界标记。

## 第二层授权契约

### 对象强度判断

请求：

```json
{
  "identityId": "identity-001",
  "objectRef": "wallet-key-main",
  "objectType": "signature_key",
  "requestedAction": "signature"
}
```

响应核心字段：

```json
{
  "requiredStrength": "high",
  "gridPolicy": {
    "gridSize": 9,
    "requiredCells": 9
  }
}
```

### 九宫格验证生成

请求必须包含：

1. `identityId`
2. `objectRef`
3. `requiredStrength`
4. `requestId`
5. `nonce`
6. `expiry`

响应包含 `verificationId`、九宫格 cell 描述与过期时间。cell 中不得包含答案原文。

### 本地授权

请求：

```json
{
  "identityId": "identity-001",
  "objectRef": "wallet-key-main",
  "verificationId": "chl-001",
  "allowedAction": "signature",
  "answers": [
    {
      "cellId": "cell-1",
      "answer": "normalized answer"
    }
  ]
}
```

成功响应包含：

1. `approved`
2. `authorizationId`
3. `identityId`
4. `objectRef`
5. `allowedAction`
6. `expiresAt`

预算规则：

1. `signature` 与 `password_fill` 默认 `readBudget=1, writeBudget=0`。
2. 普通 `authorize` 默认 `readBudget=0, writeBudget=0`。
3. session 只表示短时授权结果，不是长期能力。

## 第三层签名请求契约

`requestSignature` 输入：

```json
{
  "requestId": "req-sign-001",
  "nonce": "nonce-001",
  "expiry": 1760000300,
  "identityId": "identity-001",
  "keyId": "wallet-key",
  "algorithm": "ed25519",
  "payload": "sign this payload",
  "resourceId": "wallet-main",
  "primaryRole": "private_key",
  "eip712Nonce": "1001"
}
```

第三层生成 `StoryLockSignatureRequest` EIP-712 结构，并通过 `transport` 或 `signatureExecutor` 交给本地执行环境。

## 第三层密码填充契约

`requestPasswordFill` 输入：

```json
{
  "requestId": "req-fill-001",
  "nonce": "nonce-002",
  "expiry": 1760000300,
  "identityId": "identity-001",
  "credentialRef": "cred-example",
  "targetOrigin": "https://example.com"
}
```

第三层默认设置：

1. `capability=requestPasswordFill`
2. `scope=password_fill_basic`
3. `requestedRetention=audit_meta_only`
4. `policyHints.noRemoteSecretReturn=true`

## 统一错误码

当前代码错误码以 `SLG` 前缀输出：

| 错误码 | 类型 | 含义 |
| --- | --- | --- |
| `SLG-001` | `validation_error` | 请求字段不合法 |
| `SLG-002` | `replay_rejected` | 请求重放被拒绝 |
| `SLG-003` | `challenge_failed` | 九宫格答案未通过 |
| `SLG-004` | `challenge_locked` | 连续失败后锁定 |
| `SLG-005` | `session_invalid` | session 无效 |
| `SLG-006` | `budget_exhausted` | 预算耗尽 |
| `SLG-007` | `object_not_found` | 对象不存在 |
| `SLG-008` | `redaction_required` | 结果必须脱敏 |
| `SLG-009` | `scope_insufficient` | scope 不足 |
| `SLG-010` | `secret_unavailable` | SecretStore 不可用 |
| `SLG-011` | `request_expired` | 请求过期 |
| `SLG-012` | `internal_error` | 内部错误 |
| `SLG-013` | `replay_detected` | requestId 或 nonce 冲突 |

## 禁止规则

1. 第三层不得直接暴露 `createChallenge`、`submitChallengeAnswers`、`issueSession`。
2. 第三层不得返回私钥、密码、九宫格答案或 signing key bytes。
3. 第二层不得承担故事正文读写职责。
4. 第一层不得绕过第二层访问受保护对象。
5. `storylock-skill-engine` 是兼容与演示包，不是新的第四层安全边界。
