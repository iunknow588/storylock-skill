# StoryLock 安全规范

| --- | --- |

## 目的

本文档把当前阶段必须冻结的安全实现口径集中到一处,避免分散在多个设计文档里导致遗漏。

## 当前阶段最小安全基线

当前阶段明确采用:

1. 根密钥派生与答案摘要分开处理
2. 题集主档与 challenge 运行态分开存储
3. 对称加密统一使用 AEAD
4. 长期敏感材料优先放入操作系统密钥链

## 算法基线

| 目标 | 当前阶段建议 |
| --- | --- |
| 对称加密 | `AES-256-GCM` |
| 摘要 / HMAC | `SHA-256` / `HMAC-SHA256` |
| 工作密钥派生 | `HKDF-SHA256` |
| 本地随机数 | 加密安全随机数,长度至少 32 字节 |

## 密钥层次建议

当前阶段建议采用以下层次:

1. `masterSalt`
2. `rootKey`
3. `workKey`
4. `objectKey`

## HKDF 派生建议

工作密钥派生建议:

1. 算法:`HKDF-SHA256`
2. `salt`:固定为当前身份上下文相关盐值
3. `info`:必须显式包含用途

示例:

```text
info = "storylock/workkey/" + identityId + "/" + purpose
```

## 题集主档加密规范

题集主档当前阶段建议:

1. 明确视为中敏感对象
2. 落盘前使用 `AES-256-GCM`
3. 加密密钥来自 `rootKey` 或其派生工作密钥
4. 不允许明文长期落盘

## GCM nonce 管理

当前阶段统一要求:

1. 每次加密独立生成 96-bit 随机 nonce
2. 同一密钥下不得复用 nonce
3. nonce 必须与密文一起存储

## GCM nonce 存储格式

当前阶段建议固定两种可接受格式,避免不同实现各写一套。

### 方案 A:二进制拼接格式

推荐格式:

```text
nonce(12 bytes) || ciphertext(n bytes) || tag(16 bytes)
```

适用:

1. 本地二进制文件
2. SQLite BLOB 字段
3. 需要紧凑存储的场景

### 方案 B:结构化 JSON 格式

推荐格式:

```json
{
  "algorithm": "AES-256-GCM",
  "nonce": "base64...",
  "ciphertext": "base64...",
  "tag": "base64..."
}
```

适用:

1. 调试与审计友好场景
2. 需要跨语言序列化的场景

当前阶段建议:

1. 落盘主格式优先使用方案 A
2. 跨接口或文档示例优先使用方案 B

## masterSalt 规范

`masterSalt` 建议:

1. 在单故事终身制下，应被视为**稳定根上下文的本地受控表示**
2. 长度至少 32 字节
3. 长期保存于操作系统密钥链
4. 不允许明文写入普通配置文件

说明：

1. 当前阶段不建议把原始故事文本简单哈希后直接当成所有密钥
2. 更合理的做法是先形成稳定根上下文，再把 `masterSalt` 作为本地受控根参数使用

身份盐值建议:

```text
identitySalt = HMAC-SHA256(masterSalt, identityId)
```

## 答案摘要规范

challenge answers 当前阶段建议:

1. 先做本地规范化
2. 再使用 `identitySalt`
3. 最后计算 `HMAC-SHA256(identitySalt, normalizedAnswer)`

## 长期密钥存储原则

长期敏感材料包括:

1. `masterSalt`
2. 签名根密钥
3. 恢复材料包装所需长期密钥

当前阶段要求:

1. 优先操作系统密钥链
2. 不直接入 SQLite 普通表
3. 不直接入 `.env` 明文文件

## 密钥轮换策略（用户确认版）

经用户确认,StoryLock 采用**单故事终身制**密钥管理策略,核心原则如下:

### 核心原则

1. **单故事绑定**:每个用户只维护一个故事根上下文,所有密钥均从该故事派生
2. **根默认稳定**:未出现高风险因素前,不主动轮换故事根或 masterSalt
3. **故事优化而非替换**:需要更新时,优先在原有故事基础上做精确化优化
4. **用户零记忆负担**:用户只需记住一个故事,即可管理所有密钥
5. **常规轮换不等于换根**:题集、派生上下文、对象封装层可轮换,但不默认触发根级重建

### 风险触发条件（满足任一即提示优化或重建）

| 风险等级 | 触发条件 | 响应方式 |
|----------|---------|----------|
| 高 | 检测到底层密钥被暴力破解尝试（如 challenge 失败次数异常激增） | 强制提示,建议立即优化故事；若根上下文已泄露则进入根级重建 |
| 高 | 用户主动报告故事可能泄露（如被他人知晓） | 强制提示,建议立即优化故事；必要时进入根级重建 |
| 中 | 长期未使用（超过 12 个月）且题集强度评估下降 | 建议性提示,推荐优化故事 |
| 低 | 系统检测到新的安全标准或算法过时 | 信息性提示,可选优化 |

### 故事优化流程（非替换）

当触发常规风险时,执行**增量优化**而非完全替换:

1. **保留核心故事骨架**:用户原有故事的主体内容不变
2. **增加精确化细节**:引导用户补充更具体的细节（如时间、地点、感官记忆）
3. **扩展题集问题**:基于优化后的故事生成新的问题,加入原有 24 题题集
4. **平滑过渡**:新题集与旧题集并行存在一段时间,用户可选择回答任一题集
5. **渐进式淘汰**:旧题集在确认新题集稳定后标记为 deprecated,不再用于新 challenge

### 根级重建边界

只有在以下情况才进入根级重建:

1. 故事根被确认泄露
2. 现有故事根无法通过常规优化恢复可信度
3. 用户主动要求完全重建

根级重建时,需要同时重建:

1. 故事根上下文
2. masterSalt
3. identitySalt
4. 题集主档
5. 对象封装层

根级重建不是常规轮换,不得频繁触发。

### 单故事模式下的轮换对象

在该模式下,默认不把“轮换”理解成“换故事”。

当前阶段建议按下列层级处理:

1. **故事根上下文**
   - 默认不变
   - 仅在灾难级风险时重建
2. **题集版本**
   - 可扩展
   - 可标记 `active` / `deprecated` / `pending`
3. **派生上下文版本**
   - 可通过 `versionTag` 调整
   - 不要求用户感知
4. **对象封装层**
   - 可重加密
   - 可切换到新的 objectKey / workKey 派生上下文

### 密钥层次与故事的关系

```
用户故事（唯一、终身）
    │
    ├── masterSalt（从故事派生,固定不变）
    │       └── 通过 HKDF 派生各层密钥
    │
    ├── 题集主档（24题,可扩展至 24+N 题）
    │       └── 新题加入时,旧题标记 deprecated
    │
    └── identitySalt（按 identity 派生,固定不变）
            └── 用于答案摘要和身份隔离
```

### 与原有密钥层次的关系

原有四层密钥层次（masterSalt → rootKey → workKey → objectKey）保持不变,但:

- **masterSalt**:从用户故事中一次性派生,终身不变
- **rootKey / workKey / objectKey**:通过 HKDF 按需派生,派生参数中包含版本标识
- **版本标识**:仅用于区分同一故事不同优化阶段的派生上下文,不强制用户感知

### 实现约束

1. **SecretStore 中 masterSalt 不可删除**:作为终身根密钥,必须永久保留
2. **题集主档支持版本标记**:每道题标记 `active` / `deprecated` / `pending` 状态
3. **Challenge 创建时优先选择 active 题**:确保新 challenge 不使用已淘汰题目
4. **答案验证时兼容多版本**:同一 identity 的历史答案摘要按版本隔离存储
5. **对象封装支持版本标记**:题集主档和秘密对象应有封装版本或派生上下文标记
6. **根级重建属于灾难恢复**:不得作为常规轮换路径频繁触发
7. **兼容窗口必须显式配置**:旧题集摘要、旧对象密文、旧 session 均应有明确失效时间

### 用户交互设计

- **正常情况**:用户完全无感知,只需记住一个故事
- **风险提示时**:提示文案强调"优化故事"而非"更换故事",降低用户焦虑
- **优化引导**:通过对话式引导帮助用户补充细节,而非要求重新构思新故事

## 结论

当前阶段最重要的不是把安全功能铺得很宽,而是先把以下五件事做对:

1. 密钥分层
2. 题集加密
3. nonce 唯一性
4. 答案摘要盐值隔离
5. 长期敏感材料不上普通磁盘明文

**补充原则（单故事终身制）**:

6. 用户只需记住一个故事,所有密钥从该故事派生
7. 不主动轮换,仅在风险触发时提示优化故事
8. 故事优化是增量过程,而非完全替换
9. 题集可扩展,旧题渐进式淘汰而非立即删除
