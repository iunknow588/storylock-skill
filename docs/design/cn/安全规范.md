# StoryLock 安全规范

| 项目 | 内容 |
| --- | --- |
| 文档版本 | v1.1 |
| 日期 | 2026-06-20 |
| 状态 | 当前主线采用 |
| 适用范围 | 第一层、第二层、第三层共享安全约束 |

## 目标

本文档汇总当前主线必须统一遵守的安全实现口径，避免关键约束分散在多份设计文档中。

## 当前阶段的最小安全基线

当前主线明确采用：

1. 根密钥派生与答案摘要分开处理。
2. 题集主文件与 challenge 运行态分开存储。
3. 对称加密统一使用 AEAD。
4. 长期敏感材料优先进入平台 SecretStore。
5. 远程响应默认经过第三层统一脱敏。
6. 平台 SecretStore 依赖缺失时，第二层 Host 启动必须直接失败。

## 算法基线

| 目标 | 当前建议 |
| --- | --- |
| 对称加密 | `AES-256-GCM` |
| 摘要 / HMAC | `SHA-256` / `HMAC-SHA256` |
| 工作密钥派生 | `HKDF-SHA256` |
| 本地随机数 | 加密安全随机数，长度至少 32 字节 |

## 密钥层次建议

当前建议采用以下层次：

1. `masterSalt`
2. `rootKey`
3. `workKey`
4. `objectKey`

其中：

1. `masterSalt` 作为长期稳定的本地根上下文。
2. `rootKey` 由 `masterSalt` 派生。
3. `workKey` 按身份、用途和版本派生。
4. `objectKey` 用于具体对象或封装层。

## HKDF 派生建议

工作密钥派生建议：

1. 算法使用 `HKDF-SHA256`。
2. `salt` 与当前身份上下文相关。
3. `info` 必须显式包含用途。

示例：

```text
info = "storylock/workkey/" + identityId + "/" + purpose
```

## 题集主文件加密规范

题集主文件当前建议：

1. 明确视为中高敏对象。
2. 落盘前使用 `AES-256-GCM`。
3. 加密密钥来自 `rootKey` 或其派生工作密钥。
4. 不允许明文长期落地。

## GCM nonce 管理

当前统一要求：

1. 每次加密独立生成 96-bit 随机 nonce。
2. 同一密钥下不得复用 nonce。
3. nonce 必须与密文一起存储。

## GCM nonce 存储格式

当前建议接受两种格式。

### 方案 A：二进制拼接

推荐格式：

```text
nonce(12 bytes) || ciphertext(n bytes) || tag(16 bytes)
```

适用：

1. 本地二进制文件。
2. SQLite BLOB 字段。
3. 需要紧凑存储的场景。

### 方案 B：结构化 JSON

推荐格式：

```json
{
  "algorithm": "AES-256-GCM",
  "nonce": "base64...",
  "ciphertext": "base64...",
  "tag": "base64..."
}
```

适用：

1. 调试与审计友好场景。
2. 跨语言序列化场景。

当前建议：

1. 落盘主格式优先方案 A。
2. 文档示例、接口包装和调试信息优先方案 B。

## masterSalt 规范

`masterSalt` 建议：

1. 在单故事终身制下，视为稳定根上下文的本地受控表示。
2. 长度至少 32 字节。
3. 长期保存在平台 SecretStore。
4. 不允许明文写入普通配置文件。

当前代码口径：

1. 生产路径下，持久化 Host 必须使用受控 SecretStore。
2. 仅开发模式允许显式使用 `MemorySecretStore`。
3. 使用平台 SecretStore 的 Host 启动时，会先执行可用性检查。
4. 平台依赖缺失时直接报错，不进入模糊降级路径。

## 答案摘要规范

challenge answers 当前建议：

1. 先做本地规范化。
2. 再使用身份相关上下文隔离。
3. 最后计算 `HMAC-SHA256` 摘要。

要求：

1. 不保存明文答案。
2. 不把答案摘要直接暴露给第三层或第三方。
3. 多版本题集下按版本隔离摘要。

## 长期密钥存储原则

长期敏感材料包括：

1. `masterSalt`
2. 签名根材料或平台签名别名
3. 恢复材料包装密钥

当前要求：

1. 优先使用平台 SecretStore。
2. 不直接进入 SQLite 普通表。
3. 不直接进入 `.env` 明文文件。
4. 第二层 Host 启动时必须先显式检查平台 SecretStore 可用性。

## 开发模式约束

当前代码已经收紧为：

1. `MemorySecretStore` 必须显式 `developmentMode=true`。
2. `createPlatformSecretStore({ allowMemoryFallback: true })` 仍要显式要求 `developmentMode=true`。
3. 使用平台 SecretStore 的 Host 启动时必须先执行可用性检查。
4. 启动时应输出开发专用警告。

开发模式下仍必须遵守：

1. 不允许静默把长期秘密改为明文文件存储。
2. 不允许绕过 challenge、session、scope、expiry 校验。
3. 不允许把本地敏感结果直接暴露给第三层。
4. 不得把开发回退描述为生产安全配置。

## 请求与会话安全

当前主线要求：

1. 所有远程请求携带 `requestId`、`nonce`、`expiry`。
2. 过期请求必须拒绝，错误码使用 `SLG-011`。
3. replay 冲突必须拒绝，错误码使用 `SLG-013`。
4. 本地授权结果只形成短时 `session`，不形成长期远程能力令牌。

## 远程响应安全

第三层 `remote-gateway` 当前必须：

1. 对远程响应递归脱敏。
2. 对敏感字段执行 `[redacted]` 替换。
3. 发现未脱敏敏感字段时抛出 `SLG-008`。

当前默认高敏字段包括：

1. `answers`
2. `signingKey`
3. `signingKeyBytes`
4. `secretBytes`
5. `secretValue`
6. `password`
7. `privateKey`
8. `mnemonic`
9. `seed`
10. `rawSecret`
11. `keyMaterial`
12. `token`
13. `accessToken`
14. `refreshToken`
15. `apiKey`

## 单故事终身制补充原则

当前项目采用单故事终身制口径：

1. 用户只需维护一个稳定故事根上下文。
2. 默认不主动轮换 `masterSalt`。
3. 题集、派生上下文和对象封装层可以版本化演进。
4. 只有在根上下文确认泄露或失信时才进入根级重建。

## 结论

当前阶段最重要的，不是把安全功能铺得很满，而是先把这些基础做扎实：

1. 密钥分层。
2. 题集加密。
3. nonce 唯一性。
4. 答案摘要隔离。
5. 长期敏感材料走平台 SecretStore。
6. 平台 SecretStore 缺依赖时 fail fast。
7. 开发模式不能伪装成生产安全配置。
8. 第三层远程响应必须强制脱敏。
