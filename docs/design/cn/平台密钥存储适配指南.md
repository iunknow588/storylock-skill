# StoryLock 平台密钥存储适配指南

| 项目 | 内容 |
| --- | --- |
| 文档版本 | v1.2 |
| 日期 | 2026-06-20 |
| 状态 | 当前主线采用 |
| 适用范围 | 本地 Host、第二层、执行器 |

## 目标

本文档用于统一说明 StoryLock 长期敏感材料在不同平台上的存储方式、当前代码已经实现到什么程度，以及宿主启动时的安全守门行为。

## 建议放入平台密钥存储的对象

当前建议优先进入平台 SecretStore 的对象包括：

1. `storylock/masterSalt`
2. 本地签名根材料或平台签名别名
3. 恢复材料包装密钥
4. 平台级凭据别名与最小元数据

## 当前代码适配现状

`src/shared/secret-store.js` 当前提供：

1. `WindowsCredentialSecretStore`
2. `LinuxSecretServiceStore`
3. `MacOSKeychainSecretStore`
4. `MemorySecretStore`
5. `createPlatformSecretStore(...)`

当前主线结论：

1. Windows 已接入 CredentialManager PowerShell 模块路径。
2. Linux 已接入 `secret-tool` / Secret Service 路径。
3. macOS 已接入基于 `security` CLI 的 Keychain 路径。
4. `MemorySecretStore` 只允许在 `developmentMode=true` 时显式启用。
5. 第二层 Host 在启动时会对非 memory SecretStore 显式执行 `checkAvailable()`。
6. 如果平台依赖缺失，Host 必须直接失败，不能静默降级到明文存储或 MemorySecretStore。

## Windows

当前建议优先级：

1. Credential Manager / 系统受保护凭据能力。
2. 后续如有需要，再补更强的平台密钥容器封装。

当前实现特点：

1. 使用 `storylock/<key>` 形式组织目标名。
2. 通过 PowerShell `CredentialManager` 模块读写。
3. 默认不支持 `listKeys`。
4. 支持 `checkAvailable()` 作为启动前可用性检查。

部署要求：

1. 目标环境必须具备 `CredentialManager` PowerShell 模块。
2. Host 启动阶段会先执行 `checkAvailable()`。
3. 若模块缺失，应直接报错并阻止启动。

## Linux

当前建议：

1. 优先使用 Secret Service / `secret-tool`。
2. 在桌面环境可用时使用 `secret-tool store` / `lookup` / `clear`。
3. 不建议降级为明文文件。

当前实现特点：

1. 默认 schema/service 为 `storylock`。
2. 默认不支持 `listKeys`。
3. 支持 `checkAvailable()` 作为启动前可用性检查。

部署要求：

1. 目标环境必须具备 `secret-tool` 且能访问 Secret Service。
2. Host 启动阶段会先执行 `checkAvailable()`。
3. 若依赖不可用，应直接报错并阻止启动。

## macOS

当前设计建议：

1. 正式平台适配器优先使用 Keychain。
2. 只把长期敏感材料放入 Keychain。
3. 运行态 `session`、`nonce`、`requestId` 不进入 Keychain。

当前实现要点：

1. 代码中已经提供 `MacOSKeychainSecretStore`。
2. 通过 `security add-generic-password`、`find-generic-password -w`、`delete-generic-password` 读写 Keychain。
3. `checkAvailable()` 通过 `security list-keychains` 做启动前可用性检查。
4. 默认不支持 `listKeys`，与 Windows / Linux 保持一致的保守边界。
5. 第二层 Host 继承同样的启动期 fail-fast 守门逻辑。

部署要求：

1. 目标环境必须具备可用的 `security` CLI。
2. 当前用户会话必须能访问 Keychain 服务。
3. Host 启动阶段会先执行 `checkAvailable()`。
4. 若 Keychain 不可用，应直接报错并阻止启动。

## 统一接口语义

当前接口语义可视为：

```ts
interface SecretStore {
  getSecret(key: string): Promise<Uint8Array | null> | Uint8Array | null;
  setSecret(key: string, value: Uint8Array): Promise<void> | void;
  deleteSecret(key: string): Promise<void> | void;
  listKeys?(prefix: string): Promise<string[]> | string[];
  checkAvailable?(): Promise<void> | void;
}
```

说明：

1. 当前主线实现以同步接口为主。
2. `checkAvailable()` 用于 Host 启动前的显式依赖检查。
3. 未来如接入更多平台或远程桥接，可平滑扩展为异步形式。

## 命名建议

建议统一使用：

1. `storylock/masterSalt`
2. `storylock/signingRoot/<identityId>`
3. `storylock/recoveryKey/<identityId>`
4. `storylock/signature/<keyId>`

命名要求：

1. 前缀稳定。
2. 身份相关对象显式带上 `identityId` 或 `keyId`。
3. 不把业务敏感含义直接写入别名正文。

## 不应进入平台密钥存储的对象

以下对象当前更适合进入 SQLite 或受控运行态存储：

1. `requestId` 去重记录
2. `nonce` 去重记录
3. `session` 运行态记录
4. 大体积题集密文主体
5. 审计日志

## 降级原则

当平台安全存储不可用时：

1. 默认返回明确错误。
2. 第二层 Host 对平台存储先做显式守门检查。
3. 只有调用方显式进入开发模式时，才允许使用 `MemorySecretStore`。
4. 开发模式必须保留风险提示。

不允许：

1. 静默写入明文文件。
2. 静默切换到生产不可接受的弱存储。
3. 平台 SecretStore 检查失败后继续按生产路径运行。

## 开发模式边界

当前代码已经明确：

1. `new MemorySecretStore()` 必须显式传入 `developmentMode=true`。
2. `createPlatformSecretStore({ allowMemoryFallback: true })` 也必须显式要求 `developmentMode=true`。
3. `createAccessHost(...)` 在使用平台 SecretStore 时会先执行 `checkAvailable()`。
4. 默认会输出“仅供开发使用，不得用于生产”的警告。

开发模式下仍必须满足：

1. 仅限本地开发或测试。
2. 审计或日志中应能识别 `developmentMode`。
3. 不得表述为生产安全配置。

## 当前阶段结论

当前阶段最重要的主线约束是：

1. 长期敏感材料统一走 `SecretStore`。
2. 普通运行态数据统一走 SQLite。
3. 平台存储依赖缺失时，Host 启动必须 fail fast。
4. Memory 模式只能作为显式开发回退，不能伪装成生产路径。

下一步重点仍然是：

1. 继续做 macOS 本机验收记录与说明固化。
2. 继续推进 Android 真机签名与平台存储正式化。
3. 在多平台宿主上统一启动期可用性检查与错误文案。
