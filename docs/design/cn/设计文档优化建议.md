# StoryLock 设计文档优化建议

本文是历史整理建议的收敛版，只保留对当前代码仍有价值的内容。当前实现基线以以下文档为准：

1. `README.md`
2. `系统Skill表与能力边界.md`
3. `三包接口契约.md`
4. `本地Agent网关设计.md`
5. `Session与防重放策略.md`

## 已经完成的收敛

1. 三层主线已经收敛为故事处理、本地访问授权、远程网关三个包。
2. 第二层已经从“故事读取/写回”收敛为“对象强度、九宫格验证、本地授权”。
3. 第三层已经从 `requestChallengeSign` 收敛为更中性的 `requestSignature`。
4. 第三层当前只保留 `requestSignature` 与 `requestPasswordFill` 两个主接口。
5. 签名审计与授权状态已经落到 SQLite。

## 仍有价值的优化方向

### 1. 继续统一术语

建议所有主线文档统一使用：

| 推荐术语 | 不再作为主线术语 |
| --- | --- |
| `requestSignature` | `requestChallengeSign` |
| `SignatureAuthorizationSkill` | `ChallengeSigningAuthorizationSkill` |
| 本地访问授权 | 本地故事访问 |
| 对象强度策略 | 故事读取强度 |
| 九宫格验证 | 故事读取 challenge |

### 2. 文档持续按代码校准

每次改动代码后，至少检查：

1. `三包接口契约.md`
2. `系统Skill表与能力边界.md`
3. `本地Agent网关设计.md`
4. `Session与防重放策略.md`
5. `开发完善实施计划_20260617.md`

### 3. 保持功能描述克制

文档描述应区分三类状态：

1. 已实现：代码中已经存在并有自测覆盖。
2. 计划完善：已有接口或雏形，但还需要补齐生产约束。
3. 后续探索：只是应用方向，不写成当前能力。

尤其是多账户交易、多平台发布、多链钱包等方向，只能作为“面向应用领域探索典型场景”，不能写成当前已经支持。

### 4. 继续补充验收标准

后续新增能力时，建议每一项都写清楚：

1. 涉及哪个包。
2. 是否改变对外接口。
3. 是否涉及本地敏感材料。
4. SQLite 是否需要迁移。
5. 自测命令是什么。

## 不再采纳的历史建议

以下建议已经失效，不再作为后续开发依据：

1. 增加故事读取、故事写回作为第二层主接口。
2. 在第三层保留 `requestStoryRead`、`requestStoryWrite`。
3. 不应把已废弃旧接口 `requestChallengeSign` 作为主签名接口。
4. 把远程网关描述为可直接读取本地故事或 secret object。
5. 把 Pharos Skill Engine 放在本地安全访问边界内。

## 当前优先事项

1. 补齐端到端演示：远程请求、第二层授权、本地签名或密码填充、第三层脱敏返回。
2. 明确 Node.js 22 运行时要求。
3. 扩展九宫格问题来源与强度策略，而不是只使用占位数据。
4. 为 SQLite 表结构和错误码维护稳定文档。
5. 为生产环境 SecretStore 接入系统密钥链预留实现。
