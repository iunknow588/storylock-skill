## 2. 背景与摘要

数字化个人代理正在把传统的“用户亲自授权”逐步转向“Agent 自主发起请求”。这提高了自动化效率，也把更多敏感能力暴露给远端服务环境。

因此需要一种机制，把签名授权、密码填充和敏感对象访问从中心化服务侧拉回到用户可控的本地设备，同时保留 Agent 协作的便利性。

易安与 StoryLock 的分工正是基于这一目标：

1. 远端 Agent 可以提出请求
2. 私人智能助理可以解释请求来源和风险
3. StoryLock 本地核心负责 challenge、授权和敏感执行
4. 用户始终在本地设备上完成关键确认

一句话概括：

> 易安负责远程入口和本地确认接入，StoryLock 负责本地敏感执行与授权边界，二者共同把远程请求安全地拉回用户设备侧完成。

## 3. 产品定位

易安是 StoryLock 使用流程中的远程访问入口与本地确认入口。它把来自外部 Agent、云服务或自动化流程的重要请求送达用户侧，再由用户自己的本地设备完成解释、确认和敏感执行。

从产品定位上看，系统承担三类职责：

1. 远程访问入口  
   对外提供下载、绑定、请求接入和状态查看能力。
2. 本地确认入口  
   用户安装的易安 App 运行在可控本地设备上，承载私人智能助理。
3. 本地授权与秘密管理  
   StoryLock 本地核心管理故事文件、文件密钥、私钥和敏感对象访问策略。

因此，易安 App 是连接“易安远程入口”和“StoryLock 本地核心”的本地确认入口。

## 4. 运行结构

从软件运行结构上，系统分为三部分：

1. 易安远程入口  
   部署在云平台上，负责说明入口、下载入口、绑定入口、请求入口和状态视图。
2. 私人智能助理  
   运行在用户可控的本地设备上，接收请求、解释来源和风险，并把需要敏感执行的部分交给 StoryLock 本地核心。
3. StoryLock 本地核心  
   运行在本地设备侧，负责故事文件、密钥隔离、本地确认和敏感执行。

整体请求路径如下：

```text
外部 Agent / 云服务 / PHAROS Agent / OpenClaw
  -> 易安远程入口
  <-> 私人智能助理
  <-> StoryLock 本地核心
```

关键边界是：

1. 易安远程入口不直接访问 StoryLock 本地核心
2. StoryLock 本地核心只把最小必要结果返回给私人智能助理
3. 敏感执行始终保留在本地设备侧

## 5. 三层 Skill 能力

### 5.1 第一层：本地故事处理

| 项目 | 内容 |
| --- | --- |
| 代码目录 | `src/skills/local-story-processing` |
| 主要能力 | 故事草稿生成、故事润色、题集强度评估 |
| 典型 Skill | `StoryDraftSkill`、`StoryRefineSkill`、`StrengthReviewSkill` |

第一层负责提供可解释的故事与记忆线索材料，为后续授权判断提供上下文。

### 5.2 第二层：本地受控授权

| 项目 | 内容 |
| --- | --- |
| 代码目录 | `src/skills/local-story-access` |
| 主要能力 | 对象级强度策略、九宫格验证、本地授权、短时会话、防重放 |
| 典型 Skill | `ObjectStrengthPolicySkill`、`GridChallengeSkill`、`LocalAuthorizationSkill`、`LocalRevocationSkill` |

第二层集中处理“是否允许访问某个敏感对象”“需要多高强度确认”“授权窗口维持多久”等问题。

### 5.3 第三层：远程网关与受控委托

| 项目 | 内容 |
| --- | --- |
| 代码目录 | `src/skills/remote-gateway` |
| 主要接口 | `requestSignature`、`requestPasswordFill` |
| 主要能力 | 远程请求封装、EIP-712 表达、Web2 密码填充请求包装、最小结果返回 |

第三层让远端 Agent 可以使用本地能力，同时把实际授权、签名、密码填充和敏感对象访问保留在本地确认链路中。

## 7. 核心机制

### 7.1 关联记忆线索

StoryLock 使用故事、提示和题集作为关联记忆线索，辅助用户完成可理解、可回忆的授权确认。

它承担三类作用：

1. 帮助用户理解敏感对象与授权请求之间的关系
2. 为不同对象配置不同强度的本地确认方式
3. 降低长期使用中遗忘固定口令或机械凭据的风险

### 7.2 请求状态

请求状态是系统运行时的核心视图。一个请求通常经历以下状态：

1. 已创建
2. 已接收
3. 已送达
4. 待确认
5. 已确认或暂不确认
6. 已完成

### 7.3 本地确认

本地确认包含两个层次：

1. 用户确认  
   用户查看来源、内容和风险，判断请求是否与当前操作一致。
2. 设备确认  
   设备通过解锁、生物识别或设备凭据确认用户意图。

### 7.4 最小结果返回

StoryLock 本地核心完成敏感操作后，只把最小必要结果返回给私人智能助理，再由私人智能助理按需同步状态给易安远程入口。

## 8. 安全机制

| 能力 | 说明 |
| --- | --- |
| 本地持有 | 私钥、文件密钥、故事文件和敏感材料由本地设备侧管理 |
| 对象级强度策略 | 根据目标对象决定所需确认强度 |
| 九宫格验证 | 根据确认强度生成本地验证挑战 |
| 防重放 | 使用 `requestId`、`nonce` 和过期时间约束重放与过期请求 |
| 短时会话 | 通过短时 session 控制授权窗口 |
| 自动锁定与恢复 | 连续失败后进入锁定窗口，到期后恢复 |
| 对象加密 | 使用加密机制保护本地对象内容 |
| 答案摘要 | 使用摘要保存 challenge 答案，避免明文存储 |

## 9. 安装与校验

易安 App 下载安装前，应核对版本号、文件大小和 SHA-256 校验值。

建议流程：

1. 从易安远程入口进入下载入口
2. 记录页面显示的版本号、文件大小和 SHA-256
3. 下载完成后，对本地文件计算 SHA-256
4. 比对本地文件大小和 SHA-256 是否与页面一致
5. 一致后再继续安装

## 10. 典型应用场景

| 场景 | StoryLock 的作用 |
| --- | --- |
| Web2 密码填充 | 远端 Agent 发起请求，用户在本地设备确认目标站点和请求来源，StoryLock 本地核心完成本地授权 |
| 签名确认 | 外部 Agent、PHAROS Agent 或 OpenClaw 通过 Skill 发起签名确认请求，本地核心返回最小必要结果 |
| 故事文件与敏感对象访问 | 根据目标对象执行强度策略，生成本地验证流程，并在确认后返回授权结果 |
| 多账号与多对象协作 | 为不同账号、凭据对象和本地文件配置不同确认强度 |

## 11. 总结

易安与 StoryLock 组成了一套面向 Agent 场景的本地授权和敏感对象管理方案。

它的核心价值包括：

1. 把远端请求带回用户本地设备确认
2. 用三层 Skill 结构拆分故事处理、本地访问授权和远程委托入口
3. 用关联记忆线索辅助用户完成可理解、可回忆的授权判断
4. 用对象级强度策略、九宫格验证、防重放、短时会话和最小结果返回控制敏感对象访问
5. 让远端 Agent 可以使用本地能力，同时保留用户在本地设备上的最终确认权
