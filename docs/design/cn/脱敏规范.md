# StoryLock 脱敏规范

| 项目 | 内容 |
| --- | --- |
| 文档版本 | v1.0 |
| 日期 | 2026-06-20 |
| 状态 | 当前主线采用 |
| 适用层级 | 第一层、第二层、第三层 |

## 目标

本文档用于明确 StoryLock 在“本地处理、远程包装、第三方集成”链路中如何做脱敏。

当前主线必须回答三件事：

1. 哪些字段可以跨边界传输
2. 哪些字段必须删除、替换或摘要化
3. 返回结果如何按照层级和对象类型控制暴露范围

## 核心原则

脱敏不是简单地“删掉最明显的秘密”，而是：

1. 在跨层、跨 Agent、跨服务传输前做字段级最小暴露
2. 让第三层和第三方调用方只看到完成任务所必需的信息
3. 让审计记录记录“做过脱敏”，而不是保存原始敏感内容

## 三个脱敏级别

| 级别 | 含义 | 适用场景 |
| --- | --- | --- |
| `none` | 不脱敏 | 同一可信本地边界内的纯本地处理 |
| `partial` | 局部脱敏 | 第二层返回第三层、第三层返回受控调用方 |
| `full` | 全量脱敏或仅保留摘要 | 第三方只需要结果结论，不需要内容明文 |

## 默认规则

### 第一层与第二层

默认：

1. 可使用 `none`
2. 前提是仍位于同一可信本地边界内

### 第二层与第三层

默认：

1. 优先 `partial`
2. 对高敏对象默认禁止 `none`

### 第三层与第三方 Skill 或远程服务

默认：

1. 优先 `partial` 或 `full`
2. 不得返回 challenge answers、私钥、长期 session 能力或原始秘密材料

## 高敏字段清单

以下字段默认视为高敏：

1. 真实姓名
2. 手机号
3. 邮箱地址
4. 身份证号、护照号或其他证件号
5. 家庭住址
6. 精确地理位置
7. 用户账号名与密码组合
8. 私钥、助记词、seed、signing key bytes
9. challenge answers 原文
10. 未脱敏的私密故事正文

## 最小脱敏检查清单

在把内容发送到远程 LLM 或第三方 Skill 前，至少检查以下项目：

1. 是否包含真实姓名
2. 是否包含手机号
3. 是否包含邮箱地址
4. 是否包含身份证号、护照号或其他证件号
5. 是否包含家庭住址或精确地理位置
6. 是否包含账号名与密码组合
7. 是否包含私钥、seed、助记词或 key bytes
8. 是否包含 challenge answers 原文
9. 是否包含未脱敏的私密故事正文
10. 是否包含可唯一定位用户身份的组织名、学校名、医院名、具体门牌号等组合线索

建议执行顺序：

1. 先做结构化字段检查
2. 再做规则匹配
3. 最后做人工确认或本地规则复核

## 字段处理规则

| 字段类型 | 默认动作 | 示例 |
| --- | --- | --- |
| 标识符 | 哈希、映射或局部显示 | 手机号改为 `138****0000` |
| 长文本 | 摘要化、局部遮挡或替换 | 私密故事只返回摘要 |
| 密钥材料 | 绝不返回 | 私钥、seed、key bytes 一律禁止外发 |
| challenge 材料 | 绝不返回 | 题目答案与比对线索一律禁止外发 |
| 审计字段 | 仅保留最小元数据 | `requestId`、`capability`、`status` |

## 推荐脱敏动作

### 1. 局部遮挡

适用于：

1. 电话
2. 邮箱
3. 用户名

示例：

```json
{
  "phone": "138****0000",
  "email": "st***@example.com"
}
```

### 2. 摘要替换

适用于：

1. 私密故事正文
2. 评估原始数据
3. 长文本材料

示例：

```json
{
  "storySummary": "一段关于童年记忆与身份认同的私密叙事摘要。"
}
```

### 3. 结构保留、值清空

适用于：

1. 第三方仍需要结果结构
2. 但不应看到内容明文

示例：

```json
{
  "result": {
    "storyObjectId": "story-001",
    "title": "[redacted]",
    "content": "[redacted]"
  }
}
```

## 各对象类别的默认脱敏级别

| 对象类别 | 第二层 -> 第三层 | 第三层 -> 第三方 |
| --- | --- | --- |
| `story_public` | `none` 或 `partial` | `partial` |
| `story_private` | `partial` | `full` 或受控 `partial` |
| `auth_login` | `partial` | `full` |
| `auth_signing` | `full` | `full` |
| `review_data` | `partial` | `partial` 或 `full` |

## 响应结构中的脱敏表达

所有跨边界响应建议显式返回：

1. `redactionLevel`
2. `retentionGranted`
3. `result`

示例：

```json
{
  "requestId": "req-001",
  "status": "success",
  "redactionLevel": "partial",
  "retentionGranted": "result_only",
  "result": {
    "storySummary": "已脱敏摘要"
  }
}
```

## 审计要求

审计日志建议记录：

1. 原始对象类别
2. 应用的脱敏级别
3. 是否存在高敏字段
4. 是否拒绝返回原文

但不得记录：

1. 被删除字段的原始内容
2. 私钥材料
3. challenge answers 原文

## 与三包边界的关系

第一层负责内容处理，但不决定远程可见范围。

第二层负责：

1. 根据对象策略决定是否必须脱敏
2. 对跨边界结果做第一道收口

第三层负责：

1. 不放宽第二层已经施加的脱敏结果
2. 按 `retentionGranted` 进一步限制缓存、转发和对外返回

## 当前最小实现建议

建议先稳定支持以下规则：

1. 高敏字段黑名单删除
2. 私密故事摘要化
3. 登录或签名结果仅返回状态和必要结果，不返回底层秘密
4. 调用远程 LLM 前执行最小脱敏检查清单

## 结论

StoryLock 的脱敏规范本质上是三层 Skill 边界的输出约束。

它保证：

1. 本地可以处理完整数据
2. 远程只能获得最小结果
3. 第三方 Skill 不会因为集成便利而反向扩大数据暴露面
