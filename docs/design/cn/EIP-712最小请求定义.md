# StoryLock EIP-712 最小请求定义

## 目的

本文档用于定义 StoryLock 当前阶段最小可实现的 EIP-712 请求结构。

适用范围：

1. 第三层远程 Skill / 代理授权 Skill
2. 本地签名委托请求
3. 第三方 Skill 发起的结构化签名请求

## 当前阶段目标

当前阶段不追求覆盖全部链上执行场景，只解决：

1. 请求可结构化表达
2. 请求字段清晰可审计
3. 本地签名网关不再接受模糊的“盲签名式”请求

## 最小使用场景

当前最优先支持：

1. `request_signature`
2. `delegated_action_sign`

也就是：

1. 对远程提交的结构化 payload 进行签名
2. 对第三方 Skill 委托过来的结构化动作进行签名

## EIP-712 结构建议

### Domain

建议当前阶段使用如下结构：

```javascript
const domain = {
  name: "StoryLock",
  version: "1-placeholder",
  chainId: 1,
  verifyingContract: "0x0000000000000000000000000000000000000000"
};
```

## 当前阶段说明

⚠️ 当前阶段的 `domain.version = "1-placeholder"` 且 `verifyingContract` 为占位地址。

这意味着：

1. 当前定义只能作为结构化签名请求格式
2. 不应被视为可直接用于生产链上验证的正式 domain
3. 进入生产前必须替换为真实链环境参数

1. `chainId`
   - 当前阶段建议由运行时配置注入
   - 默认值可为目标链环境，但不建议在代码中永久写死主网值
2. `verifyingContract`
   - 当前阶段若尚未绑定具体合约，可先使用占位地址
   - 后续若接入智能账户或链上验证，再替换为真实地址
3. `version`
   - 当前阶段建议使用 `1-placeholder`
   - 后续若绑定正式合约，可升级为 `2` 或更明确的正式版本名

## Types

建议定义一个最小请求类型：

```javascript
const types = {
  StoryLockSignatureRequest: [
    { name: "action", type: "string" },
    { name: "resource", type: "string" },
    { name: "scope", type: "string" },
    { name: "expiry", type: "uint256" },
    { name: "nonce", type: "uint256" },
    { name: "requestedBy", type: "string" },
    { name: "delegationContext", type: "string" }
  ]
};
```

## Value

建议最小值对象如下：

```javascript
const value = {
  action: "request_signature",
  resource: "storylock:wallet-key",
  scope: "signature_basic",
  expiry: 1760000000,
  nonce: 1001,
  requestedBy: "remote-agent",
  delegationContext: "user:12345/skill:third-party-demo"
};
```

## 字段解释

| 字段 | 含义 | 当前阶段是否必填 |
| --- | --- | --- |
| `action` | 请求动作，如 `request_signature` | 是 |
| `resource` | 目标资源标识 | 是 |
| `scope` | 请求作用域 | 是 |
| `expiry` | 请求过期时间 | 是 |
| `nonce` | 防重放随机数或单调值 | 是 |
| `requestedBy` | 请求发起方标识 | 是 |
| `delegationContext` | 委托上下文，记录来源链路 | 是 |

## 与 StoryLock 网关的关系

远程网关不直接发送原始字符串消息，而应发送：

1. 结构化请求对象
2. 对应的 EIP-712 domain
3. 对应的 types
4. 对应的 value

也就是说，`requestSignature` 的 payload 后续可扩展为：

```json
{
  "identityId": "identity-001",
  "keyId": "wallet-key",
  "algorithm": "ed25519",
  "payload": "0x1234...",
  "resourceId": "eth-main",
  "primaryRole": "private_key",
  "eip712": {
    "domain": {},
    "types": {},
    "value": {}
  }
}
```

## 当前阶段边界

这份最小定义当前只解决：

1. 请求结构化
2. 可审计
3. 可附带 delegation context

当前阶段先不强行解决：

1. 多链统一 domain 策略
2. 合约账户完整验证流程
3. EIP-1271 的实际链上返回格式

这些属于下一阶段扩展。

## 与技术评审意见的对应

这份文档主要回应了技术评审中的以下问题：

1. `domainSeparator` 的具体构造参数需要明确
2. `chainId` 不能只停留在概念层
3. `verifyingContract` 需要有当前阶段的占位或注入策略

## Domain 升级路径建议

为了避免后续接入真实链上验证时破坏语义，建议：

1. 当前阶段把占位版本显式标识为 `placeholder`
2. 正式链上验证启用后，升级 `version`
3. 变更 `verifyingContract` 时同步记录版本切换点

这样可以明确区分：

1. 当前阶段的“结构化签名请求”
2. 下一阶段的“可链上验证签名请求”

## 生产环境检查清单

进入正式链上验证或生产部署前，至少检查以下项目：

1. `version` 不再使用 `placeholder`
2. `verifyingContract` 不再使用零地址
3. `chainId` 已切换为目标正式网络
4. `nonce` 生成与持久化策略已验证
5. `expiry` 窗口符合生产策略

## 当前阶段最小实现建议

建议最先实现：

1. `domain`
2. `types`
3. `value`
4. `nonce`
5. `expiry`

然后通过本地签名委托链路完成一次最小 demo。

## 结论

EIP-712 在 StoryLock 当前阶段最合适的定位不是“完整链上账户抽象标准”，而是：

1. 第三方 Skill 与远程 Agent 的结构化签名请求格式
2. 本地签名委托的最小可审计输入标准
