# StoryLock 挑战答案存储策略

| 项目 | 内容 |
| --- | --- |
| 文档版本 | v1.0 |
| 日期 | 2026-06-16 |
| 状态 | 当前阶段采用 |
| 适用层级 | 第二层 `storylock-local-story-access-skill` |

## 目的

本文档用于明确第二层本地访问 Skill 如何处理 challenge answers。

要解决的不是“答案能不能存”，而是：

1. 哪些数据可以长期保存
2. 哪些数据只能短时存在内存
3. 如何避免答案本身变成新的高敏感泄露点

## 核心结论

当前阶段采用以下原则：

1. 原始 challenge answers 不做长期持久化
2. 原始答案只允许存在于本地短时内存
3. 持久化层只保存校验所需的不可逆材料或派生材料
4. 远程 Agent、第三方 Skill、第三层远程网关不得接触答案原文

## 数据分类

| 数据项 | 是否敏感 | 是否允许持久化 | 说明 |
| --- | --- | --- | --- |
| `raw_answers` | 高 | 否 | 用户本次提交的原始答案，只能短时存在内存 |
| `answer_digest` | 中到高 | 是 | 对答案做规范化后计算的摘要，用于比对或审计 |
| `challenge_manifest` | 中 | 是 | 本次 challenge 的题目、阈值、TTL、scope 等 |
| `verification_result` | 中 | 是 | 是否通过、通过时间、失败次数 |
| `session_binding` | 中 | 是 | 由 challenge 派生出的 session 元数据，不含答案原文 |

## 当前阶段存储原则

### 1. 原始答案只在内存存在

本地 Agent / Host 接收答案后：

1. 进入校验流程前放入短时内存
2. 校验完成后立即清理
3. 不写入日志
4. 不写入远程缓存
5. 不写入通用数据库明文字段

⚠️ 当前阶段特别注意：

1. JavaScript 的 `String` 是不可变对象
2. 即使把字符串变量设为 `null`，也不代表底层内存立即清除
3. 对极高安全场景，敏感答案优先使用 `Buffer` 而不是 `String`

Node.js 实现建议：

```js
function clearSensitiveBuffer(buf) {
  if (Buffer.isBuffer(buf)) {
    buf.fill(0);
  }
}
```

更严格的覆盖示例：

```js
import { randomFillSync } from "node:crypto";

function wipeBuffer(buf) {
  if (!Buffer.isBuffer(buf)) return;
  randomFillSync(buf);
  buf.fill(0);
}
```

如果答案以字符串数组存在，建议：

1. 尽快转换为短生命周期结构
2. 校验完成后把数组元素覆写为空值
3. 再把引用设为 `null`

示例：

```js
function clearRawAnswers(rawAnswers) {
  if (!Array.isArray(rawAnswers)) return null;
  for (let i = 0; i < rawAnswers.length; i += 1) {
    rawAnswers[i] = "";
  }
  return null;
}
```

推荐实际顺序：

1. 若可能，先把用户输入转为 `Buffer`
2. 校验完成后优先清零 `Buffer`
3. 再清空字符串引用
4. 最后移除对象引用并等待 GC

### 2. 持久化只保存派生结果

建议持久化的数据：

1. `challengeId`
2. `identityId`
3. `scope`
4. `requiredThreshold`
5. `submittedAt`
6. `verificationPassed`
7. `failureCount`
8. `answerDigestSet`

其中 `answerDigestSet` 应是：

1. 规范化后的答案摘要集合
2. 不可逆
3. 不足以恢复原始答案

摘要建议：

1. 使用带身份隔离盐值的摘要
2. 例如 `salt = HMAC(masterSalt, identityId)`
3. 再对规范化答案计算摘要，降低彩虹表攻击风险

### 3. 审计日志只记录过程，不记录答案

审计日志中可记录：

1. 是否创建 challenge
2. 是否提交答案
3. 是否验证通过
4. 使用了哪个 scope
5. 最终签发了什么 session

审计日志中不得记录：

1. 原始答案文本
2. 可逆加密后的答案副本
3. 可直接用于重放的 challenge material

## 推荐数据结构

### Challenge Manifest

```json
{
  "challengeId": "chl-001",
  "identityId": "identity-001",
  "scope": "story_read_basic",
  "requiredThreshold": 6,
  "questionIds": ["q1", "q2", "q3"],
  "createdAt": 1760000000,
  "expiresAt": 1760000300
}
```

### Verification Record

```json
{
  "challengeId": "chl-001",
  "submittedAt": 1760000100,
  "verificationPassed": true,
  "matchedCount": 6,
  "failureCount": 0,
  "answerDigestSet": [
    "sha256:...",
    "sha256:..."
  ],
  "normalizationVersion": "v1"
}
```

## 答案规范化建议

为了减少误判和摘要分裂，建议在本地校验前做统一规范化：

1. 去除首尾空白
2. 统一全角半角
3. 统一大小写策略
4. 明确是否保留标点
5. 明确是否允许同义答案映射

注意：

1. 规范化过程必须在本地完成
2. 规范化规则应固定版本
3. 规则升级时要保留兼容策略

建议显式记录：

1. `normalizationVersion`
2. `digestAlgorithm`
3. `saltStrategyVersion`

## 规范化版本兼容策略

当 `normalizationVersion` 升级时，当前阶段建议：

1. 在迁移窗口内支持双版本校验
2. 新写入统一使用新版本
3. 校验通过后可异步把旧记录迁移到新版本摘要

这样可以避免：

1. 升级后旧 challenge 立即全部失效
2. 用户在无感知情况下被迫重建全部题集

## 与题集存储的关系

题集答案本体不应与 challenge 提交答案混存。

建议拆成两类存储：

1. **题集主档**
   - 问题、答案摘要、版本号、轮换状态
2. **挑战实例档**
   - 本次 challenge 选择了哪些题、阈值是多少、是否通过

这样做的原因是：

1. 题集主档是长期策略对象
2. 挑战实例档是短生命周期运行对象
3. 二者分开更利于最小权限和轮换管理

题集主档建议：

1. 视为中敏感对象
2. 默认做静态加密存储
3. 至少避免明文落盘

## 生命周期建议

| 数据项 | 生命周期 | 默认处理 |
| --- | --- | --- |
| `raw_answers` | 秒级 | 校验完成立即清理 |
| `challenge_manifest` | 分钟到小时级 | 到期后可归档精简元数据 |
| `verification_record` | 天到月级 | 仅保留审计所需最小字段 |
| `session_binding` | 会话级 | 过期即失效，必要时保留摘要审计 |

## 与三包边界的关系

第一包不得接触 challenge answers。

第二包负责：

1. 创建 challenge
2. 接收答案
3. 本地校验
4. 派生 session

第三包只能看到：

1. 是否需要 challenge
2. 是否验证通过
3. 是否获得 session
4. 最小审计元数据

## 当前阶段最小实现建议

当前阶段可先采用：

1. 原始答案仅内存保存
2. 本地 JSON / SQLite 保存 challenge manifest 与 verification record
3. 只记录摘要，不记录原文
4. challenge 过期即清理运行态数据
5. 校验完成后显式执行内存清理逻辑

## 结论

StoryLock 的 challenge answers 不应被视为“普通业务输入”，而应被视为高敏感授权材料。

因此当前阶段最合适的策略是：

1. 原文不持久化
2. 派生值最小持久化
3. 审计只记过程，不记秘密
