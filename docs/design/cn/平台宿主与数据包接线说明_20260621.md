# 平台宿主与数据包接线说明

| 项目 | 内容 |
| --- | --- |
| 文档版本 | v0.1 |
| 日期 | 2026-06-21 |
| 适用范围 | Windows Host、Android Host、Linux Host、StoryLock Core 数据包 |
| 对齐来源 | `story-lock/doc/architecture`、`story-lock/doc/design` |

## 1. 接线原则

StoryLock Core 数据包属于本地核心边界。平台 Host 可以启动底层 Core 应用，也可以读取脱敏权限摘要，但不能直接编辑故事原文、题库答案、密码、私钥、签名材料或加密配置。

当前平台接线统一遵循三条规则：

1. Host 只展示状态、权限摘要和执行结果。
2. 底层故事配置只能在 StoryLock Core 子应用或本地核心模块中修改。
3. 远程网关只保留 `requestSignature` 和 `requestPasswordFill`，不开放远程故事读写。

## 2. 数据包结构

第一版 StoryLock Core 数据包目录如下：

```text
storylock-package/
  package-manifest.json
  resource-catalog.json
  author-draft.json
  templates/
    login-sites.json
    signing-actions.json
    agent-tasks.json
```

其中 `author-draft.json` 属于作者态本地文件，允许 StoryLock Core 保存编辑期内容；Host 主窗口和远程网关不得读取其中的故事原文、答案或编辑备注。

## 3. Windows Host

Windows 当前使用 Rust / Slint 原生窗口：

1. 主窗口显示 Status、Local Core、Data、StoryLock Core、Diagnostics 等菜单。
2. `StoryLock Core` 按钮启动独立底层窗口，不在主 Host 页面直接编辑配置。
3. Rust 侧已具备本地 StoryLock package 初始化、资源目录读取、导出预览和权限摘要展示。
4. `cargo test --manifest-path src\host\windows-host\Cargo.toml` 覆盖授权通道、审计和权限摘要相关逻辑。
5. `npm run test:windows-package` 用 JS 共享实现与 Windows package 产物做契约检查。

Windows 当前仍需补齐桌面人工验收记录，重点确认窗口、托盘、菜单、StoryLock Core 子窗口和本地数据目录的实际表现。

## 4. Android Host

Android 当前使用 Kotlin 原生宿主：

1. 本地 HTTP Host 暴露 `GET /health`、`GET /permission-summary`、`POST /execute`。
2. `AndroidStoryLockPackageRepository` 从 asset 中读取 `storylock-resource-catalog.json`。
3. `/permission-summary` 只返回脱敏权限摘要，不返回故事原文、答案、密码、私钥或 `signingKeyBytes`。
4. Android Keystore、BiometricPrompt、challenge 输入和 relay polling 属于宿主执行路径。
5. `scripts/test/android-readiness.mjs` 检查 Android 与共享 schema、permission summary 的对齐关系。

Android Host 不是底层 StoryLock Core 配置编辑器。后续如果补 Android 端配置 UI，也应保持独立 Core 边界，而不是让 relay 或 Host 主页面直接修改配置。

## 5. Linux Host

Linux 当前使用 Node 本地宿主原型：

1. 通过 `STORYLOCK_LINUX_STORYLOCK_PACKAGE_DIR` 指向本地 StoryLock package 目录。
2. `GET /permission-summary` 通过共享 StoryLock package 模块返回脱敏权限摘要。
3. Linux Host 不直接读取或编辑 `author-draft.json` 的私密内容。
4. `scripts/linux/check_linux_host_loop.mjs` 覆盖 `/permission-summary` 和本地执行路径的最小回归。

Linux 后续重点是桌面环境、Secret Service 真环境、安装包和真实运行验收。

## 6. 权限摘要边界

Host 可读取的字段应保持在权限摘要范围内：

1. `resourceId`
2. `resourceKind`
3. `providerId`
4. `displayName`
5. `role`
6. `objectId`
7. `objectKind`
8. `action`
9. `challengePolicy`
10. `requiredGridCount`

Host 不得读取：

1. 故事原文。
2. 题库答案。
3. 密码原文。
4. 私钥、seed、签名字节。
5. 可重放 challenge 材料。
6. 作者态编辑备注。

## 7. 验收命令

```powershell
npm run test:storylock-package
npm run test:windows-package
npm run test:android-readiness
npm run test:linux-host
node scripts\test\docs-consistency.mjs
node scripts\verify\path-consistency.mjs
```

真机或桌面验收仍需人工填写 `docs/test` 下的验收记录，自动化测试不能替代托盘图标、原生窗口、Android 真机权限和 Linux 桌面集成确认。
