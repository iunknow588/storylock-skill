# StoryLock 与 Pharos Skill Engine 的技术映射说明

## 1. 说明目的

本文用于回答比赛中的关键问题：若按照 Pharos Skill Engine 的组织思路，StoryLock 当前仓库中的代码、文档与演示能力分别对应到哪个层次。

目标不是声称 StoryLock 已完全等同于某一外部规范，而是把现有成果映射为更易理解、展示与提交的 Skill 结构。

## 2. StoryLock 当前的 Skill 形态

当前已经同时具备三类与 Skill 相关的结构：

| 类型 | 路径 | 说明 |
|------|------|------|
| 代码层 Skill | `story-lock/src/adapters/skills/` | 实现可调用的类与方法 |
| 文档层 Skill | `skill/docs/ref/storylock-skill-guide/` | 提供 Skill 说明与使用指引 |
| 迁移层 Skill | `skill/src/storylock-skill-engine/` | 采用 Pharos 风格结构重新组织的可复用版本 |

这说明 StoryLock 不是为了比赛临时包装出 Skill 概念，而是已经形成真实的高层能力接口，并进一步整理出了可复用的 Skill 包结构。

## 3. 当前迁移版的结构对应

`skill/src/storylock-skill-engine/` 可按下列方式映射到 Pharos Skill Engine 常见结构：

| Pharos 结构 | StoryLock 迁移版对应 |
|------|------|
| `SKILL.md` | `skill/src/storylock-skill-engine/SKILL.md` |
| `references/` | `skill/src/storylock-skill-engine/references/` |
| `assets/` | `skill/src/storylock-skill-engine/assets/migrated/` |

这说明 StoryLock 已具备“按 Skill Engine 方式打包”的基础，只是当前仍处在“迁移复用优先、完全独立化其次”的阶段。

## 4. 可映射的组成部分

### 4.1 Skill 元数据层

可映射内容：

1. `skill/docs/ref/storylock-skill-guide/SKILL.md`
2. `skill/src/storylock-skill-engine/SKILL.md`

作用：

1. 说明 Skill 名称
2. 说明 Skill 触发条件
3. 说明 Skill 职责与边界

### 4.2 Skill 逻辑层

可映射内容：

1. `story-lock/src/adapters/skills/story-assist.js`
2. `story-lock/src/adapters/skills/strength-review.js`
3. `story-lock/src/adapters/skills/authorization-skills.js`
4. `story-lock/src/adapters/skills/recommended.js`
5. `skill/src/storylock-skill-engine/assets/migrated/skills/`

作用：

1. 提供真实可调用的类与方法
2. 定义结构化输入输出
3. 把宿主能力包装为更清晰的 Skill 接口

### 4.3 Skill 依赖层

StoryLock 当前 Skill 层依赖的最小宿主接口如下：

1. `createChallenge(identityId, scope)`
2. `submitChallengeAnswers(identityId, challengeId, answers)`
3. `readSecretObject(identityId, sessionId, secretObjectId)`

挑战签名类 Skill 额外依赖：

1. `signer()` 或 `sign()`

这说明：

1. Skill 层并不独立替代 Core
2. Skill 层本质上是建立在 Host 与 Core 之上的高层编排层

## 5. StoryLock 的 Skill 类型归属

从能力结构看，StoryLock 更接近“带有本地授权前置条件的工具型 Skill”，而不是单纯的内容生成 Skill。

主要特征：

1. 输入结构化
2. 输出结构化
3. 宿主依赖明确
4. 演示闭环明确

这是比赛中最值得强调的技术特点之一：它不是抽象概念，而是可被宿主或 Agent 调用的能力接口。

## 6. 当前最适合对外展示的两个 Skill

如果比赛展示只选两个最容易让评委理解的 Skill，建议优先展示：

| Skill | 展示重点 |
|-------|---------|
| `LocalPasswordFillSkill` | 本地授权后输出登录字段 |
| `ChallengeSigningAuthorizationSkill` | 本地授权后输出签名结果 |

理由：

1. 输入输出最直观
2. Skill 边界最清晰
3. 最容易对接 Agent 场景与工具场景

## 7. 与正式设计边界的关系

必须明确：

1. Skill 层建立在 StoryLock Core 之上
2. Skill 层不重新定义 Core
3. Skill 层是更易于调用、展示与集成的接口层
4. Demo 层仅用于验证与展示，不等同于正式安全边界

## 8. 建议比赛答辩时的技术表述

建议按以下顺序解释系统结构：

1. Core 负责本地授权判断
2. Host 负责运行时会话与对象读取
3. Skill 负责把授权访问包装为可复用能力
4. Demo 负责证明这些能力已可被展示与调用

这种说法的优点：

1. 技术层次清晰
2. 与现有代码结构一致
3. 避免把 Demo 或 Skill 层误表述为底层安全机制本体

## 9. 当前最适合提交的 Skill 版本

如果比赛材料需要给出一份“更像标准 Skill 包”的目录，当前最适合引用的是：

- `skill/src/storylock-skill-engine/`

理由：

1. 它已经按 `SKILL.md + references + assets` 重组
2. 它保留了 StoryLock 原有 skill-layer 的功能口径
3. 它比直接指向整个 `story-lock` 项目更适合评委快速理解 Skill 结构
