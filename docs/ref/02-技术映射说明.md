# StoryLock 与 Skill Engine 结构的技术映射说明

## 1. 说明目的

本文说明 StoryLock 当前 `skill/src` 代码如何映射成可提交、可展示、可调用的 Skill 项目结构。这里不声称 StoryLock 已完整等同于某个外部运行时，而是说明当前实现具备清晰的 Skill 包组织方式。

## 2. 当前代码结构

| 类型 | 路径 | 说明 |
| --- | --- | --- |
| 第一层 Skill 包 | `skill/src/storylock-local-story-processing-skill/` | 故事草稿、润色、强度评估 |
| 第二层 Skill 包 | `skill/src/storylock-local-story-access-skill/` | 对象强度、九宫格、本地授权、SQLite 审计 |
| 第三层 Skill 包 | `skill/src/storylock-remote-gateway-skill/` | 远程签名、密码填充、脱敏 |
| 兼容演示包 | `skill/src/storylock-skill-engine/` | 本地密码填充和签名授权示例 |
| 共享基础设施 | `skill/src/shared/` | SQLite、SecretStore、加密辅助 |

## 3. Skill 包结构映射

每个主包都具备以下基本结构：

| 结构 | StoryLock 对应 |
| --- | --- |
| `SKILL.md` | 包能力说明 |
| `index.js` | 可调用能力入口 |
| `assets/schemas/` | 输入输出 Schema |
| `references/` | 调用说明和边界说明 |
| `scripts/selftest.mjs` | 最小自测 |

第三层还新增：

| 脚本 | 作用 |
| --- | --- |
| `scripts/e2e-selftest.mjs` | 三层端到端签名授权演示 |

第二层还新增：

| 脚本 | 作用 |
| --- | --- |
| `scripts/cleanup-expired.mjs` | SQLite 过期状态清理命令 |
| `scripts/check-secret-store.mjs` | 平台 SecretStore 可用性检查 |

## 4. 当前最核心的可展示 Skill

建议比赛展示优先聚焦：

| 能力 | 展示重点 |
| --- | --- |
| `requestSignature` | 远程请求包装为 EIP-712 风格结构，经过本地九宫格授权后返回脱敏签名结果 |
| `requestPasswordFill` | 远程请求 Web2 密码填充，默认不返回明文密码 |
| `StrengthReviewSkill` | 评估题集是否具备基本验证强度 |

兼容包中的 `LocalPasswordFillSkill` 和 `SignatureAuthorizationSkill` 可以作为本地执行示例说明，但主线接口应以 `requestSignature` 和 `requestPasswordFill` 为准。

## 5. 本地安全边界

StoryLock 的本地安全边界主要体现在第二层：

1. `challenge_state` 记录九宫格验证状态。
2. `session_store` 记录短时授权。
3. `request_store` 和 `nonce_store` 防重放。
4. `failure_window` 记录失败次数和锁定窗口。
5. `answer_digest_set` 保存答案摘要，不保存明文答案。
6. `audit_log` 记录授权和安全事件。

第三层不能直接读取故事内容、密码或私钥，只能发起包装请求并接收脱敏结果。

## 6. 建议答辩技术表述

建议按以下顺序介绍：

1. StoryLock 不是单纯页面，也不是单一脚本，而是三层 Skill 包。
2. 第一层处理故事内容和题集强度。
3. 第二层负责真正的本地授权控制。
4. 第三层负责远程 Agent 能调用的接口包装。
5. 当前已有 selftest 和三层端到端演示证明链路可运行。

## 7. 当前不应过度表述的内容

以下内容只作为未来探索，不作为当前已实现能力：

1. 完整链上验签。
2. 多链、多钱包、多签名主体管理。
3. 多平台内容发布账号管理。
4. 把远程网关作为故事读取或写回入口。
