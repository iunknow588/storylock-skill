# StoryLock 评审讲解与演示说明

日期：2026-06-20

本文档用于统一评审讲解顺序、演示命令和真实口径，避免把设计目标误讲成已完成能力。

## 1. 建议开场口径

推荐一句话说明：

> StoryLock 当前展示的是一套“远程请求进入云端网关，再由用户本地设备完成 challenge、授权和敏感执行”的三层闭环原型。

需要同时强调：

1. 第一层负责故事与题集质量处理。
2. 第二层负责 challenge、授权、撤销和审计。
3. 第三层负责远程请求包装、EIP-712 表达和脱敏响应。

## 2. 演示前检查

在 `skill/` 根目录执行：

```powershell
npm run test
```

如果只想快速确认主线可运行，至少执行：

```powershell
node scripts/test/run-selftests.mjs
```

建议评审前额外执行：

```powershell
npm run selftest:web-api-android
```

## 3. 推荐演示顺序

### 3.1 总览

先展示：

1. `docs/ref/README.md`
2. `docs/design/cn/三包接口契约.md`
3. `docs/ref/03-演示与调用说明.md`

说明主线目录：

1. `src/skills/local-story-processing`
2. `src/skills/local-story-access`
3. `src/skills/remote-gateway`

### 3.2 三层自测闭环

执行：

```powershell
Push-Location src/skills/remote-gateway
npm run selftest:e2e
Pop-Location
```

讲解重点：

1. 远程签名请求进入第三层。
2. 第二层判定对象强度并创建 challenge。
3. 第二层完成本地授权和短期 session。
4. 执行器完成本地签名。
5. 第三层统一脱敏返回结果。

### 3.3 Web API + Android 宿主 mock 闭环

执行：

```powershell
Push-Location src/skills/remote-gateway
npm run selftest:web-api-android
Pop-Location
```

讲解重点：

1. 演示的是云端入口和本地宿主接口模型联通。
2. 当前自测入口主要验证的是 Android host mock 闭环，不等于正式真机发布态。
3. `/download/android-host`、注册、relay 和执行链路都已经连通。
4. 仓库内另有 `src/host/android-host/` 真机宿主原型工程，但当前评审不应把它说成正式交付完成。

### 3.4 题集模板与导入

执行：

```powershell
Push-Location src/skills/local-story-access
npm run generate:question-set-template
npm run validate:question-set -- --input assets/question-set-master.sample.json --require-min-active 24
Pop-Location
```

讲解重点：

1. 题集不再只是演示占位数据。
2. 当前仓库支持模板生成、结构校验和导入流程。
3. 24 题 active 集合是当前主线目标。

## 4. 建议讲解顺序

推荐按下面顺序解释：

1. 第一层决定“题集质量是否足够支撑 challenge”。
2. 第二层决定“用户是否在本地通过验证并拿到短期授权”。
3. 第三层决定“远程怎么请求、怎么表达、怎么脱敏返回”。
4. Android 宿主和 Windows 宿主属于平台实现方向，不改变三层职责边界。

## 5. 必须诚实说明的事项

评审时应明确说明：

1. Android 宿主真机原型工程已经存在，但不是当前主线交付件。
2. Android Keystore、BiometricPrompt 和 challenge 原型代码已经存在，但正式签名与正式发布链路仍属后续增强。
3. 当前仓库主打的是三层架构正确性、边界清晰性和可验证闭环。
4. `src/engine` 是兼容与演示包，不是第四层安全边界。

## 6. 避免的错误讲法

不要这样讲：

1. “Android 宿主已经是正式上架 App。”
2. “云端可以直接读取 StoryLock 本地核心的数据。”
3. “第三层会直接调用第二层内部方法给外部系统使用。”
4. “当前仓库已经完成多平台正式发布。”
5. “题集答案会回传到远程服务。”

## 7. 评审问答准备

### 7.1 如果被问“现在最完整的是哪一段？”

建议回答：

当前最完整的是三层 Skill 主线和可执行自测闭环，其次是 Web API 风格网关与本地宿主接口模型。

### 7.2 如果被问“Android 完成到什么程度？”

建议回答：

当前已完成 Android 宿主接口模型、注册与 relay 闭环验证；仓库里也已经有真机宿主原型工程、Keystore 与生物识别相关代码，但正式签名、完整高强度 challenge 流程和正式发布闭环尚未作为主线交付完成。

### 7.3 如果被问“安全边界在哪里？”

建议回答：

敏感 challenge、授权和本地执行边界在用户设备侧，第三层只处理标准请求包装和脱敏后的结果返回。
