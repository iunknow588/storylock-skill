# StoryLock 评审讲解与演示说明

日期：2026-06-18

## 1. 3 分钟演示路径

目标：展示远程请求如何进入本地授权链路，并在返回时保持长期秘密不外泄。

### 1.1 一键验证

在 `skill/` 目录运行：

```powershell
npm run test
```

说明重点：

- 四个包 selftest 均通过。
- 三层 e2e selftest 会串联 `requestSignature`、对象强度策略、九宫格验证、本地授权、本地签名执行器和脱敏返回。
- `selftest:vercel-android` 会串联“第三层 Vercel 风格入口 + Android 本地宿主 mock”。
- `check-agent-capabilities` 验证远程主接口只有 `requestSignature` 和 `requestPasswordFill`。

### 1.2 展示云端第三层与本地宿主拆分

```powershell
Push-Location src/storylock-remote-gateway-skill
npm run selftest:vercel-android
Pop-Location
```

说明重点：

- 第三层对外入口可以运行在 Vercel 风格 HTTP 入口。
- 第一层题集强度评估与第二层本地授权仍保留在 Android 本地宿主。
- 当前仓库用 Android 宿主 mock 验证链路，并已建立 `android-host/` 真实宿主代码骨架。
- 第三层当前还可以向外提供 Android 本地应用下载入口，以及第二层连接配置元数据。

### 1.3 展示题集导入 dry-run

```powershell
Push-Location src/storylock-local-story-access-skill
node scripts/import-question-set.mjs --input assets/question-set-master.sample.json --dry-run
Pop-Location
```

说明重点：

- 题集主档包含 `questionId`、`versionTag`、`promptRef`、`questionSetVersion`、`normalizationVersion` 和 `status`。
- active 题集至少需要 9 个 active questions，满足 high 强度九宫格验证。
- 持久化生产导入必须使用平台 SecretStore，不能使用临时内存密钥。

### 1.4 展示 SecretStore 检查

```powershell
Push-Location src/storylock-local-story-access-skill
npm run check:secret-store
Pop-Location
```

说明重点：

- 输出包含 platform、adapter、status、productionReady、reason 和 recommendation。
- 如果 Windows 缺少 CredentialManager 模块，脚本会给出安装建议；这说明环境仍需补齐，不代表代码路径缺失。

### 1.5 展示 EIP-712 production 配置检查

```powershell
Push-Location src/storylock-remote-gateway-skill
npm run check:eip712-config -- --environment production
Pop-Location
```

说明重点：

- production 环境必须显式配置 version、chainId 和 verifyingContract。
- 缺配置时失败是预期安全行为，说明 demo/test domain 不会被误用作生产链上验证证据。

## 2. 8 分钟技术讲解路径

### 2.1 三层边界

StoryLock 在 `skill/src` 中定义了三层 Skill。  
第一层负责故事处理与题集强度评估，第二层负责本地授权控制与审计，第三层负责远程请求包装、委托执行与脱敏返回。  
这样可以把验证材料准备、本地安全判断和远程调用入口分别落在清晰的职责边界里。

- 第一层：故事草稿、润色、题集强度评估。
- 第二层：对象强度策略、九宫格验证、本地授权、撤销、审计。
- 第三层：远程请求包装、EIP-712 风格结构化请求、可选本地执行器、脱敏返回。

强调：

- 第二层不读写故事正文。
- 第三层不直接暴露 `createChallenge`、`submitChallengeAnswers` 或长期 session material。

### 2.2 本地授权链路

1. 远程侧发起 `requestSignature`。
2. 第三层包装请求并校验 envelope。
3. 第二层根据对象类型决定强度：signature 默认 high，password_fill 默认 medium。
4. 第二层从 active question set 选择 cell，生成 verification。
5. 本地用户提交答案，系统按 cellId + questionId 校验摘要。
6. 授权通过后签发短时 session。
7. 本地 executor 执行签名或密码填充。
8. 第三层递归脱敏后返回结果。

### 2.3 生产边界

- 持久化 SQLite 主机必须使用平台 SecretStore 或注入等价安全存储。
- `MemorySecretStore` 只用于开发或测试。
- 生产题集必须是 active question set；legacy fallback 只能显式用于开发或演示。
- EIP-712 production domain 必须提供非 placeholder version、非 0 chainId、非零 verifyingContract。
- EIP-1271、ERC-4337、多链多钱包仍是后续扩展，不作为当前已实现能力。

### 2.4 当前代码证据

- `src/storylock-local-story-access-skill/scripts/selftest.mjs`：覆盖题集版本轮换、pending 不可选中、撤销、失败锁定、生产拒绝 legacy fallback。
- `src/storylock-local-story-access-skill/scripts/import-question-set.mjs`：题集主档导入与 dry-run 校验。
- `src/storylock-local-story-access-skill/references/question-set-operations.md`：题集发布、弃用和回滚流程。
- `src/storylock-remote-gateway-skill/scripts/selftest.mjs`：覆盖 production EIP-712 domain、env 配置、递归脱敏。
- `src/storylock-remote-gateway-skill/scripts/check-eip712-config.mjs`：EIP-712 production 配置检查。
- `src/storylock-remote-gateway-skill/scripts/e2e-selftest.mjs`：三层端到端签名链路。
- `api/storylock-gateway.mjs` 与 `src/storylock-remote-gateway-skill/vercel-handler.js`：第三层 Vercel 风格 HTTP 入口。

## 3. 常见问题

### 3.1 为什么第二层不读写故事？

第二层负责授权判断，不负责故事正文生命周期。这样可以把验证依据管理与受保护内容读写保持分离。

### 3.2 为什么 EIP-712 不是完整链上账户系统？

当前实现的是结构化签名请求表达和本地授权包装，不宣称已经实现链上合约验签、EIP-1271 或 ERC-4337。production domain 校验为后续链上验证预留了正确入口。

### 3.3 为什么 SecretStore 还需要平台检查？

答案摘要和 masterSalt 依赖稳定密钥材料。生产持久化场景不能使用临时内存密钥，否则重启后无法验证历史摘要。因此 `check:secret-store` 是部署前置检查。

### 3.4 为什么兼容演示包仍存在？

`storylock-skill-engine` 用于兼容旧 demo 和迁移资产，不是第四层安全边界。当前安全主线以三个运行包为准。

### 3.5 UI 演示如何定位？

`src/ui` 是辅助展示层，CLI selftest 和 e2e selftest 是验收基线。UI 演示不应展示明文答案、私钥或密码作为远程返回内容。

### 3.6 现在能否说“已经跑在 Android + Vercel 上”？

可以准确地说：当前仓库已经跑通“Vercel 风格第三层入口 + Android 本地宿主 mock”的端到端闭环，并且已经建立 `android-host/` 真实宿主代码骨架。  
但还不能说“已经完成 Android 真机编译、真机联调与生产级设备注册部署”。

### 3.7 下一步最可信的实现路线是什么？

1. 保持第三层继续运行在 Vercel 风格入口。
2. 按 `05-Android宿主实现规范.md` 把第一层和第二层落到真实 Android 宿主。
3. 先补 Android Keystore、本地 challenge UI、生物识别确认，再补设备登记和可达性方案。

### 3.8 当前第三层如何让系统整体流通？

当前第三层已具备三类能力：

1. 对外暴露远程网关入口。
2. 返回第二层连接配置，例如 direct URL、relay URL、deep link。
3. 返回或重定向 Android 本地应用下载地址。

这样外部系统可以先通过第三层获得“去哪里下载本地宿主”和“第二层如何连接”的信息，再进入本地授权链路。
