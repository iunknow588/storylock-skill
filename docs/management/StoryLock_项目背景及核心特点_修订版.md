# 易安与 StoryLock 项目背景及核心特点（修订版）

## 一、项目背景

数字化个人代理（Agent）正在将传统的"用户亲自授权"转向"Agent 自主发起请求"。这提高了自动化效率，也把更多敏感能力暴露给远端服务环境——签名私钥、账户密码、文件密钥等核心资产面临被中心化服务托管的风险。

对于强调用户自主管理资产、身份和权限的智能化个体而言，这种信任集中化趋势是一种技术倒退。市场亟需一种机制，把敏感操作从云端拉回用户可控的本地设备，同时保留 Agent 协作的便利性。

**更深层的痛点**：

1. **渐进式授权缺失**：现有方案多为"一次性授权"或"完全拒绝"，缺乏从人工确认到自动执行的渐进过渡。用户无法让 Agent 在受控范围内逐步学习、逐步接管，导致要么全程手动、要么完全失控。

2. **剪贴板泄露风险**：用户手工输入密钥时，密码、私钥等敏感材料经过系统剪贴板，极易被第三方插件、恶意软件、输入法、云同步服务窃取或篡改。传统密码管理器的"复制-粘贴"模式本质上是在系统层面裸奔。

易安与 StoryLock 正是基于这些痛点而生：远端 Agent 可以提出请求，私人智能助理解释风险，StoryLock 本地核心完成授权与敏感执行，用户始终在本地设备上掌握最终确认权。

## 二、核心特点

### 2.1 本地优先（Local-First）

签名、密码填充、API 调用、证书签名等敏感操作完全在本地设备执行，远端不接触密钥材料。私钥、密码、故事原文等最高敏感材料始终保留在设备安全区域（Android Keystore / Windows DPAPI / macOS Keychain），云端仅接收脱敏后的结果与审计元数据。

### 2.2 故事化验证（Story-Based Verification）

用户以精心设计的私密故事替代传统固定符号口令。故事由 24 个节点构成（时间、地点、人物、主题、冲突、情节、结论），用户基于个人记忆回答 challenge，而非记忆无意义的字符组合。这种验证方式既具备高熵安全性，又符合人类认知习惯。

### 2.3 三层隔离（Three-Layer Isolation）

| 层级 | 职责 | 运行位置 | 安全边界 |
|------|------|---------|---------|
| 第一层：故事处理 | 故事草稿、润色、题集评估、对象加密 | 本地设备 | 不读取受保护对象；不签发授权 |
| 第二层：本地授权 | 强度判断、九宫格挑战、短时授权、审计 | 本地设备 | 不返回长期秘密；不直接执行远程请求 |
| 第三层：远程网关 | 请求包装、结果脱敏、统一对外入口 | 华为云 | 不持有私钥/密码；不透传敏感字段 |

### 2.4 渐进式授权（Progressive Authorization）

StoryLock 支持从"完全人工确认"到"自动执行"的渐进过渡：

| 阶段 | 模式 | 说明 |
|------|------|------|
| **学习期** | 每次人工确认 | 新 Agent 或新操作类型，每次均需九宫格 challenge 确认 |
| **信任期** | 短时 session 授权 | 验证通过后，颁发 3-30 分钟短时 session，预算内自动执行 |
| **成熟期** | 策略化自动执行 | 用户可配置特定场景免确认（如"同一收款地址、同一金额范围"） |
| **审计期** | 事后追溯 | 所有操作记录审计日志，用户可随时回查、撤销、调整策略 |

**关键设计**：渐进式授权不是"开闸放水"，而是"逐步放权+随时收回"。用户可随时降级信任等级，强制恢复人工确认。

### 2.5 动态口令挑战（Dynamic Grid Challenge）

从 24 题题集中按难度分层随机抽取 9 题（3 易 + 3 中 + 3 难），每次 challenge 题目组合不同。答对阈值按对象强度策略分级：

| 强度等级 | 阈值 | 适用场景 |
|---------|------|---------|
| low | 3/9 | 一般对象读取 |
| medium | 6/9 | 密码填充、API 调用 |
| high | 9/9 | 签名授权、高敏感操作 |
| story_edit | 22/24 | 根密钥重建、灾难恢复 |

### 2.6 故事免轮换（Lifetime Story Without Rotation）

单故事终身制，通过题集渐进式淘汰实现安全更新：
- 旧题逐步退役，新题逐步加入，用户无需更换记忆内容
- 24 题题集保持动态更新，但故事核心不变
- 避免传统密码定期更换带来的记忆负担与"密码123"式弱口令

### 2.7 可审计脱敏（Auditable Redaction）

- **递归脱敏**：三层间传输按 none / partial / full 分级处理，高敏字段不出本地边界
- **审计日志**：SQLite 记录所有操作，支持按 identityId、capability、时间范围查询
- **防重放**：requestId + nonce + expiry 三重机制，防止请求重放攻击
- **失败锁定**：24 小时窗口内 3 次失败锁定 15 分钟，不泄露哪一题错误

### 2.8 剪贴板免疫（Clipboard Immunity）

StoryLock 彻底规避剪贴板泄露风险：

| 传统方式 | StoryLock 方式 |
|---------|---------------|
| 密码复制到剪贴板 → 粘贴到浏览器 | 本地核心直接填充密码，不经过剪贴板 |
| 私钥复制到剪贴板 → 粘贴到钱包 | 本地核心直接调用签名模块，私钥不出安全区域 |
| API Key 复制到剪贴板 → 粘贴到脚本 | 本地核心直接注入环境变量，不经过系统剪贴板 |

**技术实现**：
- 密码填充：通过操作系统辅助功能（Accessibility Service）直接注入目标输入框
- 签名执行：私钥在 Android Keystore / Windows DPAPI 内完成签名，仅返回签名结果
- API 调用：本地核心通过进程间通信（IPC）直接注入目标应用环境变量

**防御范围**：
- 第三方输入法：无法截获密码
- 剪贴板管理工具：无法记录密钥
- 云同步服务：无法同步敏感材料
- 恶意软件：无法读取系统剪贴板中的密钥

---

## 三、一句话定位

StoryLock 是 Agent 时代的个人数字保险箱——远端可以请求，但打开保险箱的钥匙，永远只在用户手中。渐进式授权让 Agent 逐步学会信任，剪贴板免疫让密钥永不裸奔。
