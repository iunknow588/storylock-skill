## 二、组织架构与治理

### 组织形式

| 项目 | 说明 |
|------|------|
| **法律实体** | 一人制公司（One-Person Company, OPC） |
| **股权结构** | 核心成员 100% 持股，无外部稀释，决策权与所有权高度统一 |
| **决策机制** | 核心成员全权决策，技术顾问提供外部评审与建议，无层级审批损耗 |
| **沟通效率** | 需求→设计→实现→测试闭环可在单日内完成，决策链路极简 |
| **激励制度** | 以技术成果与产品里程碑为驱动，通过 Skill 能力复用与开源社区反馈形成长期价值积累 |
| **人员配置** | 当前为最小可行团队（1 人核心 + 技术顾问外部支持），后续随产品成熟按需扩展 |

### 合理性说明

- **一人制适配性**：StoryLock 作为本地优先的安全基础设施，代码即核心资产，无需大规模销售或运营团队；OPC 模式降低了早期管理成本，使资源集中于技术深度。
- **模块化协作**：三层 Skill 架构天然支持"分阶段外包"——第一层故事处理可对接外部 NLP 能力，第二层本地授权可引入密码学审计，第三层网关可委托 DevOps 部署，接口契约保障质量。

## 三、外部资源与合作关系

### 技术顾问

| 顾问领域 | 作用 | 与项目关系 |
|---------|------|-----------|
| 密码学顾问 | 评审密钥层次、AEAD 实现、nonce 管理策略 | 按需咨询，不参与日常开发 |
| 区块链安全顾问 | 评估 EIP-712 / EIP-1271 适配方案、智能账户集成路径 | 阶段评审，提供协议层建议 |
| 合规顾问 | 指导数据本地化存储、隐私计算合规要求 | 产品化阶段介入 |

### 战略合作伙伴

| 合作方类型 | 合作内容 | 与项目关系 |
|-----------|---------|-----------|
| 云平台（Vercel） | 第三层网关部署、域名托管、Serverless 运行 | 基础设施供应商，不接触敏感逻辑 |
| 开源社区（OpenClaw / Pharos） | Skill 生态集成、Agent 能力调用标准对接 | 生态共建方，接口层合作 |
| 移动开发生态（Android / Rust） | 本地宿主构建、Keystore / DPAPI 集成 | 技术栈依赖，宿主实现参考 |
| **视频自动化发布项目** | 验证 StoryLock 在内容创作场景中的授权流程，探索媒体资产本地签名与分发 | 应用场景验证方，测试第三层网关的媒体签名请求处理能力 |
| **Web3 自动化交易项目** | 验证 StoryLock 在区块链场景中的签名授权，探索 DeFi 交易的本地确认与防重放机制 | 应用场景验证方，测试第二层九宫格验证对高风险交易的保护强度 |

### 外部资源使用原则

1. **根基原则**：所有外部合作以"构建 OPC 系统发展的个体隐私保护城墙"为根基，任何扩展不得削弱用户本地控制密钥的核心能力。
2. **深度原则**：优先向零知识证明（ZKP）全栈隐私保护靠拢，逐步将验证逻辑迁移至云端，但密钥材料与敏感执行始终保留本地。
3. **广度原则**：在确保隐私保护基线的前提下，向不同应用领域扩展（金融、医疗、教育、物联网等），通过可插拔的对象强度策略适配各场景需求。
4. **敏感边界隔离**：所有外部合作仅限于非敏感层（第三层网关部署、UI 展示、生态对接），第二层本地授权与第一层故事处理完全自主实现。
5. **可替换性**：每个外部依赖均设计有降级方案（如 Vercel 可替换为自托管 Node.js 服务，OpenClaw 可替换为标准化 HTTP API）。
6. **审计可追溯**：外部引入的代码或建议均记录在 `docs/management/` 决策日志中，确保技术债务来源可查。

## 四、发展路线图与演进策略

### 核心目标

构建 OPC 系统发展的**第一道个体隐私保护的城墙**——以本地优先架构为基石，确保用户敏感操作（签名、密码填充、文件密钥管理）始终保留在用户可控的设备侧，远端 Agent 仅可发起请求，无法触及密钥材料。

### 深度演进：与零知识证明结合

| 阶段 | 目标 | 技术路径 |
|------|------|---------|
| **当前** | 本地三层架构 | 第一层故事处理、第二层九宫格授权、第三层网关脱敏，敏感操作完全本地执行 |
| **近期** | 第二层上云 | 将第二层（对象强度策略、九宫格验证、session 管理）迁移至可信云服务，保留本地签名/密码填充执行器；引入零知识证明（ZKP）实现"验证逻辑上云，密钥材料不出本地" |
| **中期** | 第一层上云 | 故事处理与强度评估迁移至云端，用户通过 ZKP 证明其故事满足强度要求，而无需上传故事原文；云端仅接收证明，不接触原始叙事 |
| **远期** | 全栈隐私计算 | 第一层与第二层均基于 ZKP / 同态加密 / 安全多方计算（MPC）实现，形成"逻辑可验证、数据不可见"的隐私保护城墙 |

**关键约束**：无论演进至哪个阶段，**签名私钥、密码明文、故事原文**三类最高敏感材料始终保留在本地设备，云端仅处理脱敏后的证明与审计元数据。

### 宽度演进：跨领域应用扩展

| 领域 | 应用场景 | 适配方式 |
|------|---------|---------|
| **金融** | 去中心化钱包签名授权、交易确认 | 对接 EIP-712 / EIP-1271，支持多链签名请求 |
| **医疗** | 电子病历本地加密、跨机构授权查阅 | 扩展对象类型至 `medical_record`，增加 HIPAA 合规审计日志 |
| **教育** | 学历凭证本地持有、选择性披露 | 结合可验证凭证（VC）标准，实现零知识选择性披露 |
| **物联网** | 智能家居设备本地密钥管理 | 适配轻量级宿主（嵌入式 Linux / RTOS），精简 challenge 流程 |
| **企业协作** | 团队共享密钥的分级授权 | 扩展多身份支持，实现基于角色的对象强度策略（RBAC + StoryLock） |

**扩展原则**：
1. **核心层不变**：第二层九宫格验证与本地授权逻辑作为通用基座，不随领域变化而重构
2. **适配层可插拔**：各领域通过 `ObjectStrengthPolicySkill` 配置对象类型、强度阈值、会话模型
3. **生态共建**：通过开放第三层网关接口，吸引第三方 Agent 开发者接入，形成隐私保护应用生态

## 五、总结

易安与 StoryLock 作为一人制公司（OPC）项目，通过**模块化 Skill 架构**降低了对大规模团队的依赖，通过**明确的外部资源边界**保障了安全核心自主可控。核心成员的多维度能力覆盖（架构、工程、安全、产品）与外部顾问的阶段性深度介入，形成了"内全栈 + 外专家"的高效协作模式。

发展策略上，**深度优先于宽度**：先以本地优先架构筑牢个体隐私保护的第一道城墙，再逐步引入零知识证明实现"验证上云、密钥留本地"，最后横向扩展至金融、医疗、教育等多领域。每一阶段的演进均以"用户自主控制密钥"为不可逾越的底线，确保 OPC 系统的隐私保护承诺不因规模扩展而稀释。
