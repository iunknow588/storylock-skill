# 易安与 StoryLock 作品简介及应用场景（最终版）

## 一、作品简介

### 1.1 项目背景

数字化个人代理（Agent）正在将传统的"用户亲自授权"转向"Agent 自主发起请求"。这提高了自动化效率，也把更多敏感能力暴露给远端服务环境——签名私钥、账户密码、API 凭据、开发者证书等核心资产面临被中心化服务托管的风险。

对于强调用户自主管理资产、身份和权限的去中心化方向而言，这种信任集中化趋势是一种技术倒退。市场亟需一种机制，把敏感操作从云端拉回用户可控的本地设备，同时保留 Agent 协作的便利性。

易安与 StoryLock 正是基于这一痛点而生：远端 Agent 可以提出请求，私人智能助理解释风险，StoryLock 本地核心完成授权与敏感执行，用户始终在本地设备上掌握最终确认权。

### 1.2 作品主要特点

| 特点 | 说明 |
|------|------|
| **本地优先** | 签名、密码填充、API 调用、证书签名等敏感操作完全在本地设备执行，远端不接触密钥材料 |
| **故事化验证** | 用户以精心设计的私密故事概况替代传统固定符号口令，通过九宫格动态 challenge 完成授权确认 |
| **三层隔离** | 第一层处理文件加密与操作授权；第二层封装底层操作与风险评估；第三层作为远程网关统一对外入口 |
| **动态挑战** | 24 题题集随机组合生成九宫格 challenge，每次验证题目不同，避免固定口令的泄露风险 |
| **终身免轮换** | 单故事终身制，通过题集渐进式淘汰实现安全更新，无需用户定期更换记忆内容 |
| **可审计脱敏** | EIP-712 结构化请求、requestId/nonce/expiry 防重放、SQLite 审计日志、递归脱敏确保第三方 Agent 无法接触秘密材料 |

## 二、作品应用场景

### 2.1 创业者本地凭据保险库

**问题**：创业者需管理大量分散的敏感凭据——网站后台、支付平台、API 服务、签名密钥等，传统方式或明文存储于文档，或依赖云端密码管理器，均存在泄露风险；团队协作时凭据传递缺乏安全边界。

**解决方案**：
- 所有凭据由 StoryLock 本地核心加密存储，密钥由用户私密故事派生，不上传任何云端
- 凭据按类型统一资源目录管理：
  - 网站账号：`credential/<provider>/<instance>/<role>`
  - API 凭据：`api/<provider>/<instance>/<role>`
  - 签名密钥：`certificate/<provider>/<instance>/<role>`
- 每次访问需经九宫格 challenge 确认，私人智能助理展示访问目标与风险等级
- 支持资源目录导出与迁移，更换设备时凭据包可安全转移

**适合存放**：
1. 网站后台账号密码
2. 三方平台 API Key
3. 运维口令
4. 发布签名密钥
5. 支付商户后台凭据

**受众**：独立创业者、小型团队负责人、自由职业者

---

### 2.2 Agent 高风险操作人工确认机制

**问题**：自动化 Agent 执行交易、投放、结算等操作时，缺乏有效的人工确认环节，误操作或恶意指令可能导致资金损失、数据泄露、品牌风险。

**解决方案**：
- Agent 的高风险操作请求经易安远程入口送达本地，私人智能助理解析请求内容、来源、风险等级
- 用户通过九宫格 challenge 完成身份确认，StoryLock 本地执行敏感操作，仅返回脱敏结果
- 操作全程记录审计日志，支持事后追溯与责任界定
- 支持对象强度策略分级：低风险操作可配置短时会话，高风险操作强制每次确认

**适合场景**：
1. 自动化交易（DeFi 兑换、转账、合约交互）
2. 广告投放确认（预算调整、定向修改、素材替换）
3. 订单/结算/退款高风险操作
4. 私域用户批量运营动作（群发、标签修改、权益发放）

**受众**：交易员、运营人员、电商卖家、社区管理者

---

### 2.3 网站登录与运维面板填充

**问题**：开发者与运维人员需频繁登录多个后台系统，密码或分散记忆或重复使用，效率低且安全性差；低频高价值后台（如支付后台、域名管理）更易因遗忘密码而依赖不安全重置流程。

**解决方案**：
- 登录凭据由 StoryLock 本地核心加密存储，按站点与账号隔离管理
- 登录模板通过 `resourceId + role` 映射字段，支持用户名密码型、邮箱密码型、带二次校验型等多种形态
- 浏览器或 Agent 发起登录请求时，经本地九宫格 challenge 确认后自动填充
- 支持特定系统的自定义字段映射，适配非标准登录表单

**适合场景**：
1. 登录表单自动填充（GitHub、AWS 控制台、云平台后台）
2. 特定系统的账号字段映射（企业内网、自研后台、行业专用系统）
3. 低频高价值后台访问（支付商户后台、域名注册商、ICP 备案系统）

**受众**：全栈开发者、运维工程师、系统管理员

---

### 2.4 开发者工具链的敏感配置管理

**问题**：开发者在本地环境、构建流水线、部署脚本中分散管理 secrets（数据库密码、API 密钥、签名证书），硬编码或环境变量泄露风险高；个人创业 SaaS 的配置保管缺乏统一安全策略。

**解决方案**：
- 本地开发环境 secrets 由 StoryLock 统一管理，通过环境变量注入或文件挂载方式提供给应用，不暴露于版本控制
- 构建签名时，私钥由本地核心调用，支持构建后本地签名结果输出，或兼容模式下生成签名引用
- 轻量部署凭据（如 Docker 环境变量、K8s Secret）通过本地授权后安全注入
- 个人创业 SaaS 的配置保管采用资源目录统一组织，支持多环境（dev/staging/prod）隔离

**适合场景**：
1. 本地开发环境 secrets（数据库连接串、缓存密码、消息队列凭据）
2. 构建挑战后本地签名结果，或在兼容模式下构建签名引用
3. 轻量部署凭据（VPS 登录密钥、容器仓库 Token、CDN 刷新密钥）
4. 个人创业 SaaS 的配置保管（支付渠道密钥、短信服务 API、邮件服务 Token）

**受众**：独立开发者、开源维护者、个人 SaaS 创业者

---

### 2.5 多资源协同工作流（扩展场景）

**问题**：真实业务往往需要同一流程中串联多种敏感操作，例如"登录 GitHub → 构建签名 → 上传 AWS → 发布版本"，传统方案在每个环节都需要人工干预或暴露凭据。

**解决方案**：
- StoryLock 支持在同一工作流中编排多种资源类型（website_account、certificate_bundle、api_client）
- 每个步骤独立经过九宫格 challenge 确认，用户可按需调整步骤顺序和资源组合
- 工作流模板只描述动作和资源引用，不保存秘密值，真正执行时由宿主解析为本地对象
- 支持权限边界按"单对象 / 多对象 / 高敏感材料协同"分级显示

**典型流程**：
1. 登录 GitHub 查看发布仓库
2. 使用 Apple 开发证书签署构建产物
3. 使用 AWS 生产凭据上传构建产物
4. 使用 GitHub Token 发布版本说明

**受众**：全栈开发者、独立创作者、小型技术团队

---

## 三、总结

易安与 StoryLock 的核心价值在于**重建用户对敏感数字资产的控制权**。在 Agent 自动化趋势下，传统方案要求用户将信任让渡给远端服务，而 StoryLock 通过本地优先架构、故事化验证、动态挑战机制，让用户在享受自动化便利的同时，始终掌握密钥与授权的终极决策权。

**一句话定位**：StoryLock 是 Agent 时代的个人数字保险箱——远端可以请求，但打开保险箱的钥匙，永远只在用户手中。
