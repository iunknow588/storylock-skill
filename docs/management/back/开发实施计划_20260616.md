## 目的

本文档用于把参赛说明、中文设计文档和 `skill/src` 当前代码现状转成可执行的开发顺序。

当前开发主线不是继续扩展概念，而是把系统收敛为三层清晰能力：

1. 第一层：本地处理与记忆线索相关能力
2. 第二层：对象强度判断、九宫格验证与本地授权
3. 第三层：签名授权、Web2 密码填充等远程请求包装与代理授权

## 当前代码现状

### 已具备基础

| 模块 | 路径 | 当前能力 |
| --- | --- | --- |
| 共享加密 | `skill/src/shared/crypto.js` | AES-256-GCM、HKDF-SHA256、HMAC-SHA256 |
| SecretStore | `skill/src/shared/secret-store.js` | Memory、Windows Credential、Linux Secret Service 适配 |
| SQLite | `skill/src/shared/sqlite.js`、`sqlite-schema.sql` | schema 加载、迁移、请求与审计字段 |
| 第一层 | `skill/src/skills/local-story-processing` | 草稿生成、润色、记忆线索处理边界 |
| 第二层 | `skill/src/skills/local-story-access` | 已有 challenge、session、防重放、锁定、故事读写原型 |
| 第三层 | `skill/src/skills/remote-gateway` | 已有远程请求包装、EIP-712 结构、密码填充包装原型 |
| 汇总入口 | `skill/src/engine` | demo、自测、WASM 构建与验证脚本 |

### 需要收敛的历史口径

当前代码仍保留早期接口和实现名称：

1. `requestStoryRead`
2. `requestStoryWrite`
3. `requestChallengeSign`
4. `requestCapabilityStatus`
5. `requestLocalStoryAssist`
6. `queryStoryMetadata`
7. `StoryReadAccessSkill`
8. `StoryWriteAccessSkill`
9. `LocalStoryAssistAccessSkill`

这些接口可作为历史实现参考，但不再作为参赛说明中的主线接口。后续开发应收敛为：

1. 第二层：`ObjectStrengthPolicySkill`
2. 第二层：`GridChallengeSkill`
3. 第二层：`LocalAuthorizationSkill`
4. 第三层：`requestSignature`
5. 第三层：`requestPasswordFill`

## 总体策略

当前阶段采用：

1. 先冻结接口边界，再改代码命名与实现
2. 先共享安全基础设施，后第二层本地授权
3. 先第二层对象强度与九宫格，再第三层签名和密码填充包装
4. 第一层只保留本地处理与记忆线索能力，不参与授权执行

原因：

1. 第二层是本地授权安全边界核心
2. 第三层必须依赖第二层稳定授权结果
3. 网关不应承担故事读取或写回职责
4. `challenge` 是一种本地验证实现，不应出现在第三层签名接口名中

## 开发里程碑

### M0：文档与接口口径冻结

目标：

1. 以参赛说明为准冻结三层职责
2. 更新 `skill/docs/design/cn` 中的核心设计文档
3. 明确历史接口与目标接口的迁移关系

交付物：

1. `docs/design/cn/README.md`
2. `docs/design/cn/Skill定位与边界.md`
3. `docs/design/cn/系统Skill表与能力边界.md`
4. `docs/design/cn/三包接口契约.md`
5. 本开发实施计划

完成标准：

1. 设计文档不再把第三层描述为故事读取或故事写回入口
2. 设计文档不再把第二层描述为业务内容读写层
3. 第三层主接口收敛为 `requestSignature` 与 `requestPasswordFill`
4. 第二层目标能力收敛为对象强度判断、九宫格验证与本地授权

### M1：共享安全基础设施确认

目标：

1. 复核 `SecretStore`
2. 复核 SQLite schema 与迁移
3. 复核 AES-256-GCM、HKDF-SHA256、HMAC-SHA256
4. 明确开发模式与平台 SecretStore 的边界

交付物：

1. `shared/crypto.js`
2. `shared/secret-store.js`
3. `shared/sqlite.js`
4. `shared/sqlite-schema.sql`
5. SecretStore 可用性说明

完成标准：

1. 能创建或读取 `masterSalt`
2. 能完成 AES-256-GCM 加密与解密
3. 能基于 HKDF 派生用途隔离密钥
4. 能计算本地答案摘要
5. 持久化数据库必须使用平台 SecretStore 或显式开发模式

### M2：第二层目标接口实现

目标：

1. 从历史 `StoryReadAccessSkill / StoryWriteAccessSkill` 口径收敛为本地授权口径
2. 实现对象强度判断
3. 实现九宫格验证生成
4. 实现本地答案校验与授权结果返回
5. 保留 requestId / nonce / expiry 防重放
6. 保留失败计数、锁定与自动解锁

交付物：

1. `ObjectStrengthPolicySkill`
2. `GridChallengeSkill`
3. `LocalAuthorizationSkill`
4. 第二层自测脚本
5. 历史接口兼容或迁移说明

完成标准：

1. 输入目标对象后能返回所需密码强度
2. 能根据密码强度生成九宫格验证
3. 能提交答案并得到授权通过或失败结果
4. 能拒绝过期请求
5. 能拒绝 nonce 或 requestId 重放
6. 连续失败后进入锁定状态并返回 `retryAfter`
7. 锁定窗口到期后能自动恢复
8. 第二层不直接承担故事读取或写回职责

### M3：第三层网关接口收敛

目标：

1. 新增或迁移 `requestSignature`
2. 保留并完善 `requestPasswordFill`
3. 将 `requestChallengeSign` 降级为兼容别名或内部过渡接口
4. 从主线自测中移除 `requestCapabilityStatus`、`requestStoryRead`、`requestStoryWrite`
5. EIP-712 请求结构改为签名请求语义，不绑定 challenge 命名

交付物：

1. `storylock-remote-gateway-skill/index.js`
2. `storylock-remote-gateway-skill/scripts/selftest.mjs`
3. `StoryLockSignatureRequest` 结构化签名请求
4. 第三层接口迁移说明

完成标准：

1. `requestSignature` 能构造统一请求信封
2. `requestSignature` 的 `capability` 为 `requestSignature`
3. `requestSignature` 的 `scope` 不再使用 `challenge_sign`
4. `requestPasswordFill` 不返回明文密码
5. 网关不暴露故事读写主接口
6. 网关不直接持有私钥、文件密钥、凭据明文或九宫格答案

### M4：第一层本地处理能力整理

目标：

1. 保留故事草稿生成、故事润色、题集强度评估
2. 明确第一层只处理本地已批准输入
3. 第一层不直接参与授权、签名或密码填充

交付物：

1. `StoryDraftAssistSkill`
2. `StoryRefineAssistSkill`
3. `StrengthReviewSkill`
4. 第一层自测或汇总自测覆盖

完成标准：

1. 能生成结构化草稿
2. 能润色结构化草稿
3. 能输出题集强度评估结果
4. 输出边界中明确不创建授权、不签发 session、不返回秘密

### M5：汇总入口、demo 与自测对齐

目标：

1. `storylock-skill-engine` demo 对齐新三层主线
2. 自测脚本覆盖第二层和第三层目标接口
3. 文档、代码、脚本口径一致

交付物：

1. `skill/src/engine`
2. `npm run demo`
3. `npm run selftest`
4. `npm run build:wasm`
5. `npm run selftest:wasm`

完成标准：

1. demo 展示草稿生成、九宫格授权、签名请求、密码填充请求
2. selftest 不再依赖故事读写作为主线能力
3. WASM 构建与 dist 自测仍可运行
4. 文档中的接口名与自测中的接口名一致

## 当前阶段验收标准

当前阶段不以“功能很多”为目标，而以“边界正确”验收。

至少应满足：

1. 九宫格答案不明文持久化
2. requestId / nonce 重放会被拒绝
3. 第二层能根据目标对象给出密码强度
4. 第二层能生成九宫格验证并返回授权结果
5. 第三层主接口为 `requestSignature` 与 `requestPasswordFill`
6. 第三层不直接读取故事对象
7. 第三层不直接持有私钥、文件密钥或凭据明文
8. `requestPasswordFill` 不向远程返回明文密码
9. `requestSignature` 不使用 challenge 作为接口命名
10. 文档、demo、自测中的主接口名称一致

## 当前风险

1. 代码中仍存在旧接口，短期内需要兼容策略，避免自测一次性大面积失效
2. 设计文档目录中仍有部分文档使用 challenge / story read-write 旧口径，需要后续逐篇收敛
3. 如果直接删除旧接口，可能影响现有 demo 和 schema，需要先新增目标接口再迁移自测
4. 第二层从故事读写转向本地授权后，需要重新定义对象强度与九宫格的数据结构

## 结论

当前最合理的下一步是：

1. 先完成设计文档核心口径收敛
2. 再在第二层实现对象强度、九宫格验证与本地授权
3. 然后在第三层新增 `requestSignature` 并保留 `requestPasswordFill`
4. 最后更新 demo、自测和 schema，形成参赛说明一致的系统主线
