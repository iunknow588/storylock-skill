### 1.3 关键技术决策评审（修订后）

#### ✅ 已确认正确（新增）

1. **EIP-712 占位符警告**
   - 文档已增加 ⚠️ 警告："当前定义只能作为结构化签名请求格式，不应被视为可直接用于生产链上验证的正式 domain"
   - 建议：代码实现时增加 `isPlaceholder` 布尔标记，构造 domain 时若 `isPlaceholder === true` 则抛出警告

2. **脱敏检查清单**
   - 新增 10 项最小脱敏检查清单（真实姓名、手机号、邮箱、身份证号、住址、地理位置、账号密码、私钥/seed、challenge answers、私密故事正文）
   - 建议执行顺序：结构化字段检查 → 正则匹配 → 人工确认/本地规则复核

3. **签名算法决策树**
   - 明确决策路径：EVM 链上 → `secp256k1`；通用消息/代理标识 → `ed25519`；合规要求 → `SM2`（后续扩展）
   - 当前阶段推荐：以太坊相关默认 `secp256k1`，内部非链上演示可用 `ed25519`

4. **内存清零实现**
   - 提供 `clearSensitiveBuffer(buf)` 和 `clearRawAnswers(rawAnswers)` 代码示例
   - 建议：校验完成后显式执行内存清理逻辑，再置 `null` 等待 GC

5. **Node.js >= 18 LTS**
   - README 已明确技术栈版本要求，支持原生 WebCrypto API

6. **接口契约完善**
   - 契约 C/D 的 `answers` 字段结构已定义（`questionId` + `answer` + `answerFormat`）
   - 契约 E 的 `algorithm` 已明确由第二层/本地网关做白名单校验

#### ⚠️ 仍需关注（剩余风险）

1. **Challenge 答案的内存安全**
   - 文档已提供 `clearSensitiveBuffer` 和 `clearRawAnswers` 示例，但 Node.js 的 GC 不保证立即回收
   - 建议：对于极高安全场景，考虑使用 `crypto.randomFillSync` 覆盖后再释放，或引入 `secure-buffer` 类库

2. **SQLite 并发控制**
   - 文档建议 SQLite 事务锁用于多进程场景，但未明确事务隔离级别
   - 建议：明确使用 `SERIALIZABLE` 隔离级别或显式 `BEGIN IMMEDIATE` 事务

3. **GCM nonce 存储**
   - 文档要求 nonce 必须与密文一起存储，但未明确存储格式
   - 建议：定义标准格式，如 `nonce(12 bytes) || ciphertext || tag(16 bytes)`，或 Base64 编码的 `{nonce, ciphertext}` 结构

## 二、市场分析（无变化）

### 2.1 目标市场定位

StoryLock 定位在"个人私密数据的安全访问与代理授权"市场，这是一个**新兴且高潜力**的细分领域：

- **上游**：个人密码管理器（1Password、Bitwarden）→ 市场已验证
- **中游**：Agent 代理授权（AI + 加密签名）→ 市场正在形成
- **下游**：Web3 钱包 + 智能账户（ERC-4337）→ 基础设施已就绪

### 2.2 竞品对标（无变化）

| 竞品/参考 | 优势 | StoryLock 差异化 |

### 4.2 剩余风险详述

#### 风险 1：Challenge 答案内存安全（🟡 中 → 部分缓解）

**问题**：文档已提供 `clearSensitiveBuffer` 和 `clearRawAnswers` 代码示例，但 Node.js 的 V8 引擎 GC 不保证立即回收，且字符串在 JavaScript 中是不可变的，可能存在内存驻留。

**建议**：
1. 对于极高安全场景，使用 `Buffer` 而非 `String` 存储敏感数据
2. 使用 `crypto.randomFillSync` 覆盖 Buffer 后再释放
3. 考虑引入 `secure-buffer` 或类似库，提供自动清零的 Buffer 包装
4. 在文档中增加"JavaScript 字符串不可变，敏感数据优先使用 Buffer"的警告

#### 风险 2：Session 预算绕过（🟡 中）

**问题**：文档已定义 `maxReads`/`writeBudget`，但未明确预算扣减的原子性。

**建议**：
1. 在 SQLite 中使用 `BEGIN IMMEDIATE` 事务确保预算扣减的原子性
2. 预算扣减和对象读取必须在同一事务中完成
3. 增加预算为 0 时的拒绝逻辑

#### 风险 3：SQLite 并发控制（🟢 低）

**问题**：文档建议 SQLite 事务锁，但未明确隔离级别。

**建议**：
1. 明确使用 `BEGIN IMMEDIATE`（SQLite 的悲观锁模式）
2. 或明确使用 `SERIALIZABLE` 隔离级别
3. 在 `Challenge状态机.md` 中补充并发控制的具体 SQL 示例

#### 风险 4：GCM nonce 存储格式（🟢 低）

**问题**：文档要求 nonce 必须与密文一起存储，但未定义标准格式。

**建议**：
1. 定义标准格式：`nonce(12 bytes) || ciphertext || tag(16 bytes)` 的拼接
2. 或 JSON 格式：`{ "nonce": "base64...", "ciphertext": "base64..." }`
3. 在 `安全规范.md` 中补充 nonce 存储格式章节
