# 单故事终身制密钥管理策略分析

**分析日期**: 2026-06-16  
**分析对象**: StoryLock 单故事终身制密钥管理策略  
**分析依据**: 用户明确需求 + 现有设计文档体系

---

## 一、用户核心需求提炼

用户提出的策略要点：

1. **单故事绑定**: 每个用户只维持一个故事
2. **零记忆负担**: 用户只需记住一个故事，管理所有密钥
3. **被动式安全**: 无风险不轮换，不主动打扰用户
4. **增量优化**: 风险出现时不是替换故事，而是在原故事基础上精确化
5. **题集扩展**: 满足条件时可增加问题，但核心故事不变

---

## 二、策略合理性分析

### 2.1 用户体验优势

| 维度 | 单故事终身制 | 传统定期轮换 |
|------|------------|------------|
| 记忆负担 | 低（一个故事） | 高（需记忆多个或定期更新） |
| 操作摩擦 | 低（无风险无感知） | 高（定期强制重设） |
| 心理安全感 | 高（故事是用户自己的） | 低（系统强加的密码规则） |
| 可用性 | 高（故事天然易记） | 中（随机密码难记） |
| 长期粘性 | 高（故事与身份绑定） | 低（纯工具关系） |

**结论**: 单故事终身制在用户体验维度全面优于传统定期轮换，符合"以人为中心"的安全设计原则。

### 2.2 安全性分析

#### 攻击面评估

| 攻击类型 | 风险等级 | 缓解机制 |
|----------|---------|----------|
| 暴力破解题集 | 中 | 24题题集 + 阈值控制（6/12/22）+ 锁定机制 |
| 社会工程学获取故事 | 中-高 | 故事不直接存储，仅存储派生摘要 |
| 侧信道攻击（内存Dump） | 中 | 答案仅内存存在，校验后立即清零 |
| 长期密钥老化 | 低 | masterSalt 终身不变，但 workKey 按需派生 |
| 题集被推测 | 低 | 题集从故事生成，无故事则无法推测 |

#### 关键安全假设

单故事终身制的安全性依赖于以下假设：

1. **故事足够私密**: 用户选择的故事内容对他人不可知
2. **题集生成足够分散**: 24题覆盖故事不同维度，降低单点泄露风险
3. **派生密钥强度足够**: HKDF-SHA256 从 masterSalt 派生，masterSalt 本身足够随机
4. **本地执行环境可信**: 答案验证、密钥派生均在本地完成

**结论**: 在"本地执行 + 题集阈值 + 内存清零"的组合下，单故事终身制的安全性不低于传统定期轮换，且因用户记忆更牢固，实际安全效果可能更好。

### 2.3 技术可行性分析

#### 密钥派生链

```
用户故事（唯一、终身）
    │
    ├── 故事规范化材料 → 稳定根上下文
    │       │
    │       ├── masterSalt（稳定根上下文的本地受控表示）
    │       │       └── HKDF-SHA256(masterSalt, identityId + purpose) → workKey
    │       │               └── AES-256-GCM(题集主档 / 秘密对象)
    │       │
    │       └── identitySalt = HMAC-SHA256(masterSalt, identityId)
    │               └── 用于答案验证和身份隔离
    │
    └── 题集主档（24题，可扩展至 24+N 题）
            └── 每题: 问题文本 + 答案摘要(HMAC-SHA256(identitySalt, normalizedAnswer))
```

#### 关键修正建议

为了避免后续实现误把“单故事终身制”理解成“直接拿故事哈希当全部密钥”，建议补充以下约束：

1. 不建议把原始故事文本简单哈希后直接当作所有加密密钥
2. 更合理的模型是：
   - 故事作为**稳定根上下文**
   - 题集作为**可扩展、可优化的验证表达层**
   - workKey / objectKey 作为**可按版本上下文派生的封装层**
3. 也就是说，单故事终身制强调的是“用户记忆根不变”，而不是“所有底层派生参数永远冻结不变”

#### 轮换对象分层

在单故事终身制下，应明确区分“根不变”和“外围可更新”。

| 层级 | 是否默认轮换 | 说明 |
|------|-------------|------|
| 故事根上下文 | 否 | 默认终身稳定，除非发生高风险泄露 |
| masterSalt / 根派生上下文 | 否 | 默认不主动轮换 |
| identitySalt | 否 | 跟随根上下文稳定存在 |
| 题集版本 | 是 | 可扩展、可标记 `active/deprecated/pending` |
| workKey 派生上下文版本 | 是 | 可通过 version tag 调整派生上下文 |
| objectKey 封装层 | 是 | 可在对象级重加密时更新 |

#### 单故事模式下的真正“轮换”含义

在 StoryLock 里，轮换不应默认等于“换故事”或“换根秘密”。

更准确地说，轮换分成三类：

1. **题集轮换**
   - 新增更精确的问题
   - 旧题渐进式淘汰
2. **派生上下文轮换**
   - 不改故事根
   - 只调整 `purpose` / `versionTag` / `objectScope`
3. **对象封装轮换**
   - 对题集主档或秘密对象重新加密
   - 更新 objectKey 或封装版本

只有在灾难性风险下，才考虑：

4. **根级重建**
   - 重新建立故事根上下文
   - 这属于“极少发生的灾难恢复”，而不是常规轮换

#### 题集扩展机制

| 场景 | 处理方式 | 用户感知 |
|------|---------|----------|
| 初始创建 | 从故事生成24题 | 回答初始题集 |
| 强度评估不足 | 引导补充细节 → 生成新题加入 | 回答更多问题 |
| 风险触发优化 | 保留原题 + 新增精确化题目 | 回答新增题目 |
| 旧题淘汰 | 标记deprecated，不再用于新challenge | 无感知 |

**技术可行性结论**: 完全可行。稳定根上下文和 `masterSalt` 默认不变，题集扩展不影响已有密钥派生链。版本标识仅用于题集管理与对象封装层，不触及用户记忆根。

#### 对象重加密策略

单故事终身制并不排斥对象重加密，反而更需要把“根稳定”与“封装可变”分开。

建议对象重加密按以下方式执行：

1. 故事根上下文保持不变
2. 基于新的 `versionTag` 或 `purpose` 派生新的 workKey / objectKey
3. 对题集主档或秘密对象重新加密
4. 保留旧版本短时兼容窗口
5. 完成迁移后把旧对象标记为 deprecated

这意味着：

1. 用户无需重新记住新故事
2. 系统仍可提升对象封装层安全性
3. 风险处置成本集中在本地迁移，而不是用户记忆重置

#### 版本兼容窗口建议

为了平滑迁移，建议引入以下兼容窗口：

| 对象 | 建议兼容期 | 说明 |
|------|-----------|------|
| 旧题集摘要 | 30-90 天 | 用于双版本 challenge 校验 |
| 旧对象密文 | 7-30 天 | 用于对象封装层迁移回滚 |
| deprecated 题 | 90 天 | 给用户留出记忆与行为过渡期 |

兼容窗口结束后：

1. 旧题不再参与新 challenge
2. 旧对象密文不再作为主读路径
3. 审计中保留版本迁移记录

## 三、潜在风险与缓解

### 3.1 故事泄露风险

**风险**: 用户故事被他人知晓（如口述、记录泄露、社交工程）。

**缓解**:
1. 故事本身不直接存储，仅存储派生的 masterSalt 和题集
2. 即使故事泄露，攻击者仍需通过 challenge（24题选答，阈值6/12/22）
3. 用户可主动触发优化，补充新细节使旧题集失效
4. 题集答案摘要不存储原文，需实际回答才能验证

### 3.2 题集可推测风险

**风险**: 从24题反推故事内容。

**缓解**:
1. 题集生成算法加入随机盐值，相同故事不同实例生成不同题集
2. 问题设计覆盖多维度（时间、地点、人物、感官、情感），降低单维度推断
3. 答案摘要不存储原文，无法从摘要看答案
4. 题集主档加密存储，需 workKey 解密

### 3.3 长期不使用的风险

**风险**: 用户长期不用，故事记忆模糊，导致无法通过 challenge。

**缓解**:
1. 12个月未使用时建议性提示，引导用户回顾故事
2. 提供"故事提示"功能（不泄露答案，仅唤醒记忆）
3. 保留恢复机制（如备用密钥或社交恢复，需更高阈值）

### 3.4 暴力破解风险

**风险**: 攻击者尝试大量答案组合。

**缓解**:
1. challenge 失败次数锁定（连续失败进入 locked 状态）
2. 指数退避锁定时间（1分钟 → 5分钟 → 15分钟 → 1小时）
3. 本地执行环境限制，远程无法直接发起 challenge
4. 题集24题，每次 challenge 随机选子集，降低重放价值

### 3.5 根级重建风险

**风险**: 用户故事根上下文被确认泄露，导致“单故事终身制”的前提失效。

**缓解**:
1. 将此情形视为灾难级事件，而非常规轮换
2. 明确区分“故事优化”与“根级重建”
3. 仅在高风险场景下才提示用户重新建立故事根
4. 根级重建后必须同时：
   - 重建题集
   - 重建 identitySalt
   - 重建对象封装层
   - 终止旧版本 challenge 兼容窗口

## 四、与传统密钥管理对比

| 维度 | 单故事终身制 | 传统密码 + 定期轮换 | 硬件密钥（YubiKey等） |
|------|------------|-------------------|-------------------|
| 记忆负担 | 低（一个故事） | 高（多个密码，定期更换） | 极低（物理持有） |
| 可用性 | 高（任何设备，只需回忆） | 中（需密码管理器） | 中（需携带硬件） |
| 安全性 | 高（本地执行 + 题集阈值） | 中（依赖密码强度） | 极高（物理隔离） |
| 恢复能力 | 中（故事记忆） | 低（密码遗忘） | 低（硬件丢失） |
| 成本 | 低（纯软件） | 低（纯软件） | 高（硬件采购） |
| 用户体验 | 优秀（自然、无摩擦） | 差（规则复杂、频繁打扰） | 良好（简单但需硬件） |

**结论**: 单故事终身制在"安全性-可用性-成本"三角中取得了优秀平衡，特别适合 StoryLock 面向个人用户的定位。

## 五、实施建议

### 5.1 必须实现的机制

1. **masterSalt 终身存储**: 操作系统密钥链，不可删除，不可导出
2. **题集版本管理**: 每道题标记 `active`/`deprecated`/`pending`
3. **Challenge 选择策略**: 优先从 `active` 题中随机选择
4. **答案验证多版本兼容**: 同一 identity 的历史答案摘要按版本隔离
5. **风险检测与提示**: 暴力破解检测 → 强制优化；长期未用 → 建议性提示
6. **对象封装版本管理**: 题集主档和秘密对象支持封装版本标记
7. **根级重建入口**: 明确极端风险下的灾难恢复流程

### 5.2 建议增加的设计

1. **故事强度评估器**: 初始创建时评估故事质量，引导用户补充细节
2. **题集生成器**: 从故事生成24题，确保覆盖多维度、避免可推测性
3. **优化引导对话**: 风险触发时，通过对话引导用户补充细节而非更换故事
4. **记忆唤醒提示**: 长期未用时，提供不泄露答案的提示信息
5. **版本迁移器**: 支持题集版本和对象封装版本的双版本兼容迁移
6. **风险分级器**: 区分“可优化”与“必须重建”的边界

### 5.3 需要明确的边界

1. **题集上限**: 建议最大扩展至 48 题（24+24），避免无限膨胀
2. **deprecated 题保留期**: 建议保留 90 天，确保用户有足够时间过渡
3. **优化频率限制**: 建议两次优化间隔至少 30 天，避免频繁打扰
4. **masterSalt 备份**: 明确禁止导出，恢复只能通过重新输入故事
5. **根级重建触发条件**: 仅限故事根确认泄露、长期失效或用户主动重建
6. **对象迁移兼容期**: 建议 7-30 天，避免旧密文长期悬挂

### 5.4 轮换判定矩阵

| 事件 | 是否需要换故事 | 是否需要扩题 | 是否需要对象重加密 | 是否需要根级重建 |
|------|---------------|-------------|-------------------|----------------|
| 题集强度不足 | 否 | 是 | 可选 | 否 |
| 长期未使用 | 否 | 可选 | 否 | 否 |
| 发现部分题可推测 | 否 | 是 | 可选 | 否 |
| 秘密对象封装策略升级 | 否 | 否 | 是 | 否 |
| 故事根被确认泄露 | 是 | 是 | 是 | 是 |

## 六、最终结论

**用户提出的单故事终身制策略是合理、可行且优秀的。**

### 核心优势

1. **用户体验卓越**: 零记忆负担，无摩擦，符合人类自然记忆方式
2. **安全性足够**: 在本地执行 + 题集阈值 + 内存清零的组合下，安全性不低于传统方案
3. **技术完全可行**: masterSalt 终身不变，题集扩展不影响密钥派生链
4. **差异化竞争力**: 区别于传统密码管理器和硬件密钥，形成独特价值主张

### 关键成功因素

1. **题集生成质量**: 必须确保24题足够分散、不可推测、覆盖多维度
2. **风险检测准确**: 暴力破解检测必须及时，避免真实攻击成功
3. **优化引导体验**: 风险触发时的对话引导必须温和，避免用户焦虑
4. **实现严格正确**: masterSalt 不可删除、答案仅内存、挑战锁定等机制必须零漏洞

### 建议

**采纳单故事终身制作为 StoryLock 的核心密钥管理策略，并已在安全规范.md 中更新。**

后续实现重点：
1. M1: 实现 masterSalt 终身存储 + 题集版本标记
2. M2: 实现 challenge 状态机 + 题集选择策略（优先 active）
3. M3: 实现故事强度评估 + 优化引导对话
4. M4: 实现风险检测 + 暴力破解锁定
