# StoryLock Pharos 适配评估报告

| 项目 | 内容 |
| --- | --- |
| 评估日期 | 2026-06-16 |
| 评估对象 | StoryLock 三层 Skill 架构与 Pharos Skill Engine 的适配性 |
| 评估结论 | 第三层适配度高；第一层、第二层不建议交由 Pharos 承担本地安全边界 |

## 目的

本文档用于回答以下问题：

1. StoryLock 是否需要把三个 Skill 包都做成 Pharos Skill
2. Pharos Skill Engine 适合放在哪一层
3. 哪些边界必须由 StoryLock 自己实现

## 总体结论

当前阶段建议采用：

1. **第一层** 不依赖 Pharos
2. **第二层** 不依赖 Pharos
3. **第三层** 可按 Pharos Skill 包方式组织与对接

原因很直接：

1. 第一层处理原始私密故事内容
2. 第二层处理 challenge、session、secret object、signing material
3. 第三层才是远程包装、第三方接入、能力编排的天然位置

## 分层适配判断

| 层级 | 是否适合交给 Pharos | 结论 |
| --- | --- | --- |
| 第一层：本地故事处理 | 低 | 不建议 |
| 第二层：本地访问控制 | 很低 | 不建议 |
| 第三层：远程网关 / 代理授权 / 编排 | 高 | 建议 |

## 第一层评估

第一层负责：

1. 私密故事编辑
2. 私密草稿润色
3. 原始记忆材料处理

不适合交给 Pharos 的原因：

1. 本层真正的约束是“原始内容不出本地”
2. Pharos 提供的是 Skill 包组织与调用规范，不提供本地秘密处理保证
3. 就算做成 Pharos Skill，也不能替代本地安全边界

## 第二层评估

第二层负责：

1. `createChallenge`
2. `submitChallengeAnswers`
3. `session` 签发与预算控制
4. `readSecretObject`
5. `writeSecretObject`
6. `requestChallengeSign`

不适合交给 Pharos 的原因：

1. 本层是 StoryLock 真正的敏感执行边界
2. challenge answers、session、密钥使用都必须在本地受控环境中完成
3. Pharos 不能替代本地 Host 的授权控制与密钥隔离

## 第三层评估

第三层负责：

1. 接收第三方 Skill 请求
2. 结构化请求包装
3. 远程能力编排
4. 本地结果回传
5. 最小化审计与结果封装

适合采用 Pharos 的原因：

1. Pharos 强项正是 Skill 的自说明、能力索引、参考文件组织
2. 第三层本身就是“面向 Agent 的包装层”
3. 第三方集成时，Pharos 反而能降低接入门槛

## Pharos 可承担的能力

当前阶段建议由第三层 Pharos Skill 承担：

1. Capability Index
2. 用户意图到能力映射
3. 结构化请求模板
4. 第三方 Skill 接入说明
5. 远程网关能力描述

## Pharos 不应承担的能力

当前阶段不建议交给 Pharos 的部分：

1. 本地 `SecretStore`
2. challenge 答案验证
3. session 状态机
4. secret object 实际读取
5. 本地签名密钥使用

这些必须继续由 StoryLock 本地 Host 和第二层 Skill 自己实现。

## 推荐落地方式

建议采用下面的组合：

### 包 1：`storylock-local-story-processing-skill`

定位：

1. 本地处理包
2. 不要求做成 Pharos Skill

### 包 2：`storylock-local-story-access-skill`

定位：

1. 本地访问控制包
2. 不要求做成 Pharos Skill

### 包 3：`storylock-remote-gateway-skill`

定位：

1. 远程 Agent 包装层
2. 建议按 Pharos Skill 规范组织

## 风险提示

若把第一层或第二层也强行做成 Pharos 的主要运行边界，会出现以下问题：

1. 文档组织和本地安全边界被混淆
2. 开发者误以为 Skill 包结构本身等于本地密钥保护
3. 第三方框架能力被误当作敏感执行保障

## 最终建议

当前阶段最合适的策略不是“StoryLock 全部依赖 Pharos”，而是：

1. **第一层、第二层由 StoryLock 自己实现安全边界**
2. **第三层使用 Pharos 作为面向 Agent 的接入壳层**
3. **三包继续保持分离，调用链固定为 3 -> 2 -> 1**

这也是当前文档体系与实现路径最一致的方案。
