# 多平台本地宿主与下载入口方案

日期：2026-06-19

## 1. 当前结论

当前易安下载入口仍是单平台入口：

1. `/app/download` 实际进入 `src/skills/remote-gateway/web-api-handler.js`。
2. 当前优先返回或重定向 Android APK。
3. 内置静态兜底文件名为 `storylock-android-host-0.1.0-1-debug.apk`。
4. 公开元数据字段只覆盖 Android：`STORYLOCK_ANDROID_APP_DOWNLOAD_URL`、`STORYLOCK_ANDROID_APK_PATH`、`STORYLOCK_ANDROID_APK_VERSION`、`STORYLOCK_ANDROID_APK_CHECKSUM` 等。
5. 仓库已有 `android-host/` 真实 Android 宿主骨架，但没有 `windows-host/` 或 desktop host 目录。

因此，如果用户在 Windows 本地电脑上点击下载，当前并不会拿到 Windows 可直接运行版本，而是会下载 Android APK 或跳转到 Android APK 地址。这个体验需要改为“平台选择 + 明确产物类型 + 对应本地宿主”。

## 2. Windows 端实现推荐

推荐路线：Windows 本地宿主优先使用 Rust。

原因：

1. 普通用户需要“下载后直接运行”，Rust 更适合产出单文件 `.exe` 或 `.msi`，不要求用户预装 Python。
2. Windows 常驻程序、托盘、开机启动、Windows Credential Manager / DPAPI、localhost HTTP server、系统通知等能力，Rust 生态更适合做长期产品化。
3. 安全边界更清晰：可以把本地 HTTP host、relay poll、密钥存储、签名执行、日志审计打包在一个受控二进制中。
4. Python 适合保留为开发原型和自动化测试工具，但不建议作为面向普通用户的首选分发格式。

建议技术分层：

1. `windows-host/`：Rust 产品化本地宿主。
2. `src/skills/local-story-access/android-host-server.js`：继续保留为跨平台 mock / selftest host。
3. 后续可增加 `desktop-host-python/` 或 `scripts/windows/mock_host.py`，只用于调试，不作为正式下载入口。

## 3. 下载入口目标形态

易安网站应提供平台选择，而不是单一“下载 App”：

1. Windows：下载 Windows 本地宿主，文件类型 `.exe` / `.msi` / `.zip`。
2. Android：下载 Android APK。
3. `/app/download`：返回平台选择页，避免电脑用户误下 APK。
4. `/app/download/windows`：Windows 产物下载或跳转。
5. `/app/download/android`：Android APK 下载或跳转。
6. `/download/android-host`：保留兼容旧链接。
7. `/download/windows-host`：保留 Windows 兼容直链。

## 4. Windows 本地宿主契约

Windows Host 应与 Android Host 共用云端注册和 relay 契约：

1. `POST /local-host/register` 作为主注册入口，同时保留 `/android-host/register` 兼容旧客户端。
2. `POST /local-host/relay/poll` 作为主 relay 拉取入口，同时保留 `/android-host/relay/poll` 兼容旧客户端。
3. `POST /local-host/relay/respond` 作为主 relay 回传入口，同时保留 `/android-host/relay/respond` 兼容旧客户端。
4. Windows 本地服务默认监听 `127.0.0.1:4510`。
5. 本地健康检查：`GET /health`。
6. 本地执行入口：`POST /execute`。
7. 默认连接方式：`relay_url`，避免公网直连和 NAT 问题。
