## 二、单故事终身制与锁钥模型（核心设计原则）

### 2.1 基本概念

| 概念 | 类比 | 说明 |
|------|------|------|
| **文件** | 锁 | 加密对象存储，包含题集 |
| **密钥** | 钥匙 | 用户脑中记忆，从故事回忆 |
| **故事** | 记忆辅助 | 帮助用户记住钥匙 |

### 2.2 核心原则

- **故事根默认稳定**：用户长期只维持一个故事根上下文，不应把常规优化误解为“频繁换故事”
- **一把钥匙开一把锁**：每个文件有且仅有对应的密钥
- **锁钥一一对应**：文件与密钥绑定，不可混用
- **外围可演进，根不轻动**：题集版本、派生上下文、对象封装层可升级；故事根与 masterSalt 默认不主动轮换
- **根级重建是灾难恢复，不是常规轮换**：只有在故事根确认泄露或不可恢复失效时，才进入根级重建流程

### 2.3 分层轮换模型

```
常规优化 != 更换故事

常规优化包含三类：
1. 题集轮换：新增题、淘汰旧题、调整 active/deprecated/pending
2. 派生上下文轮换：调整 purpose / versionTag / objectScope
3. 对象封装轮换：对题集主档或秘密对象重加密

只有灾难场景才进入：
4. 根级重建：重建故事根上下文、masterSalt、identitySalt 与对象封装层
```

### 2.4 根级重建流程

```
根级重建 = 换根 + 换题集 + 换封装层

1. 保留旧锁（旧文件继续可用）
2. 建立新根（用户重新建立故事根上下文）
3. 重建题集主档与验证摘要
4. 预学习验证（确认用户掌握新钥匙）
5. 通过 → 新根启用，旧根进入兼容窗口后废弃
6. 未通过 → 新根丢弃，旧根继续有效
```

### 2.5 回退机制

- **常规优化失败**：仅回退题集版本、派生版本或对象封装版本，不触及故事根
- **根级重建未通过**：新根未启用，旧根旧钥匙继续有效
- **重建中断**：新根未完成，旧根旧钥匙继续有效
- **用户始终可用旧根访问旧锁**，直至兼容窗口关闭，无需额外“恢复”操作

## 三、高风险问题（必须解决）

### 问题 H1：单故事终身制下“常规轮换”与“根级重建”的边界仍需进一步冻结

**位置**：安全规范.md

**问题描述**：

当前设计已基本收敛为“单故事终身制 + 分层轮换”，即：
- 故事根上下文默认稳定
- masterSalt / identitySalt 默认不主动轮换
- 题集版本、派生上下文、对象封装层可以常规优化
- 仅在高风险泄露下才进入根级重建

但以下关键实现边界仍需进一步冻结：

1. **哪些事件属于常规优化，哪些事件属于根级重建**：例如题集强度不足、部分题可推测、对象封装升级、故事根泄露，必须有统一判定矩阵
2. **预学习强制流程适用于哪些场景**：是仅用于根级重建，还是题集大版本扩展时也要求执行
3. **兼容窗口与失效规则**：题集版本、旧对象密文、旧 session 在常规优化和根级重建下分别何时失效
4. **对象重加密与版本迁移的落点**：是否允许旧对象短时兼容，如何记录 active/deprecated/pending

**建议**：

在安全规范与实施计划中明确采用“分层轮换 + 根级重建”的冻结口径，并补充以下内容：

```markdown
## 单故事终身制下的轮换边界

### 默认稳定层
- 故事根上下文：默认终身稳定
- masterSalt：默认不主动轮换
- identitySalt：默认随根稳定存在

### 常规可轮换层
- 题集版本：支持 active / deprecated / pending
- 派生上下文版本：支持 purpose / versionTag / objectScope 演进
- 对象封装层：支持重加密迁移

### 根级重建触发条件
- 故事根被确认泄露
- 用户长期失效且无法通过常规优化恢复
- 用户主动发起完全重建

### 常规优化不触发换根
- 题集强度不足 → 扩题/调题
- 部分题可推测 → 题集优化
- 对象封装策略升级 → 对象重加密

### 根级重建必须执行
1. 建立新故事根上下文
2. 进行预学习验证（48 次打乱提问）
3. 重建题集与摘要
4. 重建对象封装层
5. 撤销旧 session，进入旧版本兼容窗口
6. 窗口结束后废弃旧根

### 兼容窗口建议
- 旧题集摘要：30-90 天
- 旧对象密文：7-30 天
- deprecated 题：90 天
```

**优先级**：P0 — 必须在 M1 安全基础设施阶段冻结

### 问题 H2：Challenge 状态机缺少明确的失败计数与锁定策略实现细节

**位置**：Challenge状态机.md

**问题描述**：

文档定义了 `maxRetryCount = 3` 和 `locked` 状态持续15分钟，但缺少以下关键实现细节：

1. **失败计数粒度**：是按 challengeId 计数、按 identityId 计数、还是按 IP/设备计数？
2. **锁定期间的拒绝策略**：locked 状态下是静默拒绝、返回特定错误码、还是引入指数退避？
3. **失败计数重置条件**：成功验证后是否重置？session 过期后是否重置？设备重启后是否重置？
4. **并发攻击场景**：若攻击者在锁定前快速发起多个 challenge，失败计数是否独立计算？

**风险**：实现阶段若各自理解不同，会导致安全边界不一致。例如：
- 若按 challengeId 计数，攻击者可无限创建新 challenge 绕过限制
- 若按 identityId 计数且永不重置，正常用户可能因偶尔手误被永久锁定

**建议**：

在 Challenge状态机.md 中增加"失败计数策略"专节，明确：

```
计数维度：identityId + 时间窗口（建议 24 小时滑动窗口）
最大重试：3 次 / 窗口
窗口重置：成功验证或 24 小时后
锁定策略：达到阈值后锁定 15 分钟，返回 SLG-003 并附带 retryAfter 字段
并发控制：同一 identityId 在 answers_submitted 状态下串行化，拒绝并行提交
```

**优先级**：P0 — 必须在 M2 第二层实现前冻结

### 问题 H3：EIP-712 占位符策略与生产升级路径存在断裂风险

**位置**：EIP-712最小请求定义.md

**问题描述**：

文档明确当前使用 `version = "1-placeholder"` 和 `verifyingContract = 零地址`，并提供了"生产环境检查清单"。但缺少以下关键衔接：

1. **版本切换的数据兼容性**：当从 placeholder 升级到正式版本时，已签名的历史请求是否仍有效？若无效，是否会导致已建立的 session 或已完成的操作被回溯拒绝？
2. **链上验证的渐进式接入**：当前阶段说"只作为结构化签名请求格式"，但下一阶段的"可链上验证签名请求"与当前阶段的接口契约如何衔接？
3. **domain 参数的注入方式**：`chainId` 和 `verifyingContract` 当前建议"由运行时配置注入"，但未定义配置来源、校验规则、降级策略

**风险**：若当前阶段的占位符实现与后续链上验证阶段不兼容，可能需要重构第三层接口契约，影响第一、二层调用方。

**建议**：

1. 在 EIP-712最小请求定义.md 中增加"版本兼容性策略"章节：
   - 明确 placeholder 版本的签名仅用于本地审计，不承诺链上可验证
   - 定义正式版本启用后的最小过渡期（建议 2 个版本周期）
   - 定义历史签名的保留策略（本地审计日志保留，但不用于新的链上验证）
2. 增加 domain 配置接口定义：
   ```ts
   interface DomainConfig {
     chainId: number;
     verifyingContract: string;
     version: string;
     source: "env" | "config" | "on-chain";
     validation: "strict" | "placeholder";
   }
   ```

**优先级**：P1 — 需在 M4 第三层实现前明确，避免接口契约冻结后无法升级

### 问题 H4：Session 预算扣减原子性假设与 SQLite 实际能力存在差距

**位置**：Session与防重放策略.md

**问题描述**：

文档提供了以下 SQLite 事务示例：

```sql
BEGIN IMMEDIATE;
SELECT read_budget FROM session_store WHERE session_id = :sessionId AND status = 'active';
UPDATE session_store SET read_budget = read_budget - 1 WHERE session_id = :sessionId AND read_budget > 0;
SELECT encrypted_object FROM protected_story_objects WHERE story_object_id = :storyObjectId;
COMMIT;
```

此示例存在以下问题：

1. **业务逻辑与 SQL 混合**：`read_budget <= 0` 的校验在应用层完成，但示例中注释"若 read_budget <= 0，则回滚并拒绝"，实际 SQL 中并无此逻辑
2. **SELECT 后 UPDATE 的竞态**：在 `SELECT read_budget` 和 `UPDATE` 之间，其他事务可能已修改该值，导致超扣
3. **对象读取与预算扣减的隔离**：示例中对象读取在预算扣减之后，若对象读取失败，预算已扣无法回滚

**建议**：

修正示例为原子操作：

```sql
BEGIN IMMEDIATE;

-- 原子扣减并返回扣减前值
UPDATE session_store 
SET read_budget = read_budget - 1 
WHERE session_id = :sessionId 
  AND status = 'active' 
  AND read_budget > 0 
  AND expires_at > :now;

-- 检查是否扣减成功
SELECT changes() AS affected_rows;

-- 若 affected_rows == 0，回滚并拒绝
-- 若 affected_rows == 1，继续读取对象

SELECT encrypted_object 
FROM protected_story_objects 
WHERE story_object_id = :storyObjectId;

COMMIT;
```

或使用 RETURNING 语法（SQLite 3.35.0+）：

```sql
UPDATE session_store 
SET read_budget = read_budget - 1 
WHERE session_id = :sessionId 
  AND status = 'active' 
  AND read_budget > 0 
  AND expires_at > :now
RETURNING read_budget;
```

**优先级**：P0 — 必须在 M2 实现阶段修正，否则预算控制可被绕过

### 问题 H5：第三层网关的接口暴露控制策略不明确

**位置**：三包接口契约.md 契约 E

**问题描述**：

文档提到："当前阶段签名请求中的 `algorithm` 建议由第二层或本地网关做校验，避免第三方任意声明算法。"

但存在以下模糊：

1. **"第二层或本地网关"是二选一还是都校验？** 若第三层已做校验，第二层是否重复？若第三层未做，第二层是否必须？
2. **允许暴露的接口未定义**：哪些第二层接口通过第三层暴露？哪些仅内部使用？
3. **算法与 keyId 的绑定关系**：同一 keyId 是否允许切换算法？

**风险**：若接口暴露控制策略不一致，可能导致敏感接口意外暴露，或合法接口被错误拦截。

**建议**：

1. 明确分层校验策略：
   - 第三层网关：接口暴露控制（定义哪些第二层接口对外暴露）
   - 第二层：语义校验（算法合法、keyId 支持该算法）
2. 定义当前阶段接口暴露列表：
   ```
   对外暴露：requestStoryRead, requestChallengeSign, queryStoryMetadata
   内部使用：createChallenge, submitChallengeAnswers, readSecretObject, deriveKey
   ```
3. 定义算法校验规则（第二层执行）：
   ```
   允许算法：ed25519, secp256k1
   待评估：rsa-pss, sm2
   明确禁止：md5, sha1, rsa-pkcs1-v1_5（已废弃）
   ```

**优先级**：P1 — 需在 M4 实现前冻结

### 问题 M2：Nonce 清理策略的窗口上限与存储增长风险

**位置**：Session与防重放策略.md

**问题描述**：

文档定义：
- 滑动窗口大小 = `2 x maxTTL`
- 默认最大窗口不超过 24 小时

但未定义：
1. **maxTTL 的最大值**：若 Host 策略允许 batch_session TTL = 30 分钟，窗口为 1 小时；若允许 2 小时，窗口为 4 小时。但如果 Host 策略被错误配置为 12 小时，窗口将达到 24 小时上限
2. **存储增长估算**：若每秒 100 个请求，24 小时窗口内 nonce 记录数 = 100 * 86400 = 864 万条。SQLite 单表 864 万条是否可接受？
3. **清理触发机制**：是定时任务？是启动时清理？是写入时触发？

**建议**：

1. 增加 Host 策略约束：
   ```
   maxSessionTtl: {
     default: 1800,      // 30 分钟
     max: 7200,          // 2 小时，硬上限
     privilegedMax: 1800  // 高敏会话 30 分钟
   }
   ```
2. 定义清理策略：
   ```
   触发方式：每次创建新 nonce 记录时，异步清理过期记录
   清理批次：每次最多删除 1000 条
   索引要求：nonce_store 必须按 createdAt 建索引
   ```

**优先级**：P2 — 在 M2 实现阶段补充

### 问题 M3：第一层本地故事处理的 LLM 依赖与脱敏策略存在实现断层

**问题描述**：

运行层级与Skill分层.md 建议：
- 当前阶段第一层"默认按本地规则处理/本地轻量模型/本地人工辅助理解"
- "若必须调用远程 LLM，前提是输入已脱敏"

## 六、故事学习机制（新增）

### 6.1 核心设计

基于用户反馈，StoryLock 采用以下学习机制：

| 阶段 | 题目数 | 校验方式 | 目的 |
|------|--------|----------|------|
| 预学习 | 48次（24题×2遍，打乱顺序） | 错误≤2次通过 | 确保可靠记忆才存储 |
| 初始期 | 22题 | 每天1次 | 巩固记忆 |
| 巩固期 | 22题 | 每2天1次 | 继续巩固 |
| 适应期 | 22题 | 每周1次 | 过渡到长期记忆 |
| 稳定期 | 22题 | 每月1次 | 维持记忆 |
| 长期期 | 22题 | 每年1次 | 定期检验 |

### 6.2 预学习阶段（保存前强制）

```markdown
## 预学习阶段（Pre-Learning Phase）

### 目标
确保用户已可靠掌握当前生效的故事根上下文或重建后的新根上下文，避免启用后因遗忘导致无法访问。

### 流程
1. **生成打乱题集**
   - 从 24 题中随机抽取，打乱顺序
   - 每题出现 2 次（共 48 次提问）
   - 两次出现间隔至少 12 题，避免短期记忆

2. **用户回答**
   - 每次回答后立即反馈正确/错误
   - 错误时显示正确答案，要求用户复述一次

3. **通过标准**
   - 48 次中错误 ≤ 2 次（准确率 ≥ 95.8%）
   - 同一题两次均错误 = 该题标记为"薄弱项"
   - 薄弱项 > 3 题 = 预学习失败

4. **失败处理**
   - 新版本或新根丢弃，旧版本或旧根继续有效
   - 用户可立即重试（重新生成打乱题集）或放弃

5. **通过处理**
   - 初始创建时：启用首个故事根与题集版本
   - 根级重建时：启用新根并撤销旧 session，旧根进入兼容窗口
   - 启动正式学习机制（初始期 3 天）
```

|------|------|------|

**建议决策**：

✅ **批准进入开发阶段**，但附带条件：
