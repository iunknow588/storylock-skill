# StoryLock 中 KMS 保存用户密钥对的风险分析

## 一、核心问题

**用户担忧：如果 KMS 系统保存用户的密钥对，会有什么风险？**

**核心回答**：
- **StoryLock 设计中，KMS 不保存用户的密钥对**
- 用户的密钥对（公钥+私钥）由**用户本地生成**，**本地存储**
- KMS 仅保存**验证用的公钥**（用于验证用户签名），不保存私钥
- 但用户提出了一个关键假设：如果 KMS 保存了用户的密钥对，会怎样？

**一句话总结**：如果 KMS 保存用户密钥对，将**完全破坏 StoryLock 的"本地优先"安全模型**，导致信任基础崩塌。

---

## 二、密钥对归属分析

### 2.1 StoryLock 中的两类密钥对

| 密钥对 | 所有者 | 生成位置 | 存储位置 | 用途 |
|--------|--------|---------|---------|------|
| **用户身份密钥对** | 用户 | 本地设备 | 本地 SecretStore | 用户身份签名、设备绑定 |
| **EIP-712 验证密钥对** | 系统 | KMS/HSM | KMS（公钥）+ HSM（私钥） | 验证 Agent 请求签名 |

### 2.2 用户密钥对的本地生成流程

```
用户注册/设备绑定：

1. 本地设备生成密钥对（ECC P-256）
   ├─ 私钥：随机生成，仅存在于本地内存
   ├─ 公钥：从私钥派生，可导出
   └─ 生成过程：本地完成，不上传任何数据

2. 私钥存储
   ├─ Windows：DPAPI 加密后存入系统密钥链
   ├─ Android：Android Keystore 硬件保护
   └─  macOS：Keychain 加密存储

3. 公钥注册
   ├─ 公钥上传到 FunctionGraph（仅公钥！）
   ├─ FunctionGraph 将公钥存入 KMS（作为验证材料）
   └─ 私钥从未离开本地设备

4. 后续使用
   ├─ 签名操作：本地使用私钥签名
   ├─ 验证操作：FunctionGraph 使用 KMS 中的公钥验证
   └─ 私钥始终本地，从未上传
```

## 三、如果 KMS 保存用户密钥对的风险分析

### 3.1 假设场景：KMS 保存了用户私钥

```
假设：KMS 中存储了用户的完整密钥对（公钥+私钥）

风险 1：系统管理员可以冒充用户
├─ 华为云运维人员可以访问 KMS
├─ 从 KMS 导出用户私钥
├─ 使用私钥签名请求
├─ FunctionGraph 验证通过（因为公钥匹配）
└─ 系统管理员成功冒充用户，绕过九宫格挑战

风险 2：KMS 被攻破 = 所有用户身份泄露
├─ 攻击者攻破 KMS 系统
├─ 批量导出所有用户私钥
├─ 可以冒充任何用户
├─ 完全绕过第二层本地授权
└─ StoryLock 安全架构彻底失效

风险 3：合规与信任崩塌
├─ 用户无法确认系统是否持有自己的私钥
├─ "不信任任何第三方"承诺成为空话
├─ 用户无法验证密钥是否真的本地生成
├─ 信任基础从"技术信任"退化为"信任华为云"
└─ StoryLock 核心价值主张失效
```

### 3.2 风险对比表

| 风险 | 本地存储私钥（StoryLock 设计） | KMS 存储私钥（假设场景） |
|------|------------------------------|------------------------|
| **系统管理员冒充** | 不可能（私钥本地，管理员无法获取） | 可能（管理员可从 KMS 导出） |
| **KMS 被攻破影响** | 仅泄露公钥（无实质影响） | 所有用户私钥泄露（灾难性） |
| **用户信任基础** | 技术信任（密码学原理） | 信任第三方（华为云） |
| **合规性** | 满足"数据本地化"要求 | 违反"敏感数据不出境/不上云" |
| **单点失效** | 无（分散存储） | 有（KMS 成为单点） |

## 四、StoryLock 如何确保 KMS 不保存用户私钥

### 4.1 技术保障

#### 1. 本地生成，不上传

```javascript
// 用户设备本地生成密钥对（示例代码）
const crypto = require('crypto');

// 生成 ECC P-256 密钥对
const { privateKey, publicKey } = crypto.generateKeyPairSync('ec', {
  namedCurve: 'P-256',
  publicKeyEncoding: { type: 'spki', format: 'pem' },
  privateKeyEncoding: { type: 'pkcs8', format: 'pem' }
});

// 私钥：仅存在于本地内存，不上传
// publicKey：上传到 FunctionGraph，存入 KMS
```

#### 2. 仅上传公钥

```javascript
// 注册时仅上传公钥
async function registerDevice(identityId, publicKey) {
  // 上传的仅公钥，私钥从未离开本地
  await functionGraph.register({
    identityId,
    publicKey,  // 仅公钥！
    deviceInfo: getDeviceInfo()
  });
  
  // KMS 中仅存储公钥，作为验证材料
  await kms.storePublicKey({
    keyId: `alias/storylock-user-${identityId}`,
    publicKey
  });
}
```

#### 3. 签名操作本地完成

```javascript
// 用户请求签名（本地使用私钥）
async function signRequest(request) {
  // 从本地 SecretStore 读取私钥
  const privateKey = await localSecretStore.getPrivateKey(identityId);
  
  // 本地签名
  const signature = crypto.sign('SHA256', request, privateKey);
  
  // 发送请求（含签名，不含私钥）
  return {
    ...request,
    signature
  };
}
```

### 4.2 架构保障

```
┌─────────────────────────────────────────┐
│  用户设备（本地）                          │
│  ├─ 密钥生成：本地完成                     │
│  ├─ 私钥存储：本地 SecretStore             │
│  ├─ 签名操作：本地完成                     │
│  └─ 仅上传：公钥                           │
└──────────────────┬──────────────────────┘
                   │ 仅公钥上传
                   ▼
┌─────────────────────────────────────────┐
│  FunctionGraph（第三层）                   │
│  ├─ 接收：公钥 + 签名请求                   │
│  ├─ 转发：签名请求到本地宿主                │
│  └─ 不存储：任何私钥                        │
└──────────────────┬──────────────────────┘
                   │ 公钥存入 KMS
                   ▼
┌─────────────────────────────────────────┐
│  KMS（华为云）                             │
│  ├─ 存储：用户公钥（验证材料）              │
│  ├─ 不存储：用户私钥（设计保证）            │
│  └─ 用途：验证签名                          │
└─────────────────────────────────────────┘
```

### 4.3 可验证性保障

**用户如何验证 KMS 没有保存自己的私钥？**

| 验证方法 | 说明 |
|---------|------|
| **代码审计** | StoryLock 开源，可审计代码确认私钥不上传 |
| **网络抓包** | 注册时抓包分析，确认仅上传公钥 |
| **本地存储检查** | 检查本地 SecretStore，确认私钥存在本地 |
| **KMS 查询** | 通过 API 查询 KMS 中存储的内容，确认无私钥 |
| **离线签名测试** | 断开网络，本地仍能签名，证明私钥本地 |

## 五、KMS 中实际存储的用户相关密钥

### 5.1 明确存储：用户公钥（验证材料）

```json
{
  "keyId": "alias/storylock-user-abc123",
  "keySpec": "ECC_NIST_P256",
  "keyUsage": "VERIFY",           // 仅验证，不签名
  "publicKey": "-----BEGIN PUBLIC KEY-----\nMFkwEwYHKoZIzj0CAQYIKoZIzj0DAQcDQgAE...\n-----END PUBLIC KEY-----",
  "tags": {
    "type": "user-identity",
    "identityId": "user_abc123",
    "purpose": "request-verification"
  }
}
```

**注意**：
- `keyUsage: "VERIFY"` — 仅用于验证，不能签名
- 即使 KMS 被攻破，攻击者只有公钥，无法伪造签名

### 5.2 明确不存储：用户私钥

```json
// 以下内容在 KMS 中不存在
{
  "privateKey": "-----BEGIN PRIVATE KEY-----\nMIGHAgEAMBMGByqGSM49AgEGCCqGSM49AwEHBG0wawIBAQQg...\n-----END PRIVATE KEY-----"
  // ^ 此内容永远不会出现在 KMS 中
}
```

## 六、极端场景分析

### 6.1 场景：华为云要求访问用户私钥

```
假设：华为云政府监管要求提供用户私钥

StoryLock 的应对：
1. 技术上不可能：私钥从未上传到 KMS
2. 即使要求，也无法提供：私钥仅存在于用户本地设备
3. 法律层面：可证明系统设计上无法提供私钥

对比：
- 如果 KMS 存储私钥：必须配合提供
- StoryLock 设计：无法提供（技术上不可能）
```

### 6.2 场景：KMS 被强制关闭

```
假设：华为云停止服务，KMS 不可用

StoryLock 的应对：
1. 用户私钥仍在本地，不受影响
2. 本地授权（第二层）完全独立运行
3. 仅影响远程网关（第三层）的跨设备功能
4. 用户可导出公钥，迁移到其他验证服务

对比：
- 如果 KMS 存储私钥：用户身份丢失，无法恢复
- StoryLock 设计：用户身份不受影响
```

## 七、结论

| 问题 | 答案 |
|------|------|
| **KMS 是否保存用户密钥对？** | **否**。KMS 仅保存用户公钥（验证材料），不保存私钥 |
| **用户私钥存储在哪里？** | 用户本地设备（Windows DPAPI / Android Keystore / macOS Keychain） |
| **如果 KMS 保存私钥会怎样？** | 完全破坏"本地优先"安全模型，系统管理员可冒充用户，信任基础崩塌 |
| **如何确保 KMS 没有私钥？** | 代码开源可审计、网络抓包可验证、本地存储可检查、离线签名可测试 |
| **KMS 中用户公钥的用途？** | 仅用于验证用户签名，不能用于签名（keyUsage: VERIFY） |
| **KMS 被攻破的影响？** | 仅泄露公钥，无实质影响（攻击者无法伪造签名） |

**一句话总结**：

> **StoryLock 设计中，KMS 明确不保存用户私钥。用户的密钥对在本地生成、本地存储、本地使用，仅公钥上传到 KMS 用于验证。即使 KMS 被攻破，攻击者也只有公钥，无法伪造签名，无法冒充用户。这是"本地优先"安全架构的核心保障。**
