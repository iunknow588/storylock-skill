# 易安与 StoryLock 整体技术架构及华为云能力应用说明（修订版 v3）

## 一、整体技术架构介绍

### 1.1 三层架构概览

StoryLock 采用**本地优先的三层隔离架构**，核心理念是**"纯故事授权，无身份验证"**——无论用户身份如何变化，只要记住故事就能打开文件。第三层远程网关仅用于设备管理（验证设备是否被允许远程访问），不验证用户身份。

```
┌─────────────────────────────────────────────────────────────┐
│  第三层：远程网关（Remote Gateway）                           │
│  职责：统一对外入口，请求包装，结果脱敏，设备管理              │
│  接口：requestSignature、requestPasswordFill                 │
│  部署：华为云 FunctionGraph / OBS 静态托管                    │
│  核心原则：不验证"你是谁"，仅验证"设备是否被允许"            │
└──────────────────┬──────────────────────────────────────────┘
                   │  HTTPS / 安全通道
                   ▼
┌─────────────────────────────────────────────────────────────┐
│  第二层：本地授权（Local Authorization）                      │
│  职责：对象强度判断、九宫格验证、短时授权、防重放、审计        │
│  存储：SQLite（8张表）                                       │
│  运行：本地设备（Android / Windows / Linux）                   │
│  核心原则：纯故事授权，知道故事就能解锁，无论你是谁            │
└──────────────────┬──────────────────────────────────────────┘
                   │  本地进程调用
                   ▼
┌─────────────────────────────────────────────────────────────┐
│  第一层：故事处理（Story Processing）                        │
│  职责：故事草稿、润色、题集强度评估、对象加密                  │
│  存储：本地文件系统（加密故事文件、题集主档）                  │
│  运行：本地设备                                               │
│  核心原则：故事是唯一的授权凭证，终身不变                      │
└─────────────────────────────────────────────────────────────┘
```

### 1.2 各层核心职责与设计理念

| 层级 | 核心组件 | 输入 | 输出 | 关键边界 | 设计理念 |
|------|---------|------|------|---------|---------|
| **第一层** | StoryDraftSkill、StoryRefineSkill、StrengthReviewSkill | 用户故事素材 | 24题题集、加密故事文件 | 不读取受保护对象；不签发授权 | 故事是唯一的授权凭证，终身不变 |
| **第二层** | ObjectStrengthPolicySkill、GridChallengeSkill、LocalAuthorizationSkill | 对象类型、用户答案 | challenge结果、短时session | 不返回长期秘密；不直接执行远程请求 | 纯故事授权，知道故事就能解锁，无论你是谁 |
| **第三层** | Web API Handler、EIP-712 Parser、Redaction Middleware、Device Manager | EIP-712结构化请求 | 脱敏结果、审计元数据 | 不持有私钥/密码；不透传敏感字段；不验证用户身份 | 设备管理（非身份验证），仅验证设备是否被允许远程访问 |

### 2.2 各技术的详细说明

#### 2.2.1 FunctionGraph（函数工作流）

**在架构中的位置**：第三层远程网关的核心计算载体

**作用说明**：
- Serverless 运行 `requestSignature`、`requestPasswordFill` 两个接口
- 处理 EIP-712 请求解析、来源校验、脱敏中间件
- 设备管理：验证请求是否来自已注册设备（非身份验证）

**与其他功能模块的协同设计**：
```
外部 Agent 请求 -> APIG（路由/限流）-> FunctionGraph（第三层网关逻辑）
                                              ↓
                                    设备验证（KMS 公钥验证签名）
                                              ↓
                                    本地宿主（Windows/Android）
                                              ↓
                                    第二层九宫格挑战 + 第一层故事验证
                                              ↓
                                    返回脱敏结果至 FunctionGraph -> 响应 Agent
```

**在项目创作开发过程中所发挥的作用**：
- **Serverless 弹性**：按需运行，无需预置服务器，适合 Agent 请求的低频突发特性。在项目开发阶段，开发者无需关心服务器运维，可专注于业务逻辑实现。
- **自动扩缩容**：根据请求量自动扩展，避免资源浪费。在测试和演示阶段，可承受突发流量而不需手动调整。
- **按需计费**：仅在请求处理时计费，降低 OPC（一人制公司）早期运营成本。对于创业初期资源有限的情况，可将资金集中于核心功能开发。
- **安全隔离**：函数实例间隔离，单实例仅处理单个请求，降低横向攻击风险。符合 StoryLock 安全优先的设计理念。

**技术实现**：
- Node.js 22 运行时，与本地 Skill 代码同构，降低维护成本
- 函数内实现 EIP-712 请求解析、来源校验、脱敏中间件、设备验证
- 通过 VPC 网络与本地宿主安全通信（或公网 HTTPS + 设备绑定认证）

#### 2.2.2 OBS（对象存储服务）

**在架构中的位置**：第三层静态资源托管与软件分发

**作用说明**：
- 托管易安官网（yian.cdao.online）静态页面
- 存储 Android APK、Windows MSI/EXE 安装包
- 软件版本管理与灰度发布

**与其他功能模块的协同设计**：
- OBS 托管的静态页面通过 JavaScript 调用 FunctionGraph API，实现"静态站点+动态网关"架构
- 下载页面展示最新版本信息，从 FunctionGraph 动态获取（版本、校验值、发布日期）
- 软件包上传至 OBS 后，FunctionGraph 更新版本元数据，形成闭环

**在项目创作开发过程中所发挥的作用**：
- **静态网站托管**：托管易安官网（yian.cdao.online），提供下载入口、绑定引导、状态展示。开发阶段可快速部署和更新网站内容。
- **软件分发**：存储 Android APK、Windows MSI/EXE 安装包，支持全球 CDN 加速。用户可从就近节点下载，提升体验。
- **版本管理**：通过 OBS 版本控制实现软件包回滚与灰度发布。在迭代开发中，可快速切换版本进行 A/B 测试。
- **低成本高可用**：按存储量与流量计费，适合 OPC 早期阶段。无需预置带宽，随用户增长自动扩展。

#### 2.2.3 APIG（API 网关）

**在架构中的位置**：第三层统一入口

**作用说明**：
- 统一路由：`/api/storylock-gateway` -> FunctionGraph
- 请求限流：防止暴力请求与 DDoS 攻击
- HTTPS 终止：统一证书管理

**与其他功能模块的协同设计**：
- APIG 将请求路由至 FunctionGraph，FunctionGraph 进行设备验证
- APIG 限流策略与第二层失败锁定策略形成双层防护
- APIG 日志可接入 CloudTable，用于请求分析与安全审计

**在项目创作开发过程中所发挥的作用**：
- **统一路由**：将 `/api/storylock-gateway` 路由至 FunctionGraph。开发者无需在函数内处理路由逻辑，聚焦业务实现。
- **请求限流**：防止暴力请求与 DDoS 攻击，保护后端 FunctionGraph。在测试阶段可模拟高并发，验证限流效果。
- **HTTPS 终止**：统一证书管理，本地宿主无需处理 TLS。简化开发配置，降低安全实现难度。

#### 2.2.4 CloudTable / DDS（可选扩展）

**在架构中的位置**：第三层审计日志长期归档

**作用说明**：
- 长期审计日志存储与分析
- 异常访问模式检测
- 合规审计支持

**与其他功能模块的协同设计**：
- 本地宿主通过 FunctionGraph 将审计日志上报至 CloudTable
- CloudTable 分析结果可反馈至第二层，动态调整对象强度策略（如检测到异常则提升挑战阈值）

**在项目创作开发过程中所发挥的作用**：
- **海量日志存储**：本地 SQLite 审计日志定期归档至 CloudTable，释放本地空间。适合长期运行的生产环境。
- **合规分析**：支持按 identityId、capability、时间范围查询审计记录。满足数据合规和审计要求。
- **异常检测**：通过 DDS 聚合分析识别异常访问模式（如高频失败、异地登录）。可主动发现安全威胁。

#### 2.2.5 KMS（密钥管理服务，可选扩展）

**在架构中的位置**：第三层设备管理（非身份验证）

**作用说明**：
- **设备公钥托管**：存储已注册设备的公钥，用于验证远程请求是否来自受信任设备。注意：这不是"身份验证"（验证你是谁），而是"设备管理"（验证这个设备是否被允许远程访问），符合 StoryLock "无身份验证，纯故事授权"的设计理念。
- **配置安全**：加密存储第三层网关的环境变量、配置项（如数据库密码、API 密钥）。防止配置泄露导致的安全风险。
- **与本地密钥分离**：形成"设备公钥云上托管、故事密钥本地持有"的分层安全策略。用户故事密钥（masterSalt/rootKey/workKey）终身不变，与 KMS 无关。

**与其他功能模块的协同设计**：
- KMS 托管的设备公钥用于验证远程请求签名，确保请求来自已注册设备
- 私钥始终本地持有，从未上传云端
- KMS 加密的环境变量在 FunctionGraph 启动时解密，保护配置安全
- **密钥轮换功能禁用**：StoryLock 采用"单故事终身制"，用户故事密钥终身不变。KMS 的密钥轮换功能被禁用，仅保留设备公钥管理和配置加密功能

**在项目创作开发过程中所发挥的作用**：
- **设备公钥管理**：用户更换设备时，新设备公钥上传到 KMS，旧设备公钥自动清理。用户故事终身不变，仅设备标识更新。
- **配置加密**：FunctionGraph 的环境变量（数据库密码、API 密钥）通过 KMS 加密存储，防止配置文件泄露导致的安全风险。
- **合规认证**：华为云 KMS 通过 FIPS 140-2 Level 3 认证，满足等保 2.0/3.0 密钥管理要求。在 OPC 大赛评审中，可向评委展示企业级密钥管理安全性。

## 三、技术实现原理说明

### 3.1 核心加密体系

#### 3.1.1 密钥派生链

```
masterSalt（随机 256-bit，存本地 SecretStore，终身不变）
    ↓ HKDF-SHA256
rootKey（故事根密钥，不直接暴露，终身不变）
    ↓ HKDF-SHA256 + identitySalt
workKey（工作密钥，按对象类型派生，终身不变）
    ↓ HKDF-SHA256 + objectNonce
objectKey（对象密钥，用于 AES-256-GCM 加密，终身不变）
```

**技术原理**：
- **HKDF-SHA256**：基于 HMAC 的密钥派生函数，符合 RFC 5869，确保派生密钥的伪随机性
- **分层派生**：每层密钥仅用于特定目的，降低单点泄露影响面
- **终身不变**：所有密钥终身不变，符合"单故事终身制"设计理念

#### 3.1.2 对称加密规范

| 项目 | 规范 | 说明 |
|------|------|------|
| 算法 | AES-256-GCM | 提供认证加密，防止篡改 |
| 密钥长度 | 256-bit | 符合 NIST 推荐标准 |
| nonce | 96-bit 随机数 | 每次独立生成，同一密钥下不得复用 |
| 身份盐值 | HMAC-SHA256(rootKey, identityId) | 绑定用户身份 |
| 答案摘要 | HMAC-SHA256(identitySalt, normalizedAnswer) | 验证答案但不存储原文 |

#### 3.1.3 九宫格 Challenge 机制

**技术原理**：
1. 从 24 题题集中按难度分层随机抽取 9 题（3 易 + 3 中 + 3 难）
2. 用户基于私密故事记忆回答，答案经规范化后与派生摘要比对
3. 答对阈值达到对象强度策略要求（L1=6, L3=12, L4=22）即通过
4. 每次 challenge 题目组合不同，防止答案组合被枚举

**安全性分析**：
- 24 题题集空间：即使攻击者知道故事，需从 24 题中选 9 题组合，且每题答案空间巨大
- 答案摘要不可逆：HMAC-SHA256 输出无法逆向还原原始答案
- 失败锁定：3 次失败锁定 15 分钟，防止暴力破解

### 3.2 防重放机制

**技术原理**：
- **requestId**：UUID v4，全局唯一，24 小时去重窗口
- **nonce**：uint256 递增随机数，nonce_store 记录已使用 nonce
- **expiry**：请求过期时间戳，默认 5 分钟，过期拒绝
- **三重校验**：任一条件不满足即返回 SLG-013（REPLAY_DETECTED）

### 3.3 递归脱敏机制

**技术原理**：
- 定义 10 类高敏字段（私钥、密码、答案、session 密钥等）
- 三层间传输时按脱敏级别处理：
  - **none**：内部层间，不脱敏
  - **partial**：第二层→第三层，遮盖高敏字段，保留结构
  - **full**：第三层→外部，仅返回状态/摘要，清空所有值

### 3.4 设备管理机制（第三层）

**技术原理**：
- 设备注册：本地设备生成密钥对，公钥上传到 KMS，私钥本地存储
- 设备验证：远程请求携带设备签名，FunctionGraph 从 KMS 获取公钥验证
- 设备更新：用户更换设备时，新设备公钥替换旧设备公钥（24 小时兼容期）
- 设备撤销：设备丢失时，管理员禁用该设备公钥

**关键设计**：
- 不是验证"你是谁"（无身份验证），而是验证"这个设备是否被允许"（设备管理）
- 设备验证失败 → 拒绝远程访问，但本地访问不受影响（纯故事授权）
- 用户故事终身不变，设备标识可更新

| **远期** | 全栈隐私计算 | MPC/HE 实现数据不可见、逻辑可验证 | 6-12 个月 |

## 五、总结

StoryLock 的技术架构以**"纯故事授权，无身份验证"**为核心设计理念：

1. **第一层/第二层**：纯故事授权，知道故事就能解锁，无论你是谁。所有密钥终身不变，符合"单故事终身制"。
2. **第三层**：引入设备管理（非身份验证），验证远程请求是否来自已注册设备，保护故事授权不被滥用。
3. **华为云技术**：FunctionGraph、OBS、APIG 在第三层提供弹性、低成本的 Serverless 运行环境；KMS 用于设备公钥管理和配置加密，密钥轮换功能禁用。

技术路线清晰，所选工具均为成熟标准（AES-256-GCM、HKDF-SHA256、EIP-712、SQLite），无自研算法风险。主要待验证项为 Android 宿主真机闭环与华为云实际部署性能调优。

**一句话总结**：StoryLock 以"纯故事授权"为魂，以成熟密码学标准为基，以华为云 Serverless 为翼，构建本地优先的 Agent 安全授权体系——知道故事就能打开，无论你是谁。
