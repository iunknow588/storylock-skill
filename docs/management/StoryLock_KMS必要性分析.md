# StoryLock 为什么需要 KMS（密钥管理服务）

## 一、核心结论

**KMS 不是"锦上添花"，而是第三层安全架构的必需品。**

| 维度 | 无 KMS | 有 KMS |
|------|--------|--------|
| **配置安全** | 环境变量明文存储，泄露即暴露 | 环境变量加密存储，泄露无影响 |
| **密钥轮换** | 手动修改代码，重启服务 | 自动轮换，零停机 |
| **合规审计** | 无法证明密钥管理合规 | 提供完整审计日志，满足等保要求 |
| **运维成本** | 高（人工管理密钥） | 低（托管服务自动处理） |
| **信任基础** | 依赖运维人员不出错 | 依赖硬件安全模块（HSM），技术信任 |

**一句话：KMS 让第三层从"人管密钥"升级为"技术管密钥"，符合 StoryLock "信任技术原理"的设计哲学。**

---

## 二、具体原因分析

### 2.1 原因一：保护配置项安全（核心原因）

#### 问题场景

FunctionGraph 需要访问以下敏感配置：

```yaml
# 环境变量（无 KMS 时明文存储）
DATABASE_PASSWORD: "MyP@ssw0rd123"  # 数据库密码
API_SECRET_KEY: "sk_live_abc123"     # 第三方 API 密钥
EIP712_PRIVATE_KEY: "0xabc..."        # EIP-712 签名私钥（错误做法！）
```

**风险**：
- 配置泄露 = 所有密钥泄露
- 员工离职、代码泄露、日志暴露都可能导致配置外泄
- 一旦泄露，需紧急修改所有配置，重启服务

#### KMS 解决方案

```yaml
# 环境变量（有 KMS 时加密存储）
DATABASE_PASSWORD: "AQICAH..."  # KMS 加密后的密文
API_SECRET_KEY: "AQICAH..."     # KMS 加密后的密文
KMS_KEY_ID: "alias/storylock"   # 仅存储 KMS 密钥 ID
```

**工作流程**：
```
FunctionGraph 启动
  ↓ 读取环境变量（全是密文）
  ↓ 调用 KMS API 解密
  ↓ KMS 返回明文（仅存在于内存）
  ↓ 函数运行期间使用明文
  ↓ 函数销毁，内存清零，明文消失
```

**安全优势**：
- 配置文件中只有密文，即使泄露也无法使用
- 明文仅在内存中存在，函数销毁即消失
- 无需在代码中硬编码任何密钥

---

### 2.2 原因二：密钥自动轮换（运维原因）

#### 问题场景

**无 KMS 时**：
```
密钥泄露或到期
  ↓ 手动修改配置文件
  ↓ 重新打包部署 FunctionGraph
  ↓ 重启服务（导致短暂中断）
  ↓ 更新所有引用该密钥的代码
  ↓ 验证新密钥是否正常工作
```

**耗时**：30 分钟 - 2 小时，期间服务可能中断

#### KMS 解决方案

```
密钥到期或策略触发轮换
  ↓ KMS 自动生成新密钥
  ↓ 旧密钥标记为"待禁用"（保留 24 小时兼容期）
  ↓ 新密钥立即生效，无需重启服务
  ↓ 旧密钥自动清理
```

**耗时**：0 分钟（全自动），服务零中断

**StoryLock 场景**：
- 定期轮换 EIP-712 domain 公钥对应的私钥（如果云端持有）
- 轮换数据库加密密钥
- 轮换 API 密钥

---

### 2.3 原因三：合规与审计（商业原因）

#### 问题场景

OPC 大赛项目需要展示：
- 密钥管理符合安全标准
- 有完整的密钥使用审计日志
- 能证明"谁、何时、用了什么密钥"

**无 KMS 时**：
- 自行实现审计日志，成本高
- 无法提供第三方认证的合规证明
- 评委质疑密钥管理安全性

#### KMS 解决方案

华为云 KMS 提供：

| 合规能力 | 说明 |
|---------|------|
| **完整审计日志** | 记录每次密钥创建、使用、删除、轮换操作 |
| **操作者身份** | 记录 IAM 用户/角色，追溯责任人 |
| **时间戳** | 精确到毫秒的密钥使用时间 |
| **操作类型** | 加密、解密、签名、验证等 |
| **等保合规** | 满足等级保护 2.0/3.0 密钥管理要求 |
| **国密支持** | 支持 SM2/SM3/SM4 国产密码算法 |

**StoryLock 价值**：
- 向评委展示："我们使用企业级密钥管理服务，非自建"
- 向用户展示："密钥管理通过华为云 HSM 硬件保护"
- 向审计展示："完整密钥使用日志，可追溯"

### 2.4 原因四：硬件级安全（技术原因）

#### 问题场景

**软件加密 vs 硬件加密**：

| 维度 | 软件加密（无 KMS） | 硬件加密（KMS + HSM） |
|------|------------------|----------------------|
| **密钥存储** | 内存/磁盘，可被提取 | 专用硬件芯片，无法提取 |
| **抗攻击** | 易受内存 dump、侧信道攻击 | 物理隔离，抗侧信道攻击 |
| **合规认证** | 无 | FIPS 140-2 Level 3、国密认证 |
| **成本** | 低（软件实现） | 高（硬件成本） |

**KMS 优势**：
- 华为云 KMS 后端使用 HSM（硬件安全模块）
- 密钥在 HSM 内生成、存储、使用，永不离开硬件
- 即使华为云运维人员也无法获取密钥明文

**StoryLock 价值**：
- 第三层密钥（DEK）受 HSM 保护，安全级别最高
- 即使 FunctionGraph 被攻破，攻击者也无法获取 KMS 中的密钥
- 符合"不信任任何第三方"原则：连华为云自己也无法获取密钥

---

### 2.5 原因五：多环境一致性（工程原因）

#### 问题场景

StoryLock 需要多个环境：

| 环境 | 用途 | 配置差异 |
|------|------|---------|
| **开发环境** | 本地调试 | 测试数据库、测试 API |
| **测试环境** | CI/CD 验证 | 预发布数据库、模拟 API |
| **生产环境** | 正式服务 | 生产数据库、真实 API |

**无 KMS 时**：
- 每个环境单独管理配置文件
- 容易混淆（生产密钥误用于开发）
- 配置分散，难以统一管控

#### KMS 解决方案

```
同一 KMS 密钥，不同环境不同权限：
  ├─ 开发环境：只能解密 dev-前缀的密文
  ├─ 测试环境：只能解密 test-前缀的密文
  └─ 生产环境：只能解密 prod-前缀的密文
```

**优势**：
- 统一密钥管理入口
- 环境隔离，防止误操作
- 权限精细化控制（IAM 角色绑定）

## 三、KMS 在 StoryLock 中的具体应用场景

### 3.1 场景一：FunctionGraph 环境变量加密

```yaml
# 无 KMS：明文配置（危险）
DATABASE_URL: "postgresql://user:password@host:5432/db"
API_KEY: "sk_live_abc123"

# 有 KMS：加密配置（安全）
DATABASE_URL: "AQICAHg...encrypted..."  # KMS 加密
API_KEY: "AQICAHg...encrypted..."         # KMS 加密
KMS_KEY_ID: "alias/storylock-prod"       # 密钥标识
```

### 3.2 场景二：EIP-712 公钥托管

```javascript
// FunctionGraph 启动时从 KMS 获取公钥
const publicKey = await kms.getPublicKey({
  keyId: 'alias/storylock-eip712'
});

// 使用公钥验证 Agent 请求签名
const isValid = verifyEIP712Signature(request, publicKey);
```

**优势**：
- 公钥变更时无需修改代码
- 从 KMS 动态获取，实时生效
- 公钥轮换不影响服务运行

### 3.3 场景三：审计日志加密

```javascript
// 敏感审计日志写入前加密
const encryptedLog = await kms.encrypt({
  keyId: 'alias/storylock-audit',
  plaintext: JSON.stringify(auditLog)
});

// 写入 CloudTable
await cloudTable.put({
  partitionKey: identityId,
  sortKey: timestamp,
  logData: encryptedLog.ciphertext
});
```

**优势**：
- 审计日志加密存储，即使数据库泄露也无法读取
- 需要审计时，通过 KMS 解密，记录解密操作
- 符合 GDPR、等保等数据保护法规

## 五、无 KMS 的替代方案及问题

### 5.1 替代方案一：环境变量明文存储

```yaml
# 最简方案，但最危险
DATABASE_PASSWORD: "MyP@ssw0rd123"
```

**问题**：
- 配置文件泄露 = 所有密钥泄露
- 无法审计谁使用了密钥
- 密钥轮换需重启服务

### 5.2 替代方案二：自建加密服务

```javascript
// 自己实现加密服务
const crypto = require('crypto');
const masterKey = process.env.MASTER_KEY; // 根密钥还是明文存储

function encrypt(data) {
  const iv = crypto.randomBytes(16);
  const cipher = crypto.createCipheriv('aes-256-gcm', masterKey, iv);
  // ...
}
```

**问题**：
- 根密钥（MASTER_KEY）仍需明文存储，问题未解决
- 自行实现加密容易出错（IV 重用、填充攻击等）
- 无审计日志，无合规证明
- 运维成本高（需专人维护）

### 5.3 替代方案三：HashiCorp Vault

```bash
# 第三方密钥管理服务
vault kv put secret/storylock database_password="MyP@ssw0rd123"
```

**问题**：
- 需额外部署 Vault 服务，增加复杂度
- 非华为云生态，与 FunctionGraph 集成需额外开发
- 额外成本（服务器 + 运维）

## 六、结论

| 问题 | 答案 |
|------|------|
| **为什么需要 KMS？** | 保护配置安全、自动密钥轮换、合规审计、硬件级安全、多环境管理 |
| **KMS 管理什么？** | 第三层配置加密密钥（DEK）、EIP-712 公钥、证书、审计日志加密密钥 |
| **KMS 不管理什么？** | 第二层核心密钥（masterSalt、rootKey、workKey、私钥） |
| **不用 KMS 会怎样？** | 配置明文存储、密钥轮换困难、无合规证明、运维成本高 |
| **KMS 成本高吗？** | 月费用 ¥3-6，可忽略不计 |
| **KMS 与 StoryLock 理念冲突吗？** | 不冲突。KMS 管理"可公开/配置级"材料，第二层管理"核心敏感"密钥，分层明确 |

**一句话总结**：KMS 是第三层从"人管密钥"到"技术管密钥"的关键升级，月成本仅 ¥3-6，却带来配置安全、自动轮换、合规审计、硬件级保护四大价值，是 StoryLock 安全架构的必要组件。
