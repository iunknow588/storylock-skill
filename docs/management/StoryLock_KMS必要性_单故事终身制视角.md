# StoryLock 中 KMS 的必要性再分析 —— 从"单故事终身制"视角

## 一、核心问题

**用户观点**：
- 轮换密钥是系统部**不提倡**的技术特征
- 希望"逐渐增加故事的内涵，用一个故事解锁全部待保管的信息"
- 质疑：KMS 的必要性在哪里？

**核心矛盾**：
- 用户希望**简化架构**（一个故事解锁全部，终身不变）
- KMS 看起来是**增加复杂度**（额外的密钥管理，且与"轮换"相关）

**回答**：KMS 不是与"单故事终身制"对立，而是**支持"单故事终身制"的基础设施**。StoryLock 可以**禁用 KMS 的密钥轮换功能**，仅保留身份验证、配置加密、审计日志功能。

---

## 二、"单故事终身制"的核心诉求

### 2.1 用户的核心诉求

```
用户期望：
┌─────────────────────────────────────────┐
│  一个故事（九宫格题集）                    │
│  ├─ 记住故事 → 终身有效                   │
│  ├─ 不需要定期更换故事                    │
│  ├─ 不需要定期更换密钥                    │
│  └─ 故事是"终身密码"，不是"定期密码"      │
└─────────────────────────────────────────┘
```

### 2.2 "单故事终身制"的密钥设计

| 密钥 | 生命周期 | 存储位置 | 说明 |
|------|---------|---------|------|
| **masterSalt** | 终身不变 | 本地 SecretStore | 故事根盐值，创建时生成，永不更换 |
| **rootKey** | 终身不变 | 本地派生，不存储 | 从 masterSalt + 题集答案派生，终身不变 |
| **workKey** | 终身不变 | 本地派生，不存储 | 从 rootKey + 对象类型派生，终身不变 |
| **objectKey** | 终身不变 | 本地派生，不存储 | 从 workKey + 对象 ID 派生，终身不变 |
| **用户私钥** | 终身不变 | 本地 SecretStore | 身份签名私钥，创建时生成，永不更换 |
| **用户公钥** | 终身不变 | KMS | 身份验证公钥，创建时上传，永不更换 |

**关键**：所有用户密钥都是**终身不变**的，与 KMS 的"密钥轮换"功能无关。

## 三、KMS 在"单故事终身制"中的角色

### 3.1 KMS 功能拆分

| KMS 功能 | StoryLock 是否使用 | 原因 |
|---------|------------------|------|
| **用户公钥存储** | ✅ 使用 | 身份验证需要，终身不变 |
| **配置加密** | ✅ 使用 | 保护数据库密码等，与故事无关 |
| **审计日志加密** | ✅ 使用 | 保护操作记录，与故事无关 |
| **密钥轮换** | ❌ 禁用 | 与"单故事终身制"冲突 |
| **自动密钥更新** | ❌ 禁用 | 与"单故事终身制"冲突 |

### 3.2 KMS 在"单故事终身制"中的具体作用

```
用户注册（终身一次）：
├─ 用户在本地设备生成密钥对（终身不变）
├─ 用户记住故事（九宫格题集，终身不变）
├─ 公钥上传到 FunctionGraph
├─ FunctionGraph 将公钥存入 KMS（终身不变）
└─ 用户无需记住任何密码，只需记住一个故事（终身）

用户使用（日常）：
├─ 用户需要解锁密码库
├─ 本地九宫格挑战（验证故事记忆）
├─ 本地授权通过，生成 session 令牌
├─ 请求发送到 FunctionGraph
├─ FunctionGraph 从 KMS 获取用户公钥（终身不变）
├─ 验证用户签名（证明是本人）
├─ 转发到本地宿主，返回脱敏结果
└─ 用户用一个故事，解锁全部功能

用户更换设备（终身不变的故事）：
├─ 新设备下载 StoryLock 应用
├─ 用户输入故事（九宫格题集，同一个故事）
├─ 新设备重新派生 rootKey（与旧设备相同）
├─ 新设备生成新密钥对（新设备的新密钥）
├─ 新公钥上传到 KMS，替换旧公钥（设备级更新，非故事级）
├─ 旧设备公钥标记为"待禁用"（24 小时兼容期）
├─ 旧设备公钥 7 天后物理销毁
└─ 用户仍然只需记住一个故事（终身不变）
```

**关键**：
- 故事终身不变（masterSalt、rootKey、workKey、objectKey 终身不变）
- 用户公钥在 KMS 中终身不变（除非用户更换设备）
- KMS 仅管理公钥，不管理故事密钥
- 禁用 KMS 密钥轮换功能，保持"单故事终身制"

## 四、没有 KMS 的"单故事终身制"问题

### 4.1 问题一：新用户注册

```
有 KMS：
1. 用户本地生成密钥对
2. 公钥通过 API 上传到 KMS
3. 立即生效，无需重启
4. 用户故事终身不变

没有 KMS：
1. 用户本地生成密钥对
2. 公钥如何上传到系统？
   ├─ 方案 A：硬编码到代码 → 每用户需修改代码、重新部署
   ├─ 方案 B：配置文件 → 每用户需修改配置、重启服务
   ├─ 方案 C：本地数据库 → 无法跨设备同步
   └─ 方案 D：用户手动发送给管理员 → 用户体验差
3. 用户故事终身不变，但系统注册流程复杂
```

### 4.2 问题二：用户更换设备

```
有 KMS：
1. 用户在新设备输入故事（同一个故事）
2. 新设备生成新密钥对
3. 新公钥通过 API 上传到 KMS，自动替换旧公钥
4. 旧公钥自动清理（24 小时兼容期）
5. 用户故事终身不变，设备更换无感知

没有 KMS：
1. 用户在新设备输入故事（同一个故事）
2. 新设备生成新密钥对
3. 新公钥如何上传？
   ├─ 方案 A：硬编码 → 修改代码、重新部署（服务中断）
   ├─ 方案 B：配置文件 → 手动更新、重启服务（服务中断）
   ├─ 方案 C：本地数据库 → 无法跨设备同步
   └─ 方案 D：用户手动发送 → 用户体验差
4. 旧公钥如何清理？
   ├─ 手动清理 → 容易遗漏，安全风险
   └─ 不清理 → 旧设备仍可验证，安全风险
5. 用户故事终身不变，但设备更换流程复杂
```

### 4.3 问题三：多设备支持

```
有 KMS：
1. 用户有 3 台设备（手机、电脑、平板）
2. 每台设备生成独立密钥对
3. 所有公钥上传到 KMS，关联同一用户
4. 任一设备均可验证身份
5. 用户故事终身不变，多设备无感知

没有 KMS：
1. 用户有 3 台设备
2. 每台设备生成独立密钥对
3. 公钥如何管理？
   ├─ 硬编码 → 代码膨胀，维护困难
   ├─ 配置文件 → 多环境同步困难
   └─ 本地数据库 → 无法跨设备同步
4. 设备公钥关联困难
5. 用户故事终身不变，但多设备支持复杂
```

## 五、KMS 支持"单故事终身制"的机制

### 5.1 禁用密钥轮换

```javascript
// KMS 配置：禁用自动轮换
const kmsConfig = {
  keyId: 'alias/storylock-user-abc123',
  keySpec: 'ECC_NIST_P256',
  keyUsage: 'VERIFY',
  
  // 禁用自动轮换
  enableKeyRotation: false,
  
  // 禁用自动更新
  enableAutomaticKeyGeneration: false,
  
  // 手动更新：仅用户更换设备时触发
  manualUpdateOnly: true
};
```

### 5.2 设备级公钥更新（非故事级）

```javascript
// 用户更换设备时，手动更新公钥
async function updateDevicePublicKey(identityId, newPublicKey) {
  // 1. 新公钥上传到 KMS
  await kms.uploadPublicKey({
    keyId: `alias/storylock-user-${identityId}`,
    publicKey: newPublicKey,
    tags: {
      deviceId: 'new-device-123',
      updatedAt: new Date().toISOString()
    }
  });
  
  // 2. 旧公钥标记为"待禁用"（24 小时兼容期）
  await kms.disableKey({
    keyId: `alias/storylock-user-${identityId}`,
    version: 'old-version',
    disableAfter: '24h'
  });
  
  // 3. 旧公钥 7 天后物理销毁
  await kms.scheduleDeletion({
    keyId: `alias/storylock-user-${identityId}`,
    version: 'old-version',
    deleteAfter: '7d'
  });
}
```

**关键**：
- 公钥更新是**设备级**的（用户更换设备）
- 不是**故事级**的（故事终身不变）
- 不是**定期轮换**的（禁用自动轮换）

### 5.3 用户故事终身不变的保证

```javascript
// 用户故事密钥：终身不变，与 KMS 无关
const storyKeys = {
  masterSalt: 'local-storage-lifetime',     // 本地存储，终身不变
  rootKey: 'derived-from-story-lifetime',   // 从故事派生，终身不变
  workKey: 'derived-from-rootKey-lifetime',   // 从 rootKey 派生，终身不变
  objectKey: 'derived-from-workKey-lifetime'  // 从 workKey 派生，终身不变
};

// KMS 仅管理公钥：与故事密钥无关
const kmsKeys = {
  userPublicKey: 'stored-in-kms-lifetime',  // 存储在 KMS，终身不变（除非换设备）
  purpose: 'identity-verification-only'      // 仅用于身份验证
};
```

## 六、结论

| 问题 | 答案 |
|------|------|
| **KMS 与"单故事终身制"冲突吗？** | **不冲突**。KMS 不管理故事密钥，只管理身份验证公钥。禁用 KMS 密钥轮换功能，保持"单故事终身制" |
| **KMS 的密钥轮换功能需要吗？** | **不需要**。StoryLock 禁用 KMS 密钥轮换功能，用户公钥终身不变（除非更换设备） |
| **没有 KMS 能实现"单故事终身制"吗？** | **能，但受限**。本地功能完全可用，但跨设备、远程验证、公钥管理困难 |
| **KMS 增加了复杂度吗？** | **增加了系统复杂度，但降低了用户复杂度**。用户无需关心公钥管理，系统自动处理 |
| **KMS 支持"故事内涵"增长吗？** | **支持**。无论新增多少对象类型，KMS 统一身份验证，用户始终只需记住一个故事 |
| **KMS 的必要性？** | **必要**。即使禁用密钥轮换，KMS 仍是用户身份验证、跨设备支持、配置保护的基础设施 |

**一句话总结**：

> **KMS 不是与"单故事终身制"对立的工具。StoryLock 禁用 KMS 的密钥轮换功能，保留其身份验证、配置加密、审计日志功能。用户只需记住一个故事，终身不变；KMS 管理用户公钥，终身不变，仅用于验证身份。设备更换时公钥更新，但故事本身终身不变。**
