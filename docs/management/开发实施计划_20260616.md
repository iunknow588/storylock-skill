# StoryLock 开发实施计划

| 项目 | 内容 |
| --- | --- |
| 日期 | 2026-06-16 |
| 适用阶段 | 当前阶段本地优先实现 |
| 目标 | 从设计文档进入最小可运行开发阶段 |

## 目的

本文档用于把现有设计文档和技术评审意见转成可执行的开发顺序。

## 总体策略

当前阶段采用：

1. 先第二层，后第一层，最后第三层
2. 先安全基础设施，后能力包装
3. 先本地链路跑通，后远程接入封装

原因：

1. 第二层是安全边界核心
2. 第一层和第三层都依赖第二层的稳定接口
3. 先把 challenge / session / secret access 做稳，比先做界面和远程包装更重要

## 开发里程碑

### M1：安全基础设施

目标：

1. 建立 `SecretStore`
2. 建立 SQLite 基础表
3. 建立加密与摘要工具
4. 冻结单故事终身制下的分层轮换规则
5. 冻结根级重建与常规优化的判定边界

交付物：

1. `shared/secret-store`
2. `shared/sqlite`
3. `shared/crypto`
4. 单故事终身制轮换判定矩阵
5. 开发模式安全边界说明

完成标准：

1. 能写入和读取 `masterSalt`
2. 能对题集主档做 AES-256-GCM 加密与解密
3. 能生成 `identitySalt`
4. 能计算 challenge answer digest
5. 明确区分“题集轮换 / 派生上下文轮换 / 对象封装轮换 / 根级重建”
6. 明确根级重建触发条件、预学习要求与回退机制
7. 明确开发模式的启用条件、确认方式与降级路径

### M2：第二层本地访问控制

目标：

1. 实现 challenge 状态机
2. 实现 session 管理
3. 实现 requestId / nonce / expiry 校验
4. 冻结失败计数与锁定策略
5. 支持双版本题集校验与对象兼容窗口

交付物：

1. `storylock-local-story-access-skill`
2. 本地 challenge/session store
3. 失败计数与锁定策略实现
4. 题集版本兼容与对象迁移规则

完成标准：

1. 能创建 challenge
2. 能提交答案并校验
3. 能签发 session
4. 能控制读预算和 TTL
5. 能拒绝重放请求
6. 失败计数按 `identityId + 时间窗口` 执行
7. 达到阈值后能返回锁定错误与 `retryAfter`
8. 旧题集摘要与旧对象密文可按兼容窗口平滑迁移
9. Session 预算扣减采用原子事务，不可被并发绕过

### M3：第一层故事处理

目标：

1. 接上本地故事处理 Skill
2. 打通“读取 -> 处理 -> 写回”最小链路

交付物：

1. `storylock-local-story-processing-skill`
2. 与第二层的处理接口

完成标准：

1. 能从第二层接收已批准的故事对象
2. 能返回结构化处理结果
3. 能把处理结果交回第二层决定是否写回

### M4：第三层远程网关

目标：

1. 实现最小网关包装
2. 打通第三层 -> 第二层 的远程结构化调用

交付物：

1. `storylock-remote-gateway-skill`
2. `requestStoryRead`
3. `requestStoryWrite`
4. `requestChallengeSign`

完成标准：

1. 能构造统一请求
2. 能返回统一响应
3. 能传递 `auditMeta`
4. 不泄露 challenge answers、session material、signing key material

### M5：联调与验证

目标：

1. 全链路验证
2. 错误码验证
3. 文档与实现对齐

完成标准：

1. 覆盖成功链路
2. 覆盖失败链路
3. 覆盖重放拒绝
4. 覆盖预算耗尽
5. 覆盖脱敏输出

## 当前阶段验收标准

当前阶段不以“功能很多”为目标，而以“边界正确”验收。

至少应满足：

1. challenge answers 不持久化
2. session 预算不可绕过
3. requestId / nonce 重放会被拒绝
4. 第三层不直接读取 secret object
5. 第三层不直接持有私钥
6. 单故事模式下常规优化不触发故事根更换
7. 仅根级重建触发预学习与旧 session 撤销
8. 旧题集摘要与旧对象密文有明确兼容窗口
9. 开发模式下不得明文持久化长期密钥

## 结论

当前文档体系已经足够支撑开发启动。

最合理的下一步不是继续扩展概念，而是：

1. 先在 M1 冻结单故事终身制的轮换边界与开发模式边界
2. 再在 M2 做稳第二层本地访问控制
3. 再接第一层处理
4. 最后接第三层远程包装
