# StoryLock Skill 定位与边界分析

## 一、文档定位评估

### 1.1 定位准确性：✅ 高度准确

该文档明确定义了 StoryLock Skill 系统的三层架构：
- **Skill 层**：负责自描述、输入输出契约、执行指导
- **Host 层**：负责挑战创建、答案提交、密钥读取（实际敏感边界）
- **Storage 层**：负责加密对象存储

**核心洞察正确**：将敏感操作（挑战-应答、密钥访问）明确划归 Host 层，Skill 层不触碰安全边界。

### 1.2 关键设计原则评估

| 原则 | 评估 | 说明 |
|------|------|------|
| "按能力定义 Skill，不按存储位置" | ✅ 正确 | 避免将存储实现细节泄露到 Skill 接口 |
| "Skill 契约与安全边界分离" | ✅ 正确 | Skill 声明能力，Host 控制访问 |
| "敏感秘密访问由 Host 控制" | ✅ 正确 | 符合最小权限原则 |
| "远程安全文本处理与本地秘密处理分离" | ✅ 正确 | 支持混合架构 |
| "编排作为 Skill 之上的层" | ✅ 正确 | 避免 Skill 过度耦合 |

## 二、Skill 分类分析

### 2.1 远程安全 Skill（Remote-safe Skills）

| Skill | 敏感性 | 运行环境 | 评估 |
|-------|--------|----------|------|
| StoryDraftAssistSkill | 低至中 | 通用提示可远程；私人记忆需本地 | ⚠️ 需明确输入脱敏要求 |
| StoryRefineAssistSkill | 低至中 | 脱敏文本可远程；私人草稿需本地 | ⚠️ 需明确输入脱敏要求 |
| StrengthReviewSkill | 中 | 本地或远程，取决于数据敏感度 | ✅ 可配置运行环境 |

**关键问题**："低至中"的敏感性评估是否准确？
- 如果用户输入包含个人经历、家庭信息、财务细节，敏感性应为**高**
- 建议增加**输入敏感性检测**机制，自动路由到本地执行

### 2.2 本地独占 Skill（Local-only Skills）

| Skill | 敏感性 | 运行环境 | 评估 |
|-------|--------|----------|------|
| LoginAuthorizationSkill | **高** | 本地独占 | ✅ 正确 |
| LocalPasswordFillSkill | **高** | 本地独占 | ✅ 正确 |
| SigningAuthorizationSkill | **高** | 本地独占 | ✅ 正确 |
| ChallengeSigningAuthorizationSkill | **高** | 本地独占 | ✅ 正确 |

**安全保证**：这些 Skill 直接依赖挑战流程、会话流程、秘密对象读取或密钥处理，必须留在本地。

### 2.3 混合编排 Skill（Hybrid Orchestration Skills）

| Skill | 敏感性 | 运行环境 | 评估 |
|-------|--------|----------|------|
| VideoPublishAgentDemo | 混合 | 混合架构 | ⚠️ 需明确编排协议 |

**关键问题**：远程 Agent 与本地 Host 之间的**结构化交接契约**尚未定义。

## 三、边界模型评估

### 3.1 三层架构的合理性

```
┌─────────────────────────────────────┐
│           编排层（Orchestration）      │  ← 远程 Agent 负责
│    - 路由用户意图                      │
│    - 选择正确 Skill                   │
│    - 准备结构化请求                    │
│    - 包装本地结果为最终输出             │
└──────────────────┬──────────────────┘
                   │
┌──────────────────┴──────────────────┐
│           Skill 层                    │  ← 能力声明与执行指导
│    - 自描述（skillId）                │
│    - 输入输出契约                     │
│    - 执行指导                         │
│    - 编排友好的结构化结果              │
└──────────────────┬──────────────────┘
                   │
┌──────────────────┴──────────────────┐
│           Host 层                     │  ← 实际敏感执行边界
│    - createChallenge()                │
│    - submitChallengeAnswers()         │
│    - readSecretObject()             │
│    - 签名器注入与密钥处理              │
└──────────────────┬──────────────────┘
                   │
┌──────────────────┴──────────────────┐
│           Storage 层                  │  ← 加密对象存储
│    - 持有加密对象                     │
│    - 提供位置与访问控制                │
│    - 支持本地、云端或合约存储          │
└─────────────────────────────────────┘
```

**评估**：架构清晰，分层合理。但存在潜在问题：

### 3.2 潜在问题识别

| 问题 | 风险等级 | 说明 | 建议 |
|------|---------|------|------|
| **Skill 层能否直接调用 Host 层？** | 中 | 文档未明确 Skill 到 Host 的调用路径 | 明确 Skill 通过依赖注入获取 Host 接口，禁止直接实例化 |
| **Host 层是否验证 Skill 身份？** | 高 | 恶意 Skill 可能伪装成合法 Skill 调用 Host | 增加 Skill 签名验证机制 |
| **Storage 层位置是否影响安全模型？** | 中 | 本地/云端/合约存储的安全假设不同 | 在 SKILL.md 中声明支持的存储类型及对应安全级别 |
| **编排层如何知道哪些步骤需本地执行？** | 高 | 远程 Agent 可能误将敏感操作路由到远程 | 在 Capability Index 中明确标注每个 Skill 的运行环境要求 |

## 四、功能组装评估

### 4.1 故事草稿工作流（Feature 1）

```
输入：提示上下文
  │
  ▼
┌─────────────────┐
│ StoryDraftAssist │ ← 远程或本地，取决于输入敏感性
│   Skill          │
└────────┬────────┘
         │ 故事草稿
         ▼
┌─────────────────┐
│ StoryRefineAssist│ ← 远程或本地，取决于草稿内容
│   Skill          │
└────────┬────────┘
         │ 精炼故事
         ▼
┌─────────────────┐
│ 可选：本地隐私审查 │ ← 本地独占，确保输出安全
└─────────────────┘
```

**评估**：流程合理，但缺少**输入敏感性自动检测**机制。建议增加：
- 输入扫描器：检测是否包含个人标识信息（PII）
- 自动路由：检测到 PII 时强制本地执行

### 4.2 密码填充工作流（Feature 3）

```
输入：站点绑定信息
  │
  ▼
┌─────────────────┐
│ LoginAuthorization│ ← 本地独占
│   Skill          │
│  - 创建挑战       │
│  - 提交答案       │
│  - 获取会话       │
└────────┬────────┘
         │ 授权结果
         ▼
┌─────────────────┐
│ LocalPasswordFill │ ← 本地独占
│   Skill          │
│  - 读取授权秘密   │
│  - 包装登录字段   │
└─────────────────┘
```

**评估**：安全。两个 Skill 均为本地独占，无远程泄露风险。

### 4.3 多步 Agent 工作流（Feature 5）

```
输入：发布任务
  │
  ▼
┌─────────────────┐     ┌─────────────────┐
│  远程 Agent      │────▶│ LocalPasswordFill│ ← 本地执行
│  - 选择工作流    │     │   Skill          │
│  - 编排步骤      │     └────────┬────────┘
│  - 包装输出      │              │ 登录字段
│                 │◀───────────────┘
│                 │
│                 │────▶┌─────────────────┐
│                 │     │ ChallengeSigning │ ← 本地执行
│                 │     │   Skill          │
│                 │     └────────┬────────┘
│                 │◀─────────────┘ 签名结果
│                 │
│                 │────▶┌─────────────────┐
│                 │     │ VideoPublishAgent│ ← 混合执行
│                 │     │   Demo           │
│                 │     └────────┬────────┘
│                 │◀─────────────┘ 发布计划
└────────┬────────┘
         │ 最终结果
         ▼
       用户
```

**评估**：架构正确，但**远程 Agent 与本地 Host 之间的交接契约**需要明确定义：
- 数据格式：JSON Schema
- 认证机制：Host 如何验证远程 Agent 的合法性？
- 错误处理：本地执行失败时如何反馈？

## 五、实用边界规则评估

文档提出的三条规则：

1. **"如果需要挑战答案、会话材料、秘密读取或签名密钥访问 → 本地独占"**
   - ✅ 正确且可操作

2. **"如果只转换已安全文本或分析非秘密结构化数据 → 可远程安全"**
   - ⚠️ 需要补充："已安全文本"的定义和验证机制

3. **"如果混合两者 → 拆分为远程编排部分和本地执行部分"**
   - ✅ 正确，但需要明确**拆分点**和**交接协议**

**建议增加规则 4**：
> "如果输入包含个人标识信息（PII）或受保护的记忆材料 → 强制本地执行，无论功能类型"
