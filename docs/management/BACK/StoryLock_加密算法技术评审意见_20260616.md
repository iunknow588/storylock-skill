## 一、核心加密算法评估

### 1.1 根密钥派生：Argon2id ✅ 通过

| 参数 | 文档定义 | 评审意见 |
|------|---------|---------|
| 根密钥 `K` 派生 | Argon2id, memory≥64MiB, iterations≥3, parallelism≥1 | **合理**，符合OWASP 2023推荐的生产环境参数下限 |
| 节点答案派生 | Argon2id, memory≥16MiB, iterations≥2, parallelism≥1 | **合理**，作为子密钥派生成本适中 |

**关键确认**：文档明确区分了"根密钥派生"与"节点答案派生"两层Argon2id调用，这是正确的设计——根密钥用于解密保险库，节点子密钥用于分片/授权封装，层级清晰。

### 1.2 对称加密：AES-256-GCM ✅ 通过

文档在多处明确或隐含使用AES-256-GCM：
- `恢复材料最小协议`："使用AES-256-GCM对旧K做封装，形成recoveryEnvelope"
- 云备份：`DEK加密vaultBytes`采用本地AES-GCM路径

**评审意见**：AES-256-GCM是当前主流选择，但需确认：
1. **nonce/IV管理策略**：GCM的nonce必须唯一，文档未明确nonce生成方式（随机96位？计数器？）
2. **密钥派生后的直接使用**：Argon2id输出直接作为AES-GCM密钥，还是经过额外HKDF提取？

### 1.3 组合强度模型：✅ 通过

| 项目 | 文档定义 | 评审意见 |
|------|---------|---------|
| 单题强度 | `difficultyBits = floor(log2(511 / accepted_answer_set_count))` | 工程化指标合理，当前固定8 bits/题 |
| 普通读取总强度 | 6题累计≥30 bits | **合理**，对应约2.2×10^14组合空间 |
| 阈值控制 | `formalEligible`剔除弱题 | 必要机制，防止弱题稀释强度 |

**重要发现**：文档明确声明`difficultyBits`是"工程化抽题指标，不等同于完整密码学熵证明"——这种克制表述是专业的，避免了过度承诺。

### 1.4 摘要算法：⚠️ 需补充

文档提到：
- `checksumVaultBlob`："继续在本地计算，例如SHA-256"
- 身份包：`integrityTag`覆盖完整包体

**缺失**：
1. 未明确challenge answers的摘要算法（对比skill/docs/design/cn中的`挑战答案存储策略.md`，后者建议使用带salt的SHA-256）
2. 未明确`integrityTag`的具体算法（HMAC-SHA256? 纯SHA-256?）

## 二、云备份加密架构评估

### 2.1 包络加密模型：✅ 优秀

```
本地保险库(vaultBytes) 
  → 本地DEK加密 → encryptedVaultBlob
  → DEK交KMS包裹 → wrappedDek
  → 组装CloudBackupEnvelope → 上传OBS
```

**关键正确决策**：
1. **DEK本地生成**：不依赖KMS CreateDatakey，减少一次网络往返
2. **vaultBytes不交KMS直接加密**：KMS只管理DEK，符合"KMS适合管理主密钥与包裹DEK"的原则
3. **双层加密**：StoryLock文件内加密（K保护）+ 云侧BYOK/KMS包裹（DEK保护），即使云存储桶暴露，攻击者仍需突破两层

### 2.2 CloudBackupEnvelope结构：✅ 通过

```typescript
type CloudBackupEnvelope = {
  version: number;
  provider: "huaweicloud_kms";
  kmsKeyId: string;
  kmsRegion: string;
  wrappedDek: Uint8Array;      // KMS包裹后的DEK
  encryptedVaultBlob: Uint8Array; // DEK加密后的保险库
  checksum: string;             // SHA-256完整性校验
  createdAt: string;
  metadata: Record<string, unknown>;
};
```

**评审意见**：
- `checksum`字段明确为SHA-256，合理
- 但建议补充`checksumAlgorithm`字段，便于未来算法升级（如迁移到BLAKE3）
- `version`字段当前冻结为1，符合"不支持的更高版本必须显式拒绝"策略

### 2.3 BYOK定位：✅ 正确

文档明确判断：
> "KMS/BYOK只能作为StoryLock之外的云侧加密包裹层，不能把华为云KMS/BYOK反过来当成StoryLock的主授权根"

这是**关键安全边界**，避免了：
1. 离线能力被破坏（开库强依赖云KMS）
2. 信任面扩大（必须信任云账号、API凭据、云可用性）
3. 安全语义变味（从"用户记忆→本地恢复K"变成"本地授权+云侧托管共同决定"）

## 三、密钥管理评估

### 3.1 根密钥`K`管理：✅ 通过

| 项目 | 设计 | 评审意见 |
|------|------|---------|
| `K`派生 | 完整24答案→Argon2id→32字节 | 正确，不依赖外部熵源 |
| `K`存储 | 不长期明文存储，仅运行时内存存在 | 正确，符合"短时存在内存"原则 |
| `K`导出 | 允许显式导出，带高风险提示 | 合理，用户有自主权 |
| `K`恢复 | 恢复材料通道（Argon2id+AES-256-GCM） | 正确，与主路径独立 |

### 3.2 多身份密钥管理：⚠️ 需补充

`story-lock/doc`文档中**未明确**多身份场景下的密钥管理策略。对比`skill/docs/design/cn/代理签名机制协议参考.md`中建议：
> "保留一个受控根密钥，按identityId派生工作密钥或签名上下文"

**gap**：`story-lock/doc`未说明：
1. 多身份时是否共享同一个`K`派生根？
2. 还是每个身份有独立的24题集和独立的`K`？
3. 若独立，恢复材料是否也独立？

**建议**：在`05-统一秘密对象结构.md`或`03-设计方案的安全性说明.md`中补充多身份密钥隔离策略。

### 3.3 恢复材料加密：✅ 通过

```
恢复口令 → Argon2id → 包装密钥 → AES-256-GCM → recoveryEnvelope
```

**评审意见**：
- 恢复口令与24题集独立，正确
- 恢复材料不纳入题库，正确（文档明确区分"题目标准答案"、"恢复材料"、"保险库根密钥K"）
- 但需确认：恢复口令的Argon2id参数是否与根密钥派生参数一致？若不一致，需在文档中明确

## 四、安全机制评估

### 4.1 防重放与防枚举：✅ 通过

| 机制 | 设计 | 评审意见 |
|------|------|---------|
| 原子性验证 | 6题一次性提交，统一判定 | **必须**，防止逐题排除攻击 |
| 无单题反馈 | 失败只返回"验证失败" | **必须**，防止信息泄露 |
| 不暴露正确项数量 | 正式界面不显示"应选几个" | **合理**，增加攻击者不确定性 |
| 候选项随机排列 | 显示位置≠认证语义 | **正确**，防止位置攻击 |
| 短时会话 | 页面刷新/关闭后失效 | **正确**，减少攻击窗口 |

### 4.2 组合强度控制：✅ 通过

文档明确：
> "普通读取不应只按'随机抽6题'执行，还应满足：抽中的6题总强度达到最小阈值，过弱题目不能进入正式授权访问池"

这是**关键质量控制机制**，防止：
- 6题中混入过多弱题（如5选1的单选题）
- 攻击者通过题库分析识别弱题并针对性攻击

### 4.3 宿主策略边界：✅ 正确

文档反复强调：
> "失败计数、等待时间、人工确认或临时锁定只属于宿主交互策略，不属于离线文件安全边界"

这种**分层安全模型**是专业的：
- 离线文件安全 = 组合强度 + KDF成本（不依赖宿主）
- 在线交互安全 = 宿主节流/锁定（可被绕过，但增加攻击成本）

## 六、关键Gap与行动建议

### 6.1 P0 阻塞风险（影响安全实现）

**1. nonce/IV管理策略未定义**
- 影响：AES-256-GCM的nonce若重复，将导致密钥流重用，完全破坏GCM安全性
- 位置：`03-设计方案的安全性说明.md`提及AES-256-GCM但未说明nonce生成
- 建议：明确采用**96位随机nonce**（来自Web Crypto/random_bytes），并在每次加密时独立生成

**2. 答案摘要算法未定义**
- 影响：challenge answers的校验方式不明确，可能导致彩虹表攻击
- 位置：`story-lock/doc`全文档未明确答案如何存储/校验（明文？hash？加密？）
- 建议：采用`HMAC-SHA256(identitySalt, normalizedAnswer)`，其中`identitySalt = HMAC(masterSalt, identityId)`

**3. 多身份密钥隔离策略未定义**
- 影响：多身份场景下密钥是否隔离不明确
- 建议：明确每个身份有独立的24题集和独立的`K`，恢复材料也独立

### 6.2 P1 质量优化

**4. CloudBackupEnvelope补充checksumAlgorithm字段**
- 当前：`checksum: string`隐含SHA-256
- 建议：增加`checksumAlgorithm: "sha256"`，便于未来算法升级

**5. 恢复口令Argon2id参数明确**
- 当前：恢复材料协议使用Argon2id但未明确参数
- 建议：明确恢复口令派生参数（可与根密钥派生参数一致，或单独定义）

**6. 题集主档加密方式**
- 当前：`story-lock/doc`未明确题集主档是否加密存储
- 建议：明确题集主档采用AES-256-GCM加密，密钥来自`K`或独立派生

### 6.3 文档对齐建议

**7. 创建`加密规范汇总.md`**
- 将分散在`03-安全性说明`、`07-云备份`、`09-KMS-OBS`、`12-BYOK`中的加密相关规范汇总到单一文档
- 包含：算法清单、参数表、密钥生命周期、nonce管理、摘要策略

**8. 更新`skill/docs/design/cn`中标记为"待实现"的文档**
- `本地Agent网关设计.md`：云备份部分可引用`story-lock/doc/07-09`的已有实现
- `EIP-712最小请求定义.md`：签名委托的加密部分需与`story-lock/doc`的密钥派生策略对齐
