# StoryLock 故事生成 Skill 可行性分析

## 核心问题

生成故事是否适合做成 Skill？脱离本地环境后如何保证故事秘密不被泄露？

## 分析框架

### 一、故事生成的本质特征

| 特征 | 分析 | 影响 |
|------|------|------|
| **输入敏感性** | 用户提供的记忆片段、个人经历、偏好信息 | 高敏感度 |
| **输出价值** | 生成的故事文本本身可能包含隐私信息 | 中等敏感度 |
| **处理过程** | 需要访问用户历史数据、记忆库、偏好设置 | 高敏感度 |
| **持久化需求** | 生成后可能需要保存、修改、版本管理 | 中等敏感度 |

### 二、Skill 化的可行性评估

#### ✅ 适合 Skill 化的部分

1. **故事结构模板**
   - 通用叙事框架（起承转合、角色弧光）
   - 不依赖用户隐私数据
   - 可安全地作为公共 Skill 资源

2. **润色与优化**
   - 语法检查、风格调整
   - 基于已脱敏的故事文本
   - 不暴露原始记忆

3. **强度评估**
   - 基于故事文本的质量评估
   - 不涉及隐私数据访问

#### ❌ 不适合 Skill 化的部分

1. **原始记忆访问**
   - 需要读取本地加密存储的记忆片段
   - 涉及高价值凭证（密码、密钥等）
   - 必须在本地环境完成

2. **个性化生成**
   - 基于用户个人经历的故事创作
   - 需要访问本地身份验证数据
   - 无法在远程环境安全执行

### 三、安全边界分析

#### 本地环境 vs 远程环境的信任假设

| 环境 | 信任假设 | 安全风险 | 适用操作 |
|------|---------|---------|---------|
| **本地环境** | 用户完全控制 | 物理访问、恶意软件 | 记忆访问、密钥操作、身份验证 |
| **远程环境** | 依赖服务提供商 | 数据泄露、中间人攻击、合规风险 | 通用模板、脱敏文本处理 |

### 四、混合架构方案

#### 方案一：本地核心 + 远程辅助（推荐）

```
用户请求
    │
    ▼
┌─────────────────┐
│  本地故事生成引擎 │ ← 处理敏感数据，生成初稿
│  (不可 Skill 化)  │
└────────┬────────┘
         │ 脱敏输出（去除个人标识、泛化细节）
         ▼
┌─────────────────┐
│  远程 Skill 服务 │ ← 润色、结构优化、风格调整
│  (可 Skill 化)   │
│  - 语法检查      │
│  - 叙事结构优化  │
│  - 风格迁移      │
└────────┬────────┘
         │ 优化建议（不涉及原始隐私）
         ▼
┌─────────────────┐
│  本地最终合成    │ ← 用户决定是否采纳建议
│  (不可 Skill 化)  │
└─────────────────┘
```

**安全保证：**
- 原始记忆始终不离开本地
- 远程服务只处理脱敏后的文本
- 用户控制最终合成权

#### 方案二：完全本地 Skill

```
Skill 包结构：
storylock-local-skill/
├── SKILL.md              ← 声明本地执行要求
├── assets/
│   ├── templates/        ← 通用故事模板（无隐私）
│   └── schemas/          ← 输入输出验证
└── references/
    ├── local-execution.md  ← 本地执行指南
    └── privacy-check.md  ← 隐私检查清单
```

**执行流程：**
1. AI 读取 SKILL.md，识别需要本地执行
2. 生成本地执行脚本（不传输敏感数据）
3. 用户在本地环境运行脚本
4. 结果返回 AI 进行后续处理

**安全保证：**
- 敏感数据始终本地处理
- AI 只提供通用指导
- 需要用户主动参与执行

### 五、关键风险与缓解

| 风险 | 描述 | 缓解措施 |
|------|------|---------|
| **提示注入** | 恶意用户通过输入提取他人故事 | 输入验证、沙箱隔离、访问控制 |
| **模型记忆** | LLM 训练数据包含用户故事 | 本地模型、差分隐私、数据最小化 |
| **侧信道攻击** | 通过生成时间、长度推断信息 | 恒定时间算法、输出长度统一化 |
| **持久化泄露** | 远程服务日志记录敏感信息 | 零日志策略、本地加密、自动清理 |
| **供应链攻击** | Skill 包被篡改植入后门 | 签名验证、哈希校验、最小权限 |

### 六、决策建议

#### 故事生成的 Skill 化策略

| 组件 | 位置 | 理由 |
|------|------|------|
| 记忆访问 | **本地** | 涉及加密密钥，不可外传 |
| 身份验证 | **本地** | 挑战-应答必须在本地完成 |
| 故事初稿生成 | **本地** | 基于个人记忆，敏感度高 |
| 模板应用 | 本地或远程 | 通用模板可远程，个性化模板需本地 |
| 润色优化 | **远程** | 基于脱敏文本，可安全外包 |
| 强度评估 | 本地或远程 | 基于故事文本，可脱敏后远程 |

#### 核心原则

> **"数据不动，算法动"** — 敏感数据始终留在本地，通用算法可以远程提供。

### 七、实施建议

#### 短期（黑客松）

1. **明确区分本地/远程边界**
   - 在 SKILL.md 中清晰标注哪些操作需要本地执行
   - 提供本地执行脚本模板

2. **最小化远程交互**
   - 远程 Skill 只接收脱敏后的故事文本
   - 不传输任何身份、记忆、密钥信息

3. **用户明确授权**
   - 每次远程交互前明确告知用户哪些数据将被发送
   - 提供"完全本地模式"选项

#### 长期

1. **本地 LLM 支持**
   - 集成本地轻量级模型（如 Llama、Mistral）
   - 完全离线的故事生成与润色

2. **联邦学习架构**
   - 模型改进不依赖原始数据上传
   - 隐私保护与效果提升兼得

3. **零知识证明**
   - 验证故事质量而不暴露内容
   - 远程评估本地生成结果

## 结论

**故事生成可以部分 Skill 化，但必须严格区分本地/远程边界：**

- **本地（不可 Skill 化）：** 记忆访问、身份验证、基于个人数据的初稿生成
- **远程（可 Skill 化）：** 通用模板、润色优化、基于脱敏文本的评估

**安全保证的核心：** 原始记忆和密钥始终不离开本地环境，远程服务只处理已脱敏的文本数据。用户保持对敏感数据的完全控制权。
