# StoryLock 加密技术路线确认：AES-256-GCM + ECC P-256 混合架构

## 一、核心结论

StoryLock 采用**混合加密架构**：
- **AES-256-GCM**：对称加密，用于**数据加密**（故事内容、对象数据、文件）
- **ECC P-256**：非对称加密，用于**身份验证**（设备签名、请求验证）

**一句话**：AES-256-GCM 保护数据，ECC P-256 验证身份。

---

## 二、AES-256-GCM（对称加密）

### 2.1 使用场景

| 场景 | 说明 |
|------|------|
| **故事内容加密** | 保护用户故事原文 |
| **对象数据加密** | 保护密码、私钥等敏感对象 |
| **题集主档加密** | 保护九宫格题集 |
| **SQLite 数据库加密** | 保护本地数据库 |
| **审计日志加密** | 保护云端审计日志（可选） |
| **配置加密** | 保护 FunctionGraph 环境变量（可选） |

### 2.2 密钥派生链

```
masterSalt（随机 256-bit，存本地 SecretStore）
    ↓ HKDF-SHA256
rootKey（故事根密钥，不直接暴露）
    ↓ HKDF-SHA256 + identitySalt
workKey（工作密钥，按对象类型派生）
    ↓ HKDF-SHA256 + objectNonce
objectKey（对象密钥，用于 AES-256-GCM 加密）
```

### 2.3 技术参数

| 参数 | 值 | 说明 |
|------|-----|------|
| 算法 | AES-256-GCM | 认证加密，防止篡改 |
| 密钥长度 | 256-bit | 符合 NIST 推荐标准 |
| Nonce | 96-bit 随机数 | 每次独立生成 |
| 认证 | GCM 内置 | 提供完整性保护 |

## 三、ECC P-256（非对称加密）

### 3.1 使用场景

| 场景 | 说明 |
|------|------|
| **设备签名** | 证明请求来自已注册设备 |
| **EIP-712 请求签名** | 验证 Agent 请求合法性 |
| **HTTPS 通信** | TLS 握手，保护传输通道 |
| **代码签名** | 验证软件包完整性 |

### 3.2 设备签名流程

```
1. 本地设备生成密钥对（ECC P-256）
   ├─ 私钥：存本地 SecretStore
   └─ 公钥：上传到 KMS

2. 发送远程请求时
   ├─ 用私钥对请求签名
   └─ 请求携带签名

3. FunctionGraph 验证
   ├─ 从 KMS 获取设备公钥
   ├─ 用公钥验证签名
   └─ 验证通过 → 转发到本地宿主
```

### 3.3 技术参数

| 参数 | 值 | 说明 |
|------|-----|------|
| 算法 | ECC P-256 | 椭圆曲线加密 |
| 曲线 | NIST P-256 | 标准曲线，广泛支持 |
| 密钥长度 | 256-bit | 等效于 3072-bit RSA 安全性 |
| 签名算法 | ECDSA | 椭圆曲线数字签名 |

## 四、混合架构总结

### 4.1 分层加密策略

| 层级 | AES-256-GCM | ECC P-256 | 说明 |
|------|------------|-----------|------|
| **第一层** | ✅ 故事内容加密 | ❌ 无 | 保护故事原文 |
| **第二层** | ✅ 对象数据加密 | ❌ 无 | 保护密码、私钥等 |
| **第三层** | ✅ 配置加密 | ✅ 设备签名验证 | 保护配置 + 验证设备 |

### 4.2 数据流加密示意

```
用户数据（故事、密码、文件）
    ↓
AES-256-GCM 加密（对称加密）
    ↓
加密数据存储（本地 / 可选云端）
    ↓
远程访问请求
    ↓
ECC P-256 签名（非对称加密）
    ↓
签名请求发送到 FunctionGraph
    ↓
ECC P-256 验证（非对称加密）
    ↓
验证通过 → 解密数据（AES-256-GCM）→ 返回脱敏结果
```

### 4.3 密钥管理对比

| 密钥类型 | 加密类型 | 存储位置 | 生命周期 | 用途 |
|---------|---------|---------|---------|------|
| **masterSalt** | 对称密钥材料 | 本地 SecretStore | 长期有效 | 派生 rootKey |
| **rootKey** | 对称密钥 | 本地派生，不存储 | 长期有效 | 派生 workKey |
| **workKey** | 对称密钥 | 本地派生，不存储 | 长期有效 | 派生 objectKey |
| **objectKey** | 对称密钥 | 本地派生，不存储 | 长期有效 | AES-256-GCM 加密 |
| **设备私钥** | 非对称私钥 | 本地 SecretStore | 长期有效 | 签名请求 |
| **设备公钥** | 非对称公钥 | KMS | 长期有效 | 验证签名 |

## 五、为什么混合使用？

### 5.1 为什么不用纯 AES-256-GCM？

```
纯对称加密的问题：
- 远程访问需要共享密钥
- 共享密钥泄露 = 所有设备可伪造请求
- 无法区分合法设备与攻击者
- 无不可否认性

混合加密的优势：
- 每个设备独立密钥对
- 私钥泄露仅影响单个设备
- 可撤销单个设备（禁用公钥）
- 提供操作不可否认性
```

### 5.2 为什么不用纯 ECC P-256？

```
纯非对称加密的问题：
- 性能差：比对称加密慢 100-1000 倍
- 数据量大：不适合加密大量数据
- 密钥长：3072-bit RSA 等效于 256-bit AES
- 标准限制：无广泛支持的非对称 AEAD 模式

混合加密的优势：
- AES-256-GCM 保护数据（性能）
- ECC P-256 验证身份（安全）
- 各司其职，优势互补
```

## 六、结论

| 问题 | 答案 |
|------|------|
| **StoryLock 用 AES-256-GCM 还是 ECC P-256？** | **两者都用**：AES-256-GCM 保护数据，ECC P-256 验证身份 |
| **数据加密用什么？** | **AES-256-GCM**：对称加密，性能高，适合大量数据 |
| **身份验证用什么？** | **ECC P-256**：非对称加密，密钥分离，适合分布式验证 |
| **为什么混合使用？** | 对称加密性能高，非对称加密安全性好，混合使用优势互补 |
| **密钥需要轮换吗？** | **不需要**：采用"内涵长期有效"策略，用户可以随时更新 |
| **华为云 KMS 管理什么？** | 管理 ECC P-256 公钥（设备验证）和 AES-256-GCM 密钥（配置加密） |

**一句话总结**：

> **StoryLock 采用 AES-256-GCM + ECC P-256 混合加密架构：AES-256-GCM 保护数据（故事、密码、文件），ECC P-256 验证身份（设备签名、请求验证）。对称加密性能高，非对称加密安全性好，两者优势互补。**
