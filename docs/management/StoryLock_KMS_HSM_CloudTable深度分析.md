# StoryLock 中 KMS 与 HSM 技术细节、密钥轮换策略及 CloudTable/DDS 数据存储分析

## 一、核心结论

**用户质疑："作为一个密钥工具，再增加一个密钥管理，这必须要有充分的理由。"**

**回答**：KMS 不是"增加一个密钥管理工具"，而是**将密钥管理从"人治"升级为"技术治理"**。CloudTable/DDS 不是"再增加一个数据库"，而是**解决审计日志长期存储与合规要求**。

| 组件 | 本质 | 解决什么问题 | 不用会怎样 |
|------|------|-----------|-----------|
| **KMS + HSM** | 硬件级密钥生命周期管理 | 私钥永不泄露、自动轮换、合规证明 | 私钥可能泄露、轮换易出错、无法合规 |
| **CloudTable/DDS** | 审计日志长期归档 | 操作可追溯、满足等保、异常分析 | 审计日志丢失、无法追溯、不合规 |

## 二、KMS 与 HSM 技术细节

### 2.1 HSM（硬件安全模块）是什么？

#### 定义

HSM（Hardware Security Module）是**专用硬件设备**，用于：
- 安全生成密钥（真随机数生成器）
- 安全存储密钥（防物理提取）
- 安全使用密钥（在硬件内执行加密运算）
- 防篡改检测（物理拆解自毁）

#### 与普通软件加密的区别

| 维度 | 软件加密（无 HSM） | 硬件加密（HSM） |
|------|------------------|----------------|
| **密钥存储** | 内存/磁盘，可被 dump | 专用硬件芯片，无法提取 |
| **随机数生成** | 伪随机数（PRNG） | 真随机数（TRNG，基于物理噪声） |
| **抗物理攻击** | 无防护 | 防拆解、防侧信道、防电磁探测 |
| **密钥使用** | 密钥加载到内存运算 | 密钥永不离开硬件，硬件内运算 |
| **合规认证** | 无 | FIPS 140-2 Level 3、国密认证 |
| **成本** | 低（软件实现） | 高（专用硬件） |

#### 华为云 KMS 后端 HSM 架构

```
┌─────────────────────────────────────────┐
│  华为云 KMS 服务层                        │
│  ├─ API 网关（请求路由）                  │
│  ├─ IAM 权限控制（谁可以访问）             │
│  ├─ 审计日志（记录所有操作）               │
│  └─ 密钥策略（生命周期管理）               │
└──────────────────┬──────────────────────┘
                   │ 内部网络
                   ▼
┌─────────────────────────────────────────┐
│  HSM 集群（硬件安全模块）                  │
│  ├─ HSM 节点 1（主）                     │
│  │   ├─ 密钥存储区（防篡改芯片）           │
│  │   ├─ 加密运算区（硬件加速）             │
│  │   └─ 随机数生成器（物理噪声源）          │
│  ├─ HSM 节点 2（备）                     │
│  └─ HSM 节点 3（备）                     │
│                                         │
│  特性：                                  │
│  - 密钥在 HSM 内生成，永不导出明文          │
│  - 所有加密运算在 HSM 内完成                │
│  - 物理拆解触发密钥自毁                    │
│  - 多副本冗余，跨可用区部署                 │
└─────────────────────────────────────────┘
```

### 2.2 StoryLock 中 KMS + HSM 的具体应用

#### 场景一：EIP-712 密钥对生成与保护

```
步骤 1：KMS 生成密钥对（在 HSM 内）
├─ 调用 KMS API：CreateKey
├─ HSM 内生成 ECC P-256 密钥对
├─ 私钥：永久留在 HSM 内，**永不导出**
└─ 公钥：导出给 FunctionGraph 使用

步骤 2：FunctionGraph 验证签名
├─ 从 KMS 获取公钥（HSM 内导出）
├─ 使用公钥验证 Agent 请求签名
└─ 私钥从未离开 HSM

步骤 3：密钥轮换（自动）
├─ KMS 在 HSM 内生成新密钥对
├─ 新公钥自动分发到 FunctionGraph
├─ 旧密钥标记为"待禁用"（保留 24 小时）
├─ 24 小时后旧密钥在 HSM 内物理销毁
└─ 全程私钥未离开 HSM
```

**安全优势**：
- 私钥从未以明文形式存在于任何软件、内存、网络中
- 即使 FunctionGraph 服务器被攻破，攻击者也无法获取私钥
- 即使华为云运维人员有服务器权限，也无法从 HSM 提取私钥

#### 场景二：配置加密密钥（DEK）保护

```
步骤 1：KMS 生成 AES-256 对称密钥（在 HSM 内）
├─ 调用 KMS API：GenerateDataKey
├─ HSM 内生成 AES-256 密钥
├─ 密钥明文：仅存在于 HSM 内
└─ 密钥密文：导出给 FunctionGraph 使用

步骤 2：FunctionGraph 加密环境变量
├─ 调用 KMS API：Encrypt
├─ 明文："MyP@ssw0rd123"
├─ HSM 内使用 AES-256 密钥加密
└─ 返回密文："AQICAHg6S9A4lP8..."

步骤 3：FunctionGraph 启动时解密
├─ 调用 KMS API：Decrypt
├─ 密文："AQICAHg6S9A4lP8..."
├─ HSM 内使用 AES-256 密钥解密
└─ 返回明文："MyP@ssw0rd123"（仅存在于 FunctionGraph 内存）

步骤 4：密钥轮换
├─ KMS 在 HSM 内生成新 AES-256 密钥
├─ 用新密钥重新加密所有环境变量
├─ 旧密钥在 HSM 内标记为禁用
└─ 24 小时后物理销毁
```

**安全优势**：
- 加密密钥明文从未离开 HSM
- FunctionGraph 仅持有密钥密文，无法独立解密
- 即使 FunctionGraph 被攻破，攻击者只有密文，无密钥

### 2.3 HSM 防攻击能力详解

#### 物理攻击防护

| 攻击类型 | HSM 防护机制 | 效果 |
|---------|------------|------|
| **拆解芯片** | 防篡改网格，拆解触发密钥自毁 | 攻击者无法物理提取密钥 |
| **侧信道攻击** | 功耗均衡、时序随机化 | 无法通过功耗/时序分析密钥 |
| **电磁探测** | 电磁屏蔽层 | 无法通过电磁信号窃取密钥 |
| **冷冻内存攻击** | 内存加密，断电即清零 | 无法通过冷冻内存提取密钥 |
| **激光切割** | 多层防护，局部破坏触发全局自毁 | 无法局部提取密钥 |

#### 逻辑攻击防护

| 攻击类型 | HSM 防护机制 | 效果 |
|---------|------------|------|
| **密钥导出** | 硬件强制禁止私钥导出 | 即使 root 权限也无法提取 |
| **未授权使用** | IAM 权限 + 密钥策略双重控制 | 无权限者无法使用密钥 |
| **重放攻击** | 每次操作需新鲜随机数 | 无法重放历史操作 |
| **频率攻击** | 操作速率限制 | 暴力破解不可行 |

## 三、密钥轮换策略

### 3.1 为什么需要密钥轮换？

#### 风险场景

```
场景 1：长期密钥泄露
├─ 密钥使用 1 年未轮换
├─ 某次内存 dump 泄露了密钥
├─ 攻击者持有密钥，可长期伪造请求
└─ 影响：1 年的所有请求都可能被伪造

场景 2：前员工风险
├─ 开发人员离职，可能持有密钥副本
├─ 密钥未轮换，离职人员仍可访问
└─ 影响：内部威胁无法消除

场景 3：合规要求
├─ 等保 2.0 要求：密钥定期轮换
├─ PCI DSS 要求：加密密钥 90 天轮换
└─ 影响：不合规无法通过审计
```

#### 轮换的价值

| 价值 | 说明 |
|------|------|
| **限制泄露影响窗口** | 密钥泄露后，影响期 = 轮换周期（而非永久） |
| **消除前员工风险** | 人员变动后，旧密钥失效 |
| **满足合规要求** | 等保、PCI DSS、GDPR 等强制要求 |
| **检测异常使用** | 轮换时发现未授权使用（旧密钥仍在使用） |

---

### 3.2 StoryLock 密钥轮换策略

#### 轮换周期设计

| 密钥类型 | 轮换周期 | 轮换方式 | 原因 |
|---------|---------|---------|------|
| **EIP-712 签名密钥对** | 每 6 个月 | 自动轮换 | 平衡安全与操作成本 |
| **数据库密码 DEK** | 每 3 个月 | 自动轮换 | 数据库密码高敏感 |
| **API 密钥 DEK** | 每 6 个月 | 自动轮换 | 第三方 API 风险 |
| **审计日志 DEK** | 每 12 个月 | 自动轮换 | 日志长期归档 |
| **环境变量主密钥** | 每 12 个月 | 手动触发 | 影响面广，需计划窗口 |

#### 自动轮换流程（以 EIP-712 密钥为例）

```
┌─────────────────────────────────────────┐
│  阶段 1：准备（轮换前 7 天）              │
│  ├─ KMS 生成新密钥对（Version 2）         │
│  ├─ 新公钥分发到 FunctionGraph 集群      │
│  ├─ 新公钥推送到 Agent 配置中心          │
│  └─ 监控：确保所有节点已更新              │
└─────────────────────────────────────────┘
                   ↓
┌─────────────────────────────────────────┐
│  阶段 2：切换（轮换日）                   │
│  ├─ KMS 标记 Version 1 为"待禁用"        │
│  ├─ FunctionGraph 开始使用 Version 2 验证 │
│  ├─ 旧签名仍接受（兼容期 24 小时）        │
│  └─ 监控：旧版本使用量应下降              │
└─────────────────────────────────────────┘
                   ↓
┌─────────────────────────────────────────┐
│  阶段 3：清理（兼容期后）                 │
│  ├─ KMS 标记 Version 1 为"已禁用"         │
│  ├─ 拒绝所有 Version 1 签名               │
│  ├─ 7 天后，Version 1 在 HSM 内物理销毁   │
│  └─ 审计日志记录完整轮换过程               │
└─────────────────────────────────────────┘
```

#### 轮换期间兼容性保证

```
关键设计：多版本共存期

时间线：
T-7 天：新密钥生成，开始分发
T-0 天：切换主密钥，旧密钥进入兼容期
T+24 小时：旧密钥禁用，仅接受新签名
T+7 天：旧密钥物理销毁

兼容性：
- 兼容期内（24 小时），新旧签名都接受
- 确保 Agent 有足够时间更新
- 零停机切换

监控指标：
- 旧版本签名占比（应快速下降）
- 新版本签名成功率（应接近 100%）
- 验证失败率（应无异常上升）
```

---

### 3.3 手动轮换触发场景

| 场景 | 触发方式 | 操作步骤 | 影响 |
|------|---------|---------|------|
| **疑似泄露** | 安全事件响应 | 立即禁用旧密钥，强制所有 Agent 更新 | 服务中断 5-30 分钟 |
| **人员变动** | 管理员手动触发 | 轮换该人员可访问的所有密钥 | 无服务中断 |
| **合规审计** | 定期计划任务 | 按计划自动轮换 | 无服务中断 |
| **密钥到期** | 自动触发 | 提前 7 天生成新密钥，到期自动切换 | 无服务中断 |

## 四、CloudTable/DDS 数据存储分析

### 4.1 为什么需要 CloudTable/DDS？

#### 用户问题："再增加一个数据库，这必须要有充分的理由"

**核心回答**：CloudTable/DDS 不是"再增加一个数据库"，而是**解决审计日志长期存储与合规要求**。

#### 问题场景

```
场景：StoryLock 需要记录所有授权操作

操作日志内容：
{
  "timestamp": "2026-06-28T09:45:00Z",
  "identityId": "user_abc123",
  "operation": "requestSignature",
  "objectId": "story_private_xyz",
  "objectType": "auth_signing",
  "accessLevel": "L5",
  "sessionId": "sess_abc123",
  "result": "success",
  "challengeResult": "passed",
  "gridDifficulty": 22,
  "ipAddress": "192.168.1.100",
  "userAgent": "Mozilla/5.0...",
  "requestDuration": 1250
}

存储需求：
- 每个用户每天 10-100 次操作
- 100 用户 = 1000-10000 条/天
- 1 年 = 36.5 万 - 365 万条
- 每条 1KB = 365MB - 3.65GB/年

问题：
1. 本地 SQLite 能存吗？能，但...
2. 本地存储满了怎么办？
3. 多设备如何同步审计日志？
4. 如何防止本地审计日志被篡改？
5. 等保要求审计日志保存多久？（通常 6 个月 - 2 年）
6. 如何分析跨用户的异常模式？
```

#### 不用 CloudTable/DDS 的问题

| 问题 | 本地 SQLite 方案 | 影响 |
|------|----------------|------|
| **存储容量** | 手机/电脑存储有限 | 日志满后无法记录，或需删除旧日志 |
| **数据持久性** | 设备损坏/丢失 = 日志丢失 | 无法审计 |
| **防篡改** | 本地文件可被修改 | 审计日志不可信 |
| **跨设备同步** | 多设备日志分散 | 无法全局视图 |
| **合规保存期** | 本地存储难以保证 2 年 | 不合规 |
| **异常分析** | 单设备数据，无法跨用户分析 | 无法检测攻击模式 |

---

### 4.2 CloudTable/DDS 存储什么数据？

#### 明确存储的数据

| 数据类型 | 内容 | 是否加密 | 保存期 | 用途 |
|---------|------|---------|--------|------|
| **操作审计日志** | 时间、identityId、操作类型、结果 | ✅ KMS 加密 | 2 年 | 操作追溯 |
| **访问记录** | IP、User-Agent、请求路径、响应码 | ✅ KMS 加密 | 1 年 | 安全分析 |
| **异常事件** | 失败原因、重试次数、锁定记录 | ✅ KMS 加密 | 3 年 | 安全事件调查 |
| **密钥使用记录** | 哪个密钥、何时使用、操作结果 | ✅ KMS 加密 | 5 年 | 密钥审计 |
| **会话元数据** | sessionId、创建时间、过期时间、状态 | ✅ KMS 加密 | 30 天 | 会话管理 |

#### 明确不存储的数据

| 数据类型 | 原因 | 存储位置 |
|---------|------|---------|
| **故事原文** | 用户隐私，绝对不上云 | 本地加密文件 |
| **题集答案** | 验证凭据，绝对不上云 | 本地 SQLite |
| **九宫格题目** | 挑战内容，动态生成 | 本地内存 |
| **masterSalt/rootKey** | 核心密钥，绝对不上云 | 本地 SecretStore |
| **私钥** | 身份凭证，绝对不上云 | 本地 SecretStore |
| **sessionKey 明文** | 会话密钥，短时有效 | 本地内存 |

### 4.3 CloudTable/DDS 与 KMS 的协同

#### 数据流

```
┌─────────────────────────────────────────┐
│  本地宿主（第二层）                        │
│  ├─ 生成审计日志（含敏感字段）              │
│  └─ 用 KMS 密钥加密敏感字段                │
└──────────────────┬──────────────────────┘
                   │ HTTPS（加密传输）
                   ▼
┌─────────────────────────────────────────┐
│  FunctionGraph（第三层）                  │
│  ├─ 接收加密日志                          │
│  ├─ 脱敏处理（移除/替换敏感字段）          │
│  └─ 写入 CloudTable/DDS                  │
└──────────────────┬──────────────────────┘
                   │ 内部网络
                   ▼
┌─────────────────────────────────────────┐
│  CloudTable/DDS（NoSQL 数据库）            │
│  ├─ 分区键：identityId（用户标识）          │
│  ├─ 排序键：timestamp（时间戳）            │
│  ├─ 属性：加密后的日志数据                  │
│  └─ 自动过期：30 天后自动清理会话数据        │
└─────────────────────────────────────────┘
```

#### 加密策略

```
敏感字段加密：
- identityId：哈希处理（SHA-256）
- IP 地址：部分掩码（192.168.x.x）
- User-Agent：截断处理
- 其他字段：KMS 加密

加密密钥：
- 审计日志 DEK：alias/storylock-audit-dek
- 轮换周期：每 12 个月
- 加密位置：FunctionGraph 写入前
```

### 4.4 CloudTable/DDS 的具体价值

#### 价值一：长期存储与自动清理

```
本地 SQLite 方案：
- 存储 100 万条日志 ≈ 1GB
- 手机存储紧张，用户可能清理
- 无法保证 2 年保存期

CloudTable/DDS 方案：
- 自动扩容，无存储上限
- 配置 TTL（Time To Live），自动清理过期数据
- 保证 2 年保存期，到期自动清理
- 成本：¥0.15/GB/月 × 3.65GB = ¥0.55/月
```

#### 价值二：防篡改

```
本地 SQLite 方案：
- 文件在本地，用户可修改
- 攻击者获取设备后，可删除日志
- 审计日志不可信

CloudTable/DDS 方案：
- 数据在云端，用户无法直接修改
- 写入后只读（配置 WORM：Write Once Read Many）
- 修改需通过 FunctionGraph，留下审计痕迹
- 审计日志可信
```

#### 价值三：跨设备同步与全局分析

```
场景：用户有 3 台设备（手机、电脑、平板）

本地 SQLite 方案：
- 每台设备独立日志
- 无法查看全局操作历史
- 无法检测跨设备异常（如：手机授权，电脑异常访问）

CloudTable/DDS 方案：
- 所有设备日志汇聚到统一数据库
- 按 identityId 分区，全局视图
- 可检测异常：
  - 同一 identityId 多地同时登录
  - 异常时间访问（凌晨 3 点）
  - 异常频率（1 分钟内 100 次请求）
```

#### 价值四：合规与审计

```
等保 2.0 要求：
- 审计日志保存期 ≥ 6 个月
- 审计日志不可篡改
- 支持按时间、用户、操作类型查询

CloudTable/DDS 满足：
- TTL 设置 2 年，满足保存期
- WORM 配置，防止篡改
- 分区键 + 排序键，支持高效查询
- 与 KMS 加密，满足数据保护要求
```

## 五、充分理由总结

### 5.1 KMS + HSM 的充分理由

| 理由 | 说明 | 不用会怎样 |
|------|------|-----------|
| **私钥永不泄露** | HSM 硬件保护，私钥永不导出 | 私钥可能泄露，攻击者可伪造签名 |
| **自动轮换** | 零停机，兼容期保证 | 人工轮换易出错，服务中断 |
| **合规证明** | FIPS 140-2、等保认证 | 无法通过安全审计 |
| **防篡改分发** | 公钥从 HSM 可信获取 | 公钥可能被替换，中间人攻击 |
| **灾难恢复** | 多副本冗余，密钥不丢失 | 密钥丢失 = 服务完全中断 |

### 5.2 CloudTable/DDS 的充分理由

| 理由 | 说明 | 不用会怎样 |
|------|------|-----------|
| **长期存储** | 自动扩容，TTL 管理 | 本地存储满，日志丢失 |
| **防篡改** | WORM 配置，云端不可改 | 本地日志可被修改，审计不可信 |
| **跨设备同步** | 全局视图，异常检测 | 多设备日志分散，无法全局分析 |
| **合规要求** | 等保 2.0 保存期要求 | 不合规，无法通过审计 |
| **成本极低** | ¥0.55/月（100 用户） | 自行维护数据库成本更高 |

## 六、结论

| 问题 | 答案 |
|------|------|
| **为什么需要 KMS + HSM？** | 不是"增加一个密钥工具"，而是**将密钥管理从"人治"升级为"技术治理"**——私钥永不泄露、自动轮换、合规证明 |
| **为什么需要 CloudTable/DDS？** | 不是"再增加一个数据库"，而是**解决审计日志长期存储、防篡改、跨设备同步、合规要求** |
| **KMS 与 HSM 的关系？** | KMS 是服务层，HSM 是硬件层。KMS 提供 API，HSM 提供硬件级安全。所有密钥操作在 HSM 内完成 |
| **密钥轮换策略？** | 自动轮换，6 个月周期，24 小时兼容期，零停机。紧急场景支持手动立即轮换 |
| **CloudTable 存储什么？** | 仅存储审计日志、访问记录、异常事件、密钥使用记录。不存储任何用户故事内容或核心密钥 |
| **成本？** | KMS：¥3-6/月。CloudTable：¥0.55/月（100 用户）。合计 ¥4-7/月，可忽略不计 |

**一句话总结**：KMS + HSM 是**密钥安全的基础设施**（私钥永不泄露），CloudTable/DDS 是**审计合规的基础设施**（日志长期可信）。两者都是 StoryLock 安全架构的必要组件，而非"多余的工具"。
