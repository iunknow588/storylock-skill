# StoryLock 开发完善实施计划

日期：2026-06-17

本文基于当前 `src` 代码制作，不再沿用已经失效的旧评审意见。历史评审中仍有价值的部分，只作为风险提醒和验收补充。

## 一、当前代码基线

| 包 | 当前状态 | 已实现能力 |
| --- | --- | --- |
| `src/storylock-local-story-processing-skill` | 已形成第一层能力 | `StoryDraftSkill`、`StoryRefineSkill`、`StrengthReviewSkill` |
| `src/storylock-local-story-access-skill` | 已形成第二层能力 | 对象强度策略、九宫格验证、本地授权、SQLite 状态与审计 |
| `src/storylock-remote-gateway-skill` | 已形成第三层入口 | `requestSignature`、`requestPasswordFill`、可选本地执行器、递归脱敏 |
| `src/storylock-skill-engine` | 兼容演示包 | 本地密码填充、签名授权示例 |

当前主线已经不再包含故事读取、故事写回、`requestChallengeSign` 和能力状态查询接口。

## 二、已完成项

1. 三层能力边界已经完成收敛。
2. 第二层 SQLite 已包含 challenge、session、request、nonce、failure、answer digest、audit log 等状态。
3. replay 冲突已经使用 `SLG-013`。
4. `SLG-008` 已收敛为 `redaction_required`。
5. 第三层接口已经收敛为 `requestSignature` 与 `requestPasswordFill`。
6. 第三层已经对敏感字段做递归脱敏。
7. 四个包均已有 `selftest` 脚本。

## 四、P0：提交前必须完成

### 1. 运行时要求显式化

目标：

1. 在相关 `package.json` 中声明 Node.js 22 或以上。
2. 在 README 和管理文档中说明原因：当前 SQLite 适配使用 `node:sqlite`。

验收：

1. 使用 Node.js 22 运行四个 selftest 通过。
2. 文档不再写 Node.js 18 LTS。

### 2. 端到端演示补齐

目标：

1. 增加一个最小端到端示例。
2. 覆盖远程 `requestSignature` 或 `requestPasswordFill`。
3. 经过第二层强度判断、九宫格验证、本地授权。
4. 调用本地执行器并返回脱敏结果。

验收：

1. 新增 demo 或 selftest 可一键运行。
2. 输出中不包含明文答案、私钥、密码或长期 secret。
3. 审计日志写入 SQLite。

### 3. 文档口径清理

目标：

1. 清除中文设计文档中旧接口的主线描述。
2. 保留历史接口时必须标注为迁移、兼容或非主线。
3. 所有签名请求统一写为 `requestSignature`。

验收：

1. `docs/design/cn` 中不再把 `requestChallengeSign` 写成当前主接口。
2. 不再把第二层写成故事读取或故事写回层。
3. `README.md`、`三包接口契约.md`、`系统Skill表与能力边界.md` 口径一致。

### 4. 错误码与审计文档同步

目标：

1. 文档明确 `SLG-013` 表示 `replay_detected`。
2. 文档明确 `SLG-008` 表示 `redaction_required`。
3. 审计日志字段与 SQLite schema 对齐。

验收：

1. 错误码文档与 `src/storylock-local-story-access-skill/errors.js` 一致。
2. SQLite schema、access host、自测脚本对审计写入的描述一致。

## 五、P1：完善核心可用性

### 1. 九宫格问题来源完善

目标：

1. 将当前占位式九宫格生成升级为基于题集或对象策略的生成。
2. 按目标对象强度选择不同问题数量、正确率和过期时间。

验收：

1. 不同对象强度能生成不同验证配置。
2. 答案只做本地校验和摘要记录，不进入远程返回。

### 2. Challenge 状态机收敛

目标：

1. 文档与代码统一 challenge 生命周期。
2. 明确 pending、verified、failed、expired 等状态含义。
3. 清晰区分 challenge、session、authorization。

验收：

1. 状态转换有自测覆盖。
2. 过期、失败、重复提交都有明确错误。

### 3. 清理任务与维护命令

目标：

1. 将 `cleanupExpired` 包装成可运行维护命令或 host API。
2. 支持批量清理，默认批次不超过 1000。

验收：

1. 可通过脚本触发清理。
2. 清理行为有日志或返回统计。

### 4. SecretStore 生产约束

目标：

1. 明确内存 SecretStore 只用于开发和测试。
2. 为 Windows Credential Manager、macOS Keychain、Linux Secret Service 预留适配位置。

验收：

1. 生产模式下使用内存 SecretStore 会给出明确错误或强警告。
2. 文档说明各平台适配边界。

## 六、P2：应用场景探索

这些方向只作为后续应用探索，不写成当前已实现能力：

1. 面向自动化交易的本地签名授权场景。
2. 面向内容生产和媒体分发的本地凭据填充场景。
3. 面向链上执行的多钱包、多签名主体场景。
4. 面向多云和跨站点代理的统一本地密钥管理场景。

每个方向进入开发前，都需要先补充最小接口、威胁模型和验收自测。

## 七、近期执行顺序

1. 完成中文设计文档与当前代码口径同步。
2. 给相关包补充 Node.js 22 运行时声明。
3. 增加远程请求到本地授权再到执行器的端到端 selftest。
4. 完善九宫格问题来源与强度配置。
5. 增加 SQLite 清理维护脚本。
6. 补充 SecretStore 生产适配说明和保护逻辑。

## 八、完成定义

一个开发项只有同时满足以下条件，才视为完成：

1. 代码实现完成。
2. 自测覆盖完成。
3. 文档同步完成。
4. 不引入旧接口主线口径。
5. 不扩大当前系统能力描述。
