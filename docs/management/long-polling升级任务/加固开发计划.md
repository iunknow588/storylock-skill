# long polling 加固开发计划

## 加固目标

在不引入未配置外部基础设施的前提下，把 long polling 从可用原型加固为可调试、可观测、可降级、资源受控的运行模式，并为后续 Redis/KV/pub-sub 多实例协调留下协议位置。

## 风险分析

1. 当前 relay 队列、等待中的 poll waiter、pending response 都在进程内存中。
2. 单实例、本地调试、固定 Node 进程中可以可靠运行。
3. Vercel 多实例或函数重启时，等待中的 poll 和 pending response 不能跨实例恢复。
4. 如果客户端重复发起多个 long poll，单 host 可能积累多个 waiter，占用内存并造成唤醒歧义。
5. 如果客户端断开连接，需要清理等待中的 waiter，避免无效等待。

## 本轮实施范围

### 阶段 1：策略协商

1. Gateway 注册响应增加 `relay.pollPolicy`。
2. Gateway 状态页增加 relay coordination 信息。
3. Windows Host 读取 `pollPolicy.waitMs` 与 `pollPolicy.clientTimeoutMs`，按 gateway 策略调整 long poll 参数。

### 阶段 2：资源保护

1. Gateway 限制单 host 同时等待中的 long poll 数量，默认 `1`。
2. 新的 long poll 到来时，替换旧 waiter，旧 waiter 返回 `idle`。
3. Gateway 对 `waitMs` 做上限钳制，默认最大 `55000ms`。

### 阶段 3：断连清理

1. `/relay/poll` 监听 response 关闭事件。
2. 客户端断开后，等待中的 waiter 立即移除。
3. 避免断线客户端继续占用 long polling waiter。

### 阶段 4：验证

1. Gateway 自测覆盖：
   - idle long poll 等待后返回。
   - 有任务时 waiting poll 立即被唤醒。
   - 重复 long poll 会替换旧 waiter。
2. Windows Host 编译验证：
   - `cargo build`
   - `cargo build --release`

## 已实施结果

1. Gateway 注册响应已返回 `relay.pollPolicy`：
   - `transport`
   - `waitMs`
   - `maxWaitMs`
   - `clientTimeoutMs`
   - `maxWaitersPerHost`
2. Windows Host 已读取 `pollPolicy` 并按 gateway 策略调整 long polling 请求参数。
3. Gateway 已限制单 host 同时等待 waiter 数量，默认 `1`。
4. 新 long poll 会替换旧 waiter，旧 waiter 立即以 `idle` 返回，并携带 `reason: "replaced_by_new_poll"`。
5. HTTP 连接关闭时会取消对应 waiter。
6. Gateway 状态页已暴露：
   - `coordination`
   - `durability`
   - `productionExternalCoordinatorRequired`
   - `waitingPollCount`
   - `idleTimeoutCount`
   - `replacedPollCount`
   - `clientClosedPollCount`
7. Windows Host 已增加周期性重新注册，用于刷新在线状态与 gateway poll 策略。
8. Gateway 状态页已暴露 `relayPolicy.readiness`，明确单实例/多实例生产边界。

## 观察调试页联调说明

当前远端站点已补充“观察调试”页，可直接用于本轮 long polling 加固的联调和验收观察。

1. `运行状态`
   - 用于查看 `relay.pollPolicy`、coordination、readiness、waiting poll、pending response。
   - 页面已按 `当前策略`、`最近事件`、`累计统计` 三组整理。
   - 顶部会自动生成一句摘要，快速概括当前 relay 健康情况。
2. `Host 联调`
   - 用于查看 Windows Host 本地管理页、状态接口、诊断接口、gateway URL、relay 状态和最近错误。
   - 页面已按 `本地接口`、`当前状态`、`最近错误` 三组整理。
   - 支持直接打开管理页、复制状态接口、复制诊断接口。
3. 颜色提示约定
   - 绿色：当前值健康或符合预期。
   - 黄色：当前值可工作，但提示存在单实例边界、等待堆积或最近异常。
   - 红色：当前值表示明确错误或最近存在错误记录。

建议验收时优先结合观察调试页确认：

1. `Relay 传输` 已切换为 long poll 相关值。
2. `默认等待` 与 `client timeout` 已按 gateway 返回策略展示。
3. `生产就绪` 明确体现“单实例可用，多实例仍需外部协调”。
4. `最近成功返回`、`最近超时` 以及对应累计计数能反映真实运行情况。

## 后续生产增强

如果部署到 Vercel 多实例或无状态函数，下一阶段应接入外部协调层：

1. Redis/Upstash：保存 relay queue、pending response、host heartbeat。
2. Pub/sub：新任务入队后唤醒正在等待的 poll。
3. 幂等响应表：`relayRequestId` 可跨实例解析与去重。
4. Host affinity：同一 host 尽量落到同一区域或同一协调分区。
