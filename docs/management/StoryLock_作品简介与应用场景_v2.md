# 易安与 StoryLock 作品简介及应用场景（修订版）

## 一、作品简介

### 1.1 项目背景

数字化个人代理（Agent）正在将传统的"用户亲自授权"转向"Agent 自主发起请求"。这提高了自动化效率，也把更多敏感能力暴露给远端服务环境——签名私钥、账户密码、API 凭据、开发者证书等核心资产面临被中心化服务托管的风险。

对于强调用户自主管理资产、身份和权限的去中心化方向而言，这种信任集中化趋势是一种技术倒退。市场亟需一种机制，把敏感操作从云端拉回用户可控的本地设备，同时保留 Agent 协作的便利性。

易安与 StoryLock 正是基于这一痛点而生：远端 Agent 可以提出请求，私人智能助理解释风险，StoryLock 本地核心完成授权与敏感执行，用户始终在本地设备上掌握最终确认权。

### 1.2 作品主要特点

| 特点 | 说明 |
|------|------|
| **本地优先** | 签名、密码填充、API 调用、证书签名等敏感操作完全在本地设备执行，远端不接触密钥材料 |
| **故事化验证** | 用户以精心设计的私密故事概况替代传统固定符号口令，通过九宫格动态 challenge 完成授权确认 |
| **三层隔离** | 第一层处理文件加密与操作授权；第二层封装底层操作与风险评估；第三层作为远程网关统一对外入口 |
| **动态挑战** | 24 题题集随机组合生成九宫格 challenge，每次验证题目不同，避免固定口令的泄露风险 |
| **终身免轮换** | 单故事终身制，通过题集渐进式淘汰实现安全更新，无需用户定期更换记忆内容 |
| **可审计脱敏** | EIP-712 结构化请求、requestId/nonce/expiry 防重放、SQLite 审计日志、递归脱敏确保第三方 Agent 无法接触秘密材料 |

## 二、作品应用场景

### 2.1 核心场景：网站账号安全登录

**问题**：用户在多个平台（GitHub、YouTube、邮箱等）使用相同或弱密码，密码管理器将主密码或加密库托管于云端，存在单点泄露风险；远端 Agent 代登录时无法验证用户真实意图。

**解决方案**：
- 用户账号密码由 StoryLock 本地核心加密存储，密钥由私密故事派生
- 登录请求经易安远程入口转达本地，私人智能助理展示目标站点、请求来源、操作类型
- 用户通过九宫格 challenge 确认身份，StoryLock 自动填充密码，远端仅接收登录结果而非密码明文
- 即使云端服务被攻破，攻击者无法获取密码库或解密密钥

**受众**：多平台开发者、内容创作者、企业运维人员

**典型资源**：GitHub 主账号、YouTube Studio 创作者账号、通用邮箱密码账号

---

### 2.2 核心场景：钱包账号安全签名

**问题**：DeFi 用户将私钥导入热钱包，或授权给远端交易机器人，面临私钥泄露、恶意签名、钓鱼攻击等风险；多链、多账号并存时密钥管理混乱。

**解决方案**：
- 用户私钥由 StoryLock 本地核心管理，存储于设备安全区域（Android Keystore / Windows DPAPI）
- 交易签名请求经易安远程入口送达本地，私人智能助理展示交易详情（金额、目标合约、Gas 费）
- 用户通过九宫格 challenge 确认身份，StoryLock 本地完成签名，仅返回签名结果上链
- 远端 Agent 从未接触私钥，钓鱼攻击无法窃取签名能力
- 多链（Ethereum、Solana 等）、多账号统一资源目录管理，避免密钥散落

**受众**：DeFi 交易者、NFT 持有者、DAO 参与者、多链开发者

**典型资源**：Ethereum 主钱包、Solana 创作者钱包

---

### 2.3 核心场景：API 凭据安全调用

**问题**：云平台访问密钥（AWS、GitHub PAT 等）硬编码在脚本或 CI/CD 配置中，泄露后影响面广；Agent 自动化任务无法安全获取临时凭据。

**解决方案**：
- API 凭据由 StoryLock 本地核心加密存储，按环境（prod/dev）隔离管理
- Agent 任务请求经易安远程入口转达本地，私人智能助理展示调用目标、操作范围、风险等级
- 用户通过九宫格 challenge 确认，StoryLock 本地注入凭据完成 API 调用，返回结果脱敏
- 支持成组访问密钥（access_key/secret_key/session_token）和单令牌（PAT）两种形态
- 凭据轮换时只需更新本地对象，不影响外部模板引用

**受众**：云原生开发者、DevOps 工程师、自动化运维人员

**典型资源**：AWS 生产环境访问密钥、GitHub Personal Access Token

---

### 2.4 核心场景：开发者证书安全签名

**问题**：Apple 开发证书、代码签名私钥分散在开发者本地机器，团队协作时通过邮件/网盘传递，存在泄露和版本混乱风险；CI/CD 流水线需要无人值守签名时无法安全授权。

**解决方案**：
- 证书与私钥由 StoryLock 本地核心统一管理，支持带口令保护的私钥
- 构建签名请求经易安远程入口转达本地，私人智能助理展示签名目标（App 包名、版本、证书身份）
- 用户通过九宫格 challenge 确认，StoryLock 本地完成签名，导出已签名产物
- 支持平台特有附加对象（如 Apple team_id），与通用签名材料共用同一骨架
- 多证书并存时通过资源目录清晰区分，避免误用

**受众**：移动应用开发者、桌面软件开发者、企业发布工程师

**典型资源**：Apple 开发证书身份、通用代码签名证书身份

---

### 2.5 扩展场景：多资源协同工作流

**问题**：真实业务往往需要同一流程中串联多种敏感操作，例如"登录 GitHub → 构建签名 → 上传 AWS → 发布版本"，传统方案在每个环节都需要人工干预或暴露凭据。

**解决方案**：
- StoryLock 支持在同一工作流中编排多种资源类型（website_account、certificate_bundle、api_client）
- 每个步骤独立经过九宫格 challenge 确认，用户可按需调整步骤顺序和资源组合
- 工作流模板只描述动作和资源引用，不保存秘密值，真正执行时由宿主解析为本地对象
- 支持权限边界按"单对象 / 多对象 / 高敏感材料协同"分级显示

**受众**：全栈开发者、独立创作者、小型技术团队

**典型流程**：登录 GitHub 查看仓库 → 使用 Apple 证书签名构建产物 → 使用 AWS 凭据上传 → 使用 GitHub Token 发布版本说明

---

## 三、总结

易安与 StoryLock 的核心价值在于**重建用户对敏感数字资产的控制权**。在 Agent 自动化趋势下，传统方案要求用户将信任让渡给远端服务，而 StoryLock 通过本地优先架构、故事化验证、动态挑战机制，让用户在享受自动化便利的同时，始终掌握密钥与授权的终极决策权。

**一句话定位**：StoryLock 是 Agent 时代的个人数字保险箱——远端可以请求，但打开保险箱的钥匙，永远只在用户手中。
